# Archive Center 2.3 GitHub Install/Update Contract

Archive Center has two different GitHub paths. They must not be mixed.

## 1. Normal User Path: GitHub Release Package

Normal users install and update from GitHub Releases, not from a raw source
checkout.

The release package contains:

- compiled backend binaries
- package launchers
- migrations
- prompts
- OS-specific runtime bootstrap scripts

This path is the default for Windows, Linux/Ubuntu, macOS, and Termux.

Install/update helpers:

- POSIX: `scripts/install-github-release.sh`
- Windows: `scripts/install-github-release.ps1`

These helpers must:

1. read the latest GitHub Release
2. select the current OS package asset
3. download `SHA256SUMS*.txt`
4. download the selected package ZIP
5. verify SHA256
6. extract into a versioned install directory
7. keep runtime data in a stable install-level data directory
8. move the local `current` pointer to the verified package
9. optionally start or register the package launcher

The package launcher is responsible for MariaDB, ChromaDB, migrations, and
`live/full_local` runtime env. A release-package install must not leave the
backend in `shadow/noop/core_lite/fallback`.

Release updates must not use the versioned package directory as the canonical
database location. The installer-owned data root is:

- POSIX/systemd: `$INSTALL_DIR/data` through `ARCHIVE_CENTER_DATA_DIR`
- Windows helper: `$InstallDir\data` through `ARCHIVE_CENTER_DATA_DIR`

If a previous release stored data under a package-local `.runtime`, the helper
may migrate that runtime into the stable data root only when the stable root has
not already been initialized. It must not overwrite an existing stable data
root during an update.

## 2. Operator Path: Git Source Checkout

Raw `git clone` is an operator/developer path. It does not by itself install
MariaDB, ChromaDB, package launchers, or system services.

A source checkout may be used to build packages, run tests, or prepare a
release. It must not be described as a normal full install unless a deploy
script has built and staged a package-shaped install root.

Required future operator helpers:

- source build/deploy script
- source update/deploy script
- `/ready` release gate after service restart

Until those helpers exist, the supported self-host install path is the GitHub
Release package helper.

## Required Ready Gate

After an install or update, the backend must pass the release gate:

```text
mode=live
store_mode=mariadb_authority
runtime_profile=full_local
vector_mode=local_native|bundled|local_proot
mariadb=configured
chromadb=configured
degraded=false
```

If `/ready` reports `shadow`, `noop`, `core_lite`, or `fallback` for a full
install, the install is not complete.
