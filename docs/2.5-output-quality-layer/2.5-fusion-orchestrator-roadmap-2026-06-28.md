# 2.5 Fusion Orchestrator Roadmap

Date: 2026-06-28

Status: active design anchor for S9+

## Purpose

This document defines the next step after the current S0-S8 foundation:
moving from conservative audit-only reader checks into an enhancement-first
multi-model fusion system.

The target is not to copy existing agent plugins or external fusion products.
The target is to build a RisuAI-specific output quality layer where the main
RisuAI LLM output remains the base draft, specialist models produce bounded
revision material, a fusion director/composer creates a better output, and a
verifier gates whether that enhanced output can be returned.

Proposal objects are an internal safety contract, not the product goal. The
product goal is a verified enhanced final output.

Low-cost small models are the intended common case, but they must not be
required. Any role may use low-cost, local, commercial, or frontier models
according to the user's configuration.

## Reference Concepts

The roadmap is inspired by two current orchestration patterns:

- Sakana Fugu: learned orchestration over a pool of LLM workers, including
  dynamic selection, agentic scaffolds, verification, and synthesis.
- OpenRouter Fusion Router: parallel panel responses followed by a judge that
  returns structured analysis such as consensus, contradictions, coverage gaps,
  unique insights, and blind spots. The outer model then uses that analysis to
  write a better answer.

This plugin should adapt those ideas to the RisuAI post-generation setting:

- The base output is already produced by RisuAI's main model.
- The plugin must preserve protected markers, images, code fences, JSON-like
  blocks, RisuAI state markers, and inspect-only context.
- The system should produce an enhanced output through bounded revision,
  fusion, and verification. It must not stop at advisory-only trace output, and
  it must not use unbounded direct full-output rewrites.

## Product Goal

The user-facing goal is:

1. RisuAI produces the normal reply.
2. The plugin captures that reply as the base draft.
3. Multiple specialist AIs inspect the base draft from different angles.
4. Each specialist produces bounded revision material for its own domain.
5. A fusion director judges overlap, conflict, priority, and risk.
6. A bounded composer creates an enhanced output candidate.
7. JavaScript verifiers reject unsafe candidates.
8. In `fusion_enhance`, verified candidates become the returned final output.
9. The user can inspect what happened in Trace.
10. If the verifier fails or the system is uncertain, the original output is
    returned.

## Non-Negotiable Safety Contract

These rules stay above every later fusion feature:

- The original base draft must always remain available for rollback.
- Protected segments must be exactly preserved.
- Inspect-only runtime context must never be mutated or written back into RisuAI.
- AI roles may propose changes only for mutable segment IDs.
- No role may patch images, RisuAI markers, code fences, JSON-like blocks, or
  state/window blocks unless that segment is explicitly mutable.
- Whole-response rewrites are rejected by default.
- Patch size must stay under a hard cap.
- Streaming output must fall back to original until the full verifier can run.
- Auxiliary/helper/submodel calls must not bypass the verifier.
- Failure, ambiguity, invalid JSON, missing content, timeout, or verifier doubt
  must keep the original output.
- Trace must show executed roles, skipped roles, provider/model, fallback path,
  token estimate, findings/proposals, protected counts, and rejection reasons.

## Architecture

### Layer 1: Base Draft Intake

Input:

- RisuAI main LLM output.
- Current request payload/system/recent-chat snapshot.
- Runtime context snapshot: character, persona, current chat, active lorebook,
  memory snapshot, and source availability.

Output:

- `base_draft`.
- `protected_marker_catalog`.
- `segment_map` with protected, inspect-only, and mutable segments.
- `context_manifest` with source IDs, source names, char counts, token
  estimates, and read-only status.

### Layer 2: Specialist Proposal Roles

Current readers should evolve from "findings only" into "proposal-capable"
roles. In proposal mode they still do not rewrite the whole output.

Initial S9 roles:

- Character Proposal Reader: voice, agency, knowledge boundary, relationship
  consistency.
- Plot/Continuity Proposal Reader: scene continuity, causality, missing setup,
  timeline contradictions.
- Style Proposal Reader: rhythm, repetition, phrasing, tone, paragraph flow.
- JS Director Merge: deterministic merge and rejection logic.

Later roles:

- Lore/World Proposal Reader.
- Secret/POV Guard.
- Scene Anchor Guard.
- Emotion/Dialogue Specialist.
- Table Read Single Panel.
- Cost/Latency Router.

### Layer 3: Proposal Schema

Specialists should return strict JSON:

```json
{
  "schema": "fusion_proposals.v1",
  "role": "character_reader",
  "model": "provider/model",
  "proposals": [
    {
      "proposal_id": "character_reader_1",
      "segment_id": "mutable_1",
      "type": "replace_sentence",
      "priority": "high",
      "risk": "low",
      "confidence": 0.86,
      "find": "exact source text or short anchor",
      "replace": "proposed replacement text",
      "reason": "why this improves the base draft",
      "evidence": [
        {
          "source": "character",
          "source_id": "character",
          "summary": "short evidence note"
        }
      ],
      "protected_risk": false
    }
  ],
  "global_notes": []
}
```

The schema should be intentionally narrow at first. It is better to reject a
useful proposal than to accept an unsafe one.

### Layer 4: Proposal Normalizer

The JavaScript normalizer runs before any fusion director:

- Validate schema version.
- Validate `segment_id`.
- Confirm `find` appears inside the target mutable segment.
- Reject protected or inspect-only target segments.
- Reject proposals touching protected markers or image tags.
- Reject full-paragraph or full-output rewrites unless the mode allows them.
- Cap replacement size and total changed characters.
- Dedupe repeated proposals.
- Detect overlapping proposals and rank them by priority, confidence, and risk.
- Attach normalized rejection reasons for Trace.

Output:

- `accepted_proposals_for_director`.
- `rejected_proposals`.
- `proposal_conflicts`.

### Layer 5: Fusion Director

The director decides what should be changed, not how to freely rewrite the whole
answer.

First implementation should be deterministic JS:

- Accept low-risk non-overlapping proposals.
- Prefer high-confidence proposals with evidence from character/lore/recent chat.
- Reject proposals with protected risk.
- Reject style-only suggestions that conflict with character or continuity.
- Prefer smaller edits over broad rewrites.
- Preserve the ending unless there is a high-confidence missing-ending issue.

Later implementation may add an optional LLM director:

- Receives only base draft excerpts, normalized proposals, conflicts, and
  context manifest.
- Returns `fusion_plan.v1`.
- Cannot directly produce the final output.

### Layer 6: Fusion Plan Schema

```json
{
  "schema": "fusion_plan.v1",
  "mode": "preview",
  "accepted": [
    {
      "proposal_id": "character_reader_1",
      "segment_id": "mutable_1",
      "operation": "replace",
      "find": "exact source text",
      "replace": "replacement text",
      "reason": "accepted because..."
    }
  ],
  "rejected": [
    {
      "proposal_id": "style_reader_2",
      "reason": "overlaps higher-priority continuity proposal"
    }
  ],
  "deferred": [],
  "risk_summary": {
    "protected_risk": false,
    "rewrite_risk": "low",
    "confidence": 0.82
  }
}
```

### Layer 7: Composer

Composer order should be conservative:

1. JS exact replacement composer for local edits.
2. JS segment-level composer with strict operation list.
3. Optional LLM segment composer only after the JS path is proven.

The composer must produce:

- `candidate_output`.
- `applied_operations`.
- `composer_rejections`.
- `candidate_diff_summary`.

### Layer 8: Verifier

The existing JS verifier remains the hard gate:

- Exact protected preservation.
- Length drop detection.
- Last paragraph preservation or intentional-change explanation.
- Duplicate block detection.
- Markdown/code fence balance.
- Image tag preservation.
- JSON-like block balance.
- RisuAI marker preservation.
- Patch hard-cap validation.
- Idempotency marker validation.

LLM verifier may be added later, but it cannot replace the JS verifier.

### Layer 9: Trace UI

Trace must expose the fusion process in plain terms:

- Base draft length and token estimate.
- Runtime context source availability.
- Active roles, skipped roles, and reasons.
- Provider/model per role.
- Raw failure preview for invalid JSON or reasoning-only responses.
- Proposal count per role.
- Accepted, rejected, deferred proposals.
- Conflict groups.
- Candidate preview/diff.
- Verifier result.
- Fallback reason.
- Final output source: original fallback or verified enhanced output.

## Mode Ladder

The plugin should not jump directly from audit-only to automatic rewriting.

### Current Foundation: S0-S8

- Safety contract.
- Pass-through audit.
- Check-only readers.
- Trace UI.
- JS verifier.
- Light patch guarded off by default.
- Runtime context collector.
- Bounded parallel execution.
- Live QA hardening.

### S9: Enhancement Contract And Revision Schema

Goal:

- Add `fusion_enhance` as the forward target mode.
- Add `fusion_proposals.v1` or `fusion_revisions.v1` as the internal bounded
  revision schema.
- Make the schema immediately useful for producing an improved candidate, not
  merely for advisory findings.
- Trace displays revision counts, examples, and normalizer rejections.

Implementation:

- Add mode UI.
- Add proposal schema parser.
- Add proposal prompts for the initial three reader roles.
- Add proposal normalizer.
- Add a minimal exact-replacement composer path for accepted safe revisions.
- Keep audit/pass-through mode available for debugging.

Exit criteria:

- `audit_only` original output exact-match is preserved.
- `fusion_enhance` can produce a candidate improved output from safe bounded
  revisions.
- Invalid JSON and reasoning-only failures are visible.
- Accepted/rejected revision lists are visible.

### S10: Verifier-Gated Fusion Enhance

Goal:

- Return an enhanced final output when the candidate passes the JavaScript
  verifier.

Implementation:

- Run the existing JS verifier against the composed candidate.
- Return the enhanced candidate in `fusion_enhance` when all gates pass.
- Return the original output on verifier failure, uncertainty, timeout, invalid
  JSON, missing content, or patch-cap breach.
- Show final output source in Trace: original fallback or verified enhanced.
- Keep a user-visible toggle/preset that can force preview/debug behavior, but
  do not make advisory-only behavior the product goal.

Exit criteria:

- Verified enhancement actually changes the returned output when safe revisions
  exist.
- Any unsafe candidate falls back to the original.
- Trace explains exactly why the enhanced output was used or rejected.

### S11: Deterministic Fusion Director

Goal:

- Convert normalized revisions into `fusion_plan.v1` with deterministic conflict
  handling.

Implementation:

- Add JS director scoring.
- Add overlap/conflict handling.
- Add global patch cap.
- Add per-role priority weights.
- Add Trace plan panel.
- Preserve smaller, evidence-backed revisions over broad style rewrites.

Exit criteria:

- Revision conflicts are deterministic.
- Same input yields the same plan.
- Fusion output is improved without protected-marker damage.

### S12: Strong Fusion Composer

Goal:

- Move beyond simple exact replacements into stronger bounded composition while
  still returning only verifier-approved enhanced output.

Implementation:

- Add segment-level composition for mutable segments.
- Add optional LLM segment composer only after JS composition is stable.
- Require operation lists and segment IDs for every composed change.
- Reject whole-output rewrite attempts unless a future explicit high-risk mode
  is created.
- Add diff summary and changed-character ratio to Trace.

Exit criteria:

- The system can perform meaningful multi-sentence improvements.
- Protected and inspect-only regions remain untouched.
- Fallback to original remains automatic on any verifier failure.

### S13: Adaptive Orchestrator

Goal:

- Stop calling every role every turn. Route roles based on the base draft and
  runtime context.

Signals:

- Output length.
- Protected segment density.
- Active lorebook count.
- Recent chat conflict indicators.
- Dialogue-heavy vs exposition-heavy output.
- Previous role failure rate.
- Provider/model latency.
- User cost preset.

Implementation:

- Add `routing_profile`: conservative, balanced, quality, cost-save.
- Add role selection trace.
- Add failure-rate-aware fallback suggestions.
- Add max parallel role cap per profile.

Exit criteria:

- Simple turns call fewer roles.
- Important scenes can call more roles.
- Trace explains routing decisions.

### S14: Dynamic Agent Scaffold

Goal:

- Introduce a Fugu-like scaffold planner for complex turns.

Implementation:

- Add an optional orchestrator role that proposes temporary specialist roles
  from a strict whitelist.
- Enforce hard caps: max temporary roles, max prompts, max tokens, max time.
- No temporary role may bypass proposal schema or verifier.
- No self-recursive orchestrator calls.

Exit criteria:

- Dynamic roles are bounded and auditable.
- Static role mode remains available.
- Cost and latency are predictable.

### S15: Table Read Fusion

Goal:

- Add opt-in higher-cost panel review for important scenes.

Implementation:

- Start with single-panel Table Read.
- Add full Table Read only as explicit important-scene opt-in.
- Table Read emits proposals and consensus notes, not direct rewrites.
- Fusion director decides which notes survive.

Exit criteria:

- Table Read is never silently expensive.
- Panel output is visible in Trace.
- Full table read remains opt-in.

### S16: Release Hardening

Goal:

- Turn the system into a stable user-facing plugin.

Implementation:

- Preset import/export.
- Provider/model diagnostics.
- Cost/token budget display.
- Korean/English UI label cleanup.
- Regression samples for protected markers, image tags, code fences, JSON, and
  RisuAI markers.
- Release zip packaging.

Exit criteria:

- New user can configure roles without reading source code.
- Existing audit-only behavior remains reliable.
- Release artifact includes docs and validation notes.

## Immediate Implementation Order

1. Correct the roadmap around enhancement-first fusion.
2. Add `fusion_enhance` mode as the main future path.
3. Add the bounded revision schema and parser.
4. Add the proposal/revision normalizer.
5. Convert the initial three readers to revision-producing prompts in
   `fusion_enhance`.
6. Add a minimal exact-replacement composer.
7. Run the existing JS verifier on the candidate.
8. Return the enhanced output when the verifier passes.
9. Fall back to original output on any uncertainty.
10. Add deterministic director scoring and stronger segment composition.

## Design Warnings

- Calling more AIs is not the same as fusion. The important part is producing a
  better final output through structured revision, conflict resolution, bounded
  composition, and verifier-gated return.
- A direct "rewrite this output better" prompt is too risky for RisuAI output
  because it can destroy images, markers, state windows, code fences, or hidden
  continuity.
- Adaptive routing should reduce waste, but it must not weaken the main goal:
  improving the returned output.
- Dynamic scaffolding should be introduced only after static proposal fusion is
  reliable.
- Frontier models should be allowed, but the plugin must remain useful with
  cheaper and smaller models.
