# 2.5 Live QA Hardening Notes

Date: 2026-06-28

Status: implemented

## Trigger

Live RisuAI trace showed that the S7/S8 pipeline worked, but several practical
issues needed hardening before broader QA:

- Runtime context trace showed very large character counts.
- Trace showed chars but not token-level cost intuition.
- Lorebook candidates were collected, but active lore matching was too weak.
- Supa/Hypa/current-chat memory fields were not found in the tested setup.
- Image output markers were not visibly detected as image-protection spans.
- One reader failed with `invalid_json_response`.

## Implemented

- Added bounded context limits for payload system, recent chat, user input,
  lorebook candidates, active lore summaries, memory snapshots, and runtime
  context blocks.
- Added heuristic `estimated_tokens` per runtime context source and
  `total_estimated_tokens` in the context summary.
- Added Trace UI display for estimated tokens next to character counts.
- Reduced lorebook candidate volume and bounded each candidate content preview.
- Expanded lorebook key aliases and normalized matching for spacing,
  punctuation, and width variants.
- Expanded memory field aliases across current chat, database, and character
  objects.
- Added Risu image marker protection for `{{img::...}}`, `{{image::...}}`,
  `{{bg::...}}`, `{{asset::...}}`, `{{emotion::...}}`, `<img="...">`, and
  `<img cmd=...>`.
- Added robust reader JSON recovery for fenced JSON, prefixed/suffixed prose,
  trailing commas, and top-level arrays.
- Added strict JSON-only output contract to reader prompts.
- Added one retry on `invalid_json_response` with lower temperature and
  provider JSON-mode hints where supported.
- Added Trace UI raw response preview for failed/retried reader calls.
- Added limited memory debug field listing so missing Supa/Hypa/current-chat
  memory locations can be diagnosed without dumping values.
- Added active lore match scores, score range trace display, top-8 active lore
  cap, and a minimum active score threshold to reduce overmatching.
- Added GLM/Ollama-compatible provider hints for structured reader calls:
  `think: false`, JSON response format, and Ollama-style `format: "json"` when
  applicable.
- Added compact invalid-JSON retry behavior: retry output is capped to at most
  two findings, retry temperature is lowered, and retry `max_output_tokens` is
  raised to a bounded minimum.
- Added `provider_reasoning_without_content` so responses with empty `content`
  and provider `reasoning` are distinguishable from normal JSON parse failures.
- Added provider wrapper summaries in Trace UI, including content chars,
  reasoning chars, finish reason, and usage tokens when available.
- Added recent model/role failure-rate trace stats so unstable GLM reader roles
  can be reassigned to another model/provider.
- Added role-level fallback model settings. A fallback model can be enabled per
  role and is used only when the primary role call fails after retry or returns
  empty findings after retry.
- Added `empty_findings_after_retry` diagnostics so format-valid but
  non-actionable reader recovery is visible in Trace.
- Added fallback Trace fields: trigger, skipped reason, fallback provider/model,
  fallback summary, final model, and whether fallback findings were used.
- Updated model stats with `problem_rate`, `empty_after_retry`, and fallback
  success counts.
- Updated plugin version to `0.1.2`.
- Updated runtime context collector version to
  `s7.10-live-qa-hardening`.

## Notes

Token counts are estimates, not model-tokenizer exact values. They are intended
for trace cost intuition and debugging, because each provider/model can tokenize
Korean, English, symbols, and RisuAI tags differently.

The plugin still returns the original output in `audit_only` and keeps output
mutation guarded behind the existing light-patch verifier path.
