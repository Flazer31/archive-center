# 2.5 MDASH / Fusion Operating Contract

Date: 2026-06-29

Status: active interpretation lock for the 2.5 output quality layer

## Purpose

This document fixes the current interpretation of the 2.5 work so later
implementation does not drift back into timid review, trace-only advice, or
weak "do no harm" positioning.

The product is not "more agents" and not "trace-only advice." The product is a
post-generation output quality orchestrator for RisuAI:

```text
RisuAI main LLM output
-> protected / inspect-only / mutable segmentation
-> read-only runtime context snapshot
-> specialist reader and guard roles
-> findings-level fusion
-> bounded revision fusion
-> segment composer
-> JS verifier
-> strong enhanced output; technical rollback only on structural failure
```

The goal is an aggressively improved final assistant output. Specialist AIs
must generate concrete stronger revision candidates; Fusion must select and
compose the best candidate set. Verification is a final QA gate for structural
integrity, not the product's main value proposition.

## What Must Be Improved

The plugin exists to defend and improve RP output quality in concrete ways:

- character voice and emotional posture
- character knowledge boundaries
- user agency preservation
- secret, private-knowledge, narrator, and POV boundaries
- scene continuity, causality, location, time, and action order
- final paragraph / ending preservation
- prose rhythm, repetition, transitions, tone, and awkward phrasing
- mechanical artifacts such as stray symbols, meta text, list-like summaries,
  translator-like wording, or model self-commentary
- protected output structures such as image tags, status windows, code fences,
  JSON-like blocks, RisuAI markers, module tags, and user regex protected spans

Protected output preservation is required, but it is a lower-level QA
constraint. The product value is visible prose, character, continuity, POV,
secret-handling, and style improvement in the returned output.

## What Is Not Allowed

These are rejected product interpretations:

- A generic "rewrite this better" whole-output prompt.
- A suggestion-only UI where findings never affect the returned output.
- A timid reviewer that refuses to generate strong grounded candidates.
- An autonomous agent tree where an AI decides which other AIs to call.
- Silent high-cost full Table Read on normal turns.
- Treating single-panel Table Read as independent multi-agent verification.
- Letting style-only preferences override character, continuity, POV, or user
  agency.
- Returning an enhanced output without JS verifier approval.
- Patching protected, inspect-only, table, code, JSON, image, marker, or
  user-regex-protected material.
- Writing RisuAI state, lorebook, memory, database, or chat content from the
  output quality layer.

## MDASH Interpretation

MDASH is used as a harness pattern, not as uncontrolled multi-model rewriting.

For this standalone 2.5 plugin, the active MDASH chain is:

```text
Input Enhance MDASH
-> RisuAI native prompt/context assembly
-> Main LLM
-> Output Check MDASH
-> Table Read MDASH when explicitly useful
-> Output Enhance MDASH
-> Verifier / protected output guard
-> RisuAI display/save
```

Meaning:

- Input Enhance is support-only guidance. It never rewrites user input.
- Output Check produces role-separated findings and bounded revision material.
- Table Read produces panel notes and proposals, not final scene prose.
- Output Enhance converts accepted findings into bounded segment operations.
- Verifier has veto authority. It is not a creative writer.

## Fusion Interpretation

Fusion means structured synthesis, not merely calling multiple AIs.

Every reader/guard output should be normalized into findings or bounded
revision proposals. The fusion layer then computes:

- `consensus`: multiple roles point at the same issue or segment.
- `conflict`: roles propose incompatible changes.
- `gap`: an expected role, lane, or risk area produced no useful signal.
- `unique_insight`: one role found a specific issue that others missed.

The Fusion Director must use these signals to score, accept, reject, or defer
changes. The Segment Composer then applies only accepted, bounded operations.

Fusion is successful only when:

```text
specialist findings
-> conflict/consensus scoring
-> bounded segment composition
-> JS verifier pass
-> enhanced output returned
```

Trace-only findings are useful for debugging, but they are not the product goal
for `fusion_enhance`.

## Fugu Interpretation

Sakana Fugu is a reference for adaptive orchestration, not an implementation
permission slip.

The structurally destructive parts are not allowed:

- no AI-controlled recursive role calls
- no AI-selected provider/model expansion
- no unbounded dynamic work tree
- no hidden escalation into expensive modes
- no full-output rewrite as a synthesis shortcut

The allowed adaptation is a deterministic JavaScript role router:

```text
base draft + segment map + runtime context manifest
-> JS risk signals
-> fixed role selection under user max mode, toggles, call budget, and
   deterministic structure rules
-> parallel role calls
-> fusion director/composer/verifier
```

The router may recommend or select roles only inside hard limits:

- user-selected maximum mode
- enabled role profiles
- max role count per turn
- max parallel calls
- estimated token/cost budget
- streaming downgrade
- auxiliary/helper/submodel/LUA bypass or audit-only rule
- full Table Read explicit opt-in

AI roles may report risk, but they do not get authority to call more AIs.

## Role Selection Contract

`fusion_enhance` should move toward this default reader set:

- `character_reader`
- `plot_continuity_reader`
- `style_reader`
- `secret_pov_guard`

Conditional roles may be added by the deterministic router:

- `scene_anchor_guard`: speaker, location, body position, object, or action
  order risk.
- `world_reader`: visible lore/world-rule contradiction.
- `single_panel_table_read`: important or multi-character scene where a compact
  simulated panel is useful and the user has enabled the cost.
- future mechanical/style artifact guard: stray symbols, meta text, list-like
  summaries, model self-commentary, translator-like phrasing.

The minimum fusion path must not depend on Table Read. Table Read is a higher
quality panel source, not the foundation.

## Table Read Contract

Table Read is compatible with MDASH/Fusion only in this form:

```text
base draft
-> Table Read panel reads protected/mutable map and inspect-only context
-> panel emits notes and recommended findings/proposals
-> Fusion Director decides what survives
-> Composer applies bounded changes
-> Verifier gates return
```

Single-panel Table Read:

- one model simulates a compact panel
- must be marked as `single_panel_simulated`
- confidence should be discounted compared with independent roles
- produces `character_notes`, `world_notes`, `plot_notes`, `style_notes`,
  `protected_risks`, and `recommended_findings`
- may feed fusion when enabled

Full Table Read:

- explicit important-scene opt-in only
- never a normal-turn default
- call budget must be visible before use
- emits proposals and consensus notes, not direct final prose
- remains future work until lower modes are stable

## Protection And Semantic Guard Split

Regex/protected-span protection handles structural risks:

- images
- status windows
- code
- JSON
- RisuAI/module markers
- user protected regex patterns

AI/Fusion guards handle semantic and RP risks:

- voice drift
- private knowledge leakage
- user agency takeover
- secret/POV breach
- causality or scene drift
- weak or missing ending
- mechanical or meta prose artifacts
- style/rhythm degradation

Both are required. Regex protection alone is not enough, and semantic LLM
review without protected segmentation is structurally destructive.

## Implementation Sequence Lock

The next implementation sequence should be:

1. Add `secret_pov_guard` to the default `fusion_enhance` reader set.
2. Strengthen agency, POV, meta-leak, and mechanical-artifact prompt rules.
3. Add deterministic router signals and trace-visible role reasons.
4. Connect `single_panel_table_read` as an optional fusion findings source.
5. Mark single-panel findings as simulated and discount their confidence.
6. Add Table Read escalation recommendations for high-risk important scenes.
7. Keep full Table Read deferred until opt-in cost controls and lower-mode
   verifier behavior are stable.

Implementation may still be sliced smaller, but it must preserve this order of
authority:

```text
protected contract
> user max mode and toggles
> deterministic JS router
> specialist role findings/proposals
> fusion director
> segment composer
> JS verifier
> returned enhanced output
```

## Acceptance Criteria

A build is aligned with this contract only if:

- `fusion_enhance` can return a verified enhanced output, not just advice.
- Trace shows why each role was executed or skipped.
- Trace shows consensus, conflict, gap, unique insight, director score, patch
  accept/reject reason, and verifier result.
- Protected spans are preserved exactly.
- Structurally broken enhancement triggers technical rollback; uncertainty alone
  must not become advice-only behavior when grounded revision candidates exist.
- Character, agency, continuity, POV/secret, style, and mechanical artifact
  risks have explicit handling paths.
- Table Read does not silently increase cost or directly write final prose.
