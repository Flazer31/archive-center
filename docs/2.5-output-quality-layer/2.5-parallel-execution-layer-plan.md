# 2.5 Parallel Execution Layer Plan

Status: S8 initial implementation completed

Date: 2026-06-27

## Goal

S8 adds a bounded scheduler for independent 2.5 work. The purpose is to let
small specialized AI roles run side by side when they do not depend on each
other, while preserving the existing safety contract and pass-through behavior.

This layer must not force low-cost models. Each role profile still chooses its
own provider, model, and tier. Low-cost specialist models are the default
design target, but premium or frontier models remain valid per-role choices.

## Execution Modes

- `auto_parallel`: default. Run independent context and reader work with the
  configured concurrency limits.
- `conservative_parallel`: cap effective concurrency at 2 for safer provider
  usage.
- `serial_debug`: force one-at-a-time execution for debugging and trace review.

The UI exposes separate limits for runtime context collection and reader AI
calls.

## Current Implementation

- Added a reusable `runConcurrentBatch` scheduler with per-item trace.
- Runtime context collection now runs independent character and database reads
  in one context batch.
- Current-chat collection remains a dependent second context batch because it
  can use character fallback data.
- Reader roles in `audit_only`, `standard_check`, `light_patch`, and
  `single_panel` are now dispatched through a reader AI batch.
- Deterministic Director merge still runs only after all reader findings are
  collected.
- Patch planner and verifier stay sequential because they depend on the merged
  decision and candidate output.
- Trace UI now shows parallel batches, execution mode, concurrency, elapsed
  time, item status, and batch failures.

## Safety Boundaries

- Failed items are isolated to the item or role; the whole turn keeps the
  original output unless an already allowed and verified light patch succeeds.
- `check_only` and `audit_only` still return the original output exactly.
- Streaming downgrade, auxiliary/helper bypass, protected spans, JS verifier,
  patch hard caps, and idempotency rules are unchanged.
- No global "max AI calls" cap is reintroduced. Which roles run is still
  controlled by mode plus enabled/configured role profiles.
- Same-provider throttling is not implemented yet; current limits are per batch.
