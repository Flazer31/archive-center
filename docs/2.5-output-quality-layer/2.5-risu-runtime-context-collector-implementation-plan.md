# 2.5 Risu Runtime Context Collector Implementation Plan

Status: active implementation checklist

Date added: 2026-06-27

This document tracks the next 2.5 work slice: adding a read-only RisuAI runtime
context collector to the standalone output quality plugin.

The collector exists so the existing readers can inspect the same kinds of
runtime material RisuAI users expect the plugin to understand: request payload,
system context, recent chat, character card, persona, current chat, lorebook,
module lore, and Supa/Hypa/current-chat memory-like fields.

It must remain read-only. It must not become a memory system, prompt replacer,
or whole-output rewriter.

## Scope

Target runtime file:

- `source/Risu Output Quality Layer 2.5.js`

Reference examples:

- `_archive/reference/external-examples/example/risu-multiagent.js`
- `_archive/reference/external-examples/example/risu_agents.js`
- `_archive/reference/external-examples/example/Serial_Gradation_Agents_for_RP.js`
- `_archive/reference/external-examples/example/multiagent-full-v0.8.1`

Reference use rule:

- Use the examples only for behavior patterns and API discovery.
- Do not copy their pipeline, prompt text, UI, or large code blocks.
- Keep the 2.5 output layer stricter than the examples: findings first,
  bounded patching only, exact protected-span preservation, and original-output
  fallback.

## Safety Contract

- The collector is read-only.
- No RisuAI character, chat, database, lorebook, module, memory, or plugin-local
  canonical state may be written by this slice.
- Missing or failing RisuAI APIs must degrade to payload-only context.
- Auxiliary/helper request modes stay pass-through.
- Context snippets are bounded before any AI call.
- Collected context is inspect-only source material for readers.
- `check_only` output must remain byte-identical to the original output.
- Protected output spans remain authoritative even if lorebook or memory context
  suggests another format.

## Implementation Checklist

### S7.1 Empty Collector Structure

Status: completed

Add a deterministic local item named `risu_runtime_context_collector`.

Expected behavior:

- Creates a stable runtime context object.
- Records collector version and source availability placeholders.
- Does not call RisuAI APIs yet.
- Does not change output or reader prompts yet.
- Adds trace entries only.

Done when:

- `check_only` still returns original output.
- Trace can show that the collector exists but has no external sources yet.
- JavaScript syntax check passes.

Evidence:

- Added `runtime_context` to every new trace.
- Added trace UI summary and source rows for the empty collector.
- No RisuAI host APIs are called in this slice.
- No reader prompts or output return paths were changed.
- Bundled Node syntax check passed on `source/Risu Output Quality Layer 2.5.js`.
- Live RisuAI host QA remains the next follow-up after S7.9.

Status: completed

Read only from the existing request payload.

Sources:

- system messages
- recent user/assistant chat
- latest user input
- request type / mode metadata if available

Done when:

- Trace shows payload source availability and char counts.
- No RisuAI host API is required for this step.
- Reader prompts remain unchanged.

Evidence:

- Added payload-only runtime context fill from the existing request context.
- Records source availability for `payload_system`, `payload_recent_chat`, and
  `payload_user_input`.
- Records item counts, char counts, and stable hashes without exposing new

- RisuAI API sources remain skipped with `s7.3_or_later_not_connected_yet`.
- No reader prompt builder was changed.
- Bundled Node syntax check passed on `source/Risu Output Quality Layer 2.5.js`.

### S7.3 Safe RisuAI API Wrappers

Status: completed

Add small wrappers for optional RisuAI host APIs.

Candidate APIs:

- `Risuai.getCharacter()`
- `Risuai.getCharacterFromIndex()`
- `Risuai.getDatabase()`
- `Risuai.getCurrentCharacterIndex()`
- `Risuai.getCurrentChatIndex()`
- `Risuai.getChatFromIndex()`

Done when:

- Each wrapper returns `{ ok, value, source, error }`.
- Missing functions are recorded as skipped, not failures.
- Thrown errors are isolated per source.

Evidence:

- Added `safeRisuApiCall(name, args)`.
- Wrapped `getCharacter`, `getCharacterFromIndex`, `getDatabase`,
  `getCurrentCharacterIndex`, `getCurrentChatIndex`, and `getChatFromIndex`.
- Missing APIs return `api_unavailable` and stay non-fatal.
- Thrown API errors are recorded per source and shown as failed source rows
  when applicable.

### S7.4 Character / Persona / Current Chat Summary

Status: completed

Collect bounded read-only summaries.

Sources:

- character name, nickname, description-like fields, scenario-like fields
- selected persona and persona text
- current chat object through direct API or `character.chats[chatPage]` fallback
- current chat author note / note-like fields

Done when:

- Trace shows character, persona, and current chat source availability.
- Context block is bounded and marked inspect-only.
- No writes occur.

Evidence:

- Added read-only character summary from name, description-like, scenario-like,
  first-message, and system-prompt-like fields.
- Added selected persona summary from database/persona bindings when available.
- Added current chat summary from direct chat API or `character.chats[chatPage]`
  fallback.
- All summaries are bounded and stored only in `runtime_context`.
- No RisuAI write APIs are called.

### S7.5 Lorebook Candidate Collection And Active Matching

Status: completed

Collect lorebook candidates and best-effort active entries.

Sources:

- character lorebook collections
- current chat lorebook collections
- enabled module lorebook collections

Matching rules:

- Use recent chat and latest user input as search material.
- Respect regex-enabled entries where discoverable.
- Catch invalid regex patterns and record them in trace.
- Treat matching as best-effort, not canonical RisuAI truth.
- Do not hardcode plugin-specific exclusions such as Libra `LIBRA_CONTAINER`,
  `LIBRA_DATA_`, or `lmai_` entries by default. Those belong to the user's
  installed Libra plugin state, not to 2.5 core behavior. If needed later, make
  such exclusions user-configurable.

Done when:

- Trace shows candidate count, active count, regex error count, and selected
  source types.
- Invalid regex cannot break the turn.

Evidence:

- Added lorebook candidate collection for character, current chat, and enabled
  module collections.
- Added best-effort active matching against latest user input and recent chat.
- Regex-enabled entries and slash-style regex keys are supported.
- Invalid regex patterns are caught, counted, and shown in trace without
  breaking the turn.
- Libra-specific managed lorebook filtering was intentionally not added to the
  default collector.
- Mock regex isolation test passed with output unchanged.

- active lore count

- Rows show source/API name, item count, char count, failure reason, and regex

### S7.8 Inspect-Only Context Block For Reader Prompts

Status: completed

Connect bounded runtime context to the existing reader prompt builders.

Rules:

- The block is labeled as read-only source material.
- Readers must still return findings, not rewrites.
- The block must be omitted when disabled, empty, or unsafe.
- The block must be included in trace with char count and source list.

Done when:

- Character Reader, Plot/Continuity Reader, Style Reader, Director, and Table
  Read can receive the same bounded context block.
- `check_only` output remains unchanged.

Evidence:

- Added `runtime_context_block` to output review packets.
- Added a labeled `Risu Runtime Context (Inspect Only)` source block to role
  prompts.
- The block is bounded and states that it is read-only inspect-only material.
- Readers still receive the same JSON-only finding instructions and cannot
  directly replace output.

### S7.9 Pass-Through Verification

Status: completed

Re-run pass-through validation after context collection and prompt connection.

Required checks:

- `check_only` returns original output exactly.
- Image tags are unchanged.
- Status/interface blocks are unchanged.
- Code fences and JSON-like blocks remain balanced.
- RisuAI markers and user regex protected spans are unchanged.
- Trace records context availability without applying patches.

Done when:

- The implementation can safely move on to context-aware light patch behavior.

Evidence:

- Bundled Node syntax check passed.
- Mock RisuAI beforeRequest/afterRequest pass-through test passed.
- The mock pass-through test included system context, recent user input,
  character, persona, current chat, module lore, active lore matching,
  Hypa-like memory, JSON code fence, and image tag preservation.
- A second mock test confirmed invalid lorebook regex is traced and isolated
  while output remains unchanged.
- Live RisuAI host QA is still recommended because host object field names can
  vary by RisuAI version and user setup.
