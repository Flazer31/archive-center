# 2.5 Role Call Catalog

Status: active 2.5 implementation contract

This document defines the default callable items for the standalone 2.5 RisuAI
output quality plugin.

The plugin is distributed as one `.js` file, but every AI lane is configured as
an independent role profile. Each role can use a different endpoint, provider,
model, temperature, timeout, prompt, and fallback policy.

## Non-Negotiable Runtime Rules

- The user's raw input is never rewritten or replaced.
- The main model draft is the source text.
- AI roles return guidance, findings, decisions, or bounded patches. They do
  not return an unbounded whole replacement unless a future explicit mode
  allows it.
- If a role profile is missing, invalid, timed out, or returns invalid JSON,
  only that role degrades.
- Plugin failure, reviewer failure, provider failure, or Archive Center adapter
  failure must pass the original RisuAI output through unchanged.
- Protected spans are masked before AI review and restored after verification.
- Trace must show the effective role profile used for every AI call.

## Shared Role Profile Shape

Every AI role uses this effective config shape after preset inheritance:

```json
{
  "role_id": "character_reader",
  "enabled": true,
  "provider": "openai_compatible",
  "endpoint": "",
  "api_key_ref": "",
  "model": "",
  "temperature": 0.2,
  "max_output_tokens": 1200,
  "timeout_ms": 45000,
  "retry_count": 0,
  "system_prompt": "",
  "task_prompt_template": "",
  "response_schema": "structured_findings.v1",
  "fallback_policy": "skip_role_keep_original",
  "cost_tier": "standard"
}
```

Allowed fallback policies:

- `skip_role_keep_original`: skip this role and continue with other roles.
- `audit_only_no_patch`: keep findings visible but do not patch.
- `director_reject_findings`: Director ignores this role's invalid findings.
- `verifier_revert_to_original`: any unsafe patch reverts to original output.
- `disable_mode_for_turn`: downgrade the current turn to check-only or off.

## Built-In Presets

These presets are convenience defaults only. Per-role values may override every
field.

| Preset | Purpose | Suggested temperature | Suggested timeout |
|---|---|---:|---:|
| `preset_router_fast` | cheap classification / routing | 0.0 | 15000 ms |
| `preset_guidance` | support-only input guidance | 0.2 | 30000 ms |
| `preset_reader` | reader findings | 0.2 | 45000 ms |
| `preset_style_reader` | style-only findings | 0.3 | 30000 ms |
| `preset_director` | arbitration | 0.1 | 45000 ms |
| `preset_patch` | bounded patch proposal | 0.2 | 45000 ms |
| `preset_verifier` | optional LLM verification | 0.0 | 30000 ms |
| `preset_table_read` | heavier rehearsal / panel | 0.2 | 90000 ms |

## Deterministic Local Items

These run inside the JS plugin and do not require an AI endpoint.

| Item ID | Stage | Default | Purpose |
|---|---|---|---|
| `auxiliary_request_bypass` | before/after hook | on | Ignore helper/model/image/module requests that are not the main RP output. |
| `protected_envelope_guardian` | pre-review + post-patch | on | Mask and restore protected tags, code, JSON, image markers, and user regex spans. |
| `local_response_classifier` | pre-review | on | Fast heuristic route before optional AI router. |
| `risu_runtime_context_collector` | pre-review context | staged | Read accessible RisuAI request, character, persona, lorebook, module, current chat, and memory-like context as bounded inspect-only source material. |
| `segmenter` | pre-review | on | Split output into protected, mutable, and inspect-only segments. |
| `patch_applier` | post-review | on | Apply accepted bounded patches only to stable mutable spans. |
| `no_regression_gate` | post-patch | on | Reject damage to protected spans, final paragraph, language, length, structure, or duplicate blocks. |
| `trace_ledger` | all stages | on | Record mode, roles, effective profiles, findings, patches, reverts, and degraded reasons. |

## Risu Runtime Context Collector

The 2.5 plugin must be standalone, but it should not ignore RisuAI context when
the host exposes it. `risu_runtime_context_collector` is a deterministic local
item that gathers a bounded read-only snapshot before AI review.

The collector may use:

- the `messages` payload for system context, recent visible chat, and latest
  user input
- `Risuai.getCharacter()` or index-based character access when available
- `Risuai.getDatabase()` for personas, selected persona, enabled modules, and
  global variables when available
- `Risuai.getCurrentCharacterIndex()`, `Risuai.getCurrentChatIndex()`, and
  `Risuai.getChatFromIndex()` for the current chat when available
- fallback `character.chats[chatPage]` when direct current-chat APIs are absent
- character, chat, and enabled-module lorebook collections
- current-chat memory-like fields such as SupaMemory, HypaMemory, summary, note,
  and last-memory fields

It must not:

- write to RisuAI character, chat, lorebook, module, database, or memory fields
- treat collected context as canonical truth beyond the current turn
- allow collected context to override protected output preservation
- run during auxiliary helper requests that should pass through
- fail the main response when any context API is missing

The collector output should include:

```json
{
  "available": true,
  "sources": {
    "payload_system": { "available": true, "chars": 1200 },
    "character": { "available": true, "source": "getCharacter", "chars": 1800 },
    "persona": { "available": true, "source": "getDatabase.selectedPersona", "chars": 600 },
    "current_chat": { "available": true, "source": "getChatFromIndex", "chars": 900 },
    "active_lorebook": { "available": true, "candidate_count": 42, "active_count": 5 },
    "memory_snapshot": { "available": true, "fields": ["hypaMemoryData", "summary"], "chars": 2400 }
  },
  "trace_notes": [],
  "bounded_context_block": ""
}
```

AI role prompts may receive `bounded_context_block` as source material, but
reviewers still return findings only. In check-only mode, this collector must
not change output.

## Protection Lessons From Example Plugins

The reference multi-agent plugins show useful patterns, but none of them should
be copied as-is for 2.5.

### `risu-multiagent.js`

Useful:

- It is mostly `beforeRequest` guidance-only.
- It wraps source material in labeled source blocks.
- It tells agent models not to write the final RP response.

Limit:

- Because it does not post-process the final output, it avoids many corruption
  risks by not touching the assistant answer.
- It does not provide a deterministic protected-span verifier for image tags,
  regex-controlled structures, status blocks, or final output format.

2.5 lesson: guidance-only mode is safe, but output patching requires additional
code-level protection.

### `risu_agents.js`

Useful:

- It has auxiliary request bypass for some helper types.
- It catches invalid lorebook regex patterns instead of crashing.
- It fail-opens on pipeline errors.

Limit:

- Its post-agent polish mode can replace the current response with the raw
  model output.
- The bypass list is too narrow for 2.5 because it does not broadly cover image,
  regex, module, inlay, title, or other helper request types.
- It relies heavily on prompt instructions rather than deterministic output
  preservation.

2.5 lesson: post-processing must not accept a raw replacement without masking,
bounded patching, and verification.

### `Serial_Gradation_Agents_for_RP.js`

Useful:

- It classifies many helper request types such as image, regex, module,
  embedding, translation, summary, memory, and aux as pass-through.
- It strips hidden assistant reasoning blocks from recent chat history.
- It detects thinking-only provider responses and retries with thinking disabled.
- It treats image payloads and large binary/base64/SVG-like payloads as
  pass-through.
- Its post-processing guard rejects shortened, incomplete, hidden-reasoning, or
  image/status-command-removing revisions.
- It has audit/off/rewrite modes for afterRequest.

Limit:

- Some guards are pattern-based and not a true protected-span mask/restore
  system.
- Rewrite mode can still pass full revised responses through if guards accept
  them.
- Image/status protection is mostly presence-based, not exact byte-for-byte
  preservation.

2.5 lesson: this is the closest reference, but 2.5 must be stricter. It should
mask protected spans before AI review, restore them after patching, and verify
exact preservation.

## Required Protected Span Categories

`protected_envelope_guardian` must protect at least:

- image generation tags and commands, including `<img ...>`, `<img cmd=...>`,
  markdown image links, NovelAI/LightBoard-style image command blocks, and
  user-defined image regex patterns
- RisuAI/module/status/interface blocks
- lorebook or module control blocks that are not ordinary prose
- regex-controlled output markers
- code fences and inline code where exact syntax matters
- JSON/YAML/XML-like structured payloads
- hidden/system/plugin markers
- provider thinking/reasoning fields when they appear in raw provider payloads
- user-defined protected regex patterns

The protection must be structural:

```text
original output
-> detect protected spans
-> replace each with stable placeholder
-> let AI readers inspect placeholder context only
-> apply bounded patches to mutable prose only
-> restore exact protected text
-> verify exact protected text equality
-> fail open to original output on any mismatch
```

Prompt instructions may remind readers to preserve these spans, but prompt
instructions are not sufficient. Exact preservation must be enforced in JS.

## AI Role Items

### 1. `input_guidance_compiler`

Stage: before main model

Default mode: optional, `guidance_only`

Purpose: create support-only guidance for the main model without replacing the
raw user input.

Input:

- current user input
- recent visible chat
- character/persona/scenario/lorebook context when available
- plugin settings

Output schema: `input_guidance_contract.v1`

Must return:

- `user_intent_contract`
- `scene_pressure`
- `character_constraints`
- `world_constraints`
- `style_constraints`
- `do_not_assume`
- `ooc_or_meta_boundary`

Must not return:

- rewritten user input
- final assistant prose
- hidden canonical truth

Fallback: `skip_role_keep_original`

### 2. `ai_response_router`

Stage: before review

Default mode: optional, usually off until local routing is insufficient

Purpose: classify the completed draft and choose which readers should run.

Output schema: `route_decision.v1`

Must return:

- `response_type`
- `recommended_roles`
- `risk_flags`
- `protected_heavy`
- `skip_reason`

Fallback: use `local_response_classifier`.

### 3. `character_reader`

Stage: output check

Default mode: on in Standard and Light Patch

Purpose: inspect character voice, emotional posture, knowledge boundary,
relationship behavior, and persona drift.

Output schema: `structured_findings.v1`

Patch authority: limited voice or knowledge-boundary suggestions only.

Must not:

- invent new facts
- decide the next user action
- rewrite the whole response

Fallback: `skip_role_keep_original`

### 4. `plot_continuity_reader`

Stage: output check

Default mode: on in Standard and Light Patch

Purpose: inspect whether the draft ignored the user action, dropped the scene
goal, contradicted recent visible context, or skipped an unresolved promise.

Output schema: `structured_findings.v1`

Patch authority: small continuity repair only.

Fallback: `skip_role_keep_original`

### 5. `style_reader`

Stage: output check

Default mode: on in Standard and Light Patch

Purpose: inspect repetition, register mismatch, awkward phrasing, and tone
smoothness while preserving the user's preferred style.

Output schema: `structured_findings.v1`

Patch authority: safest initial patch domain, but still bounded to mutable
segments.

Fallback: `skip_role_keep_original`

### 6. `world_reader`

Stage: output check

Default mode: check-only in early 2.5

Purpose: inspect local world rules, social laws, institutions, magic/technology
rules, geography, physical constraints, and faction constraints when enough
context exists.

Output schema: `structured_findings.v1`

Patch authority: check-only until the context pipeline is reliable.

Fallback: `skip_role_keep_original`

### 7. `scene_anchor_guard`

Stage: output check

Default mode: check-only after MVP core is stable

Purpose: inspect who is present, where the scene is, who is speaking, physical
posture, immediate pressure, and scene time continuity.

Output schema: `scene_anchor_findings.v1`

Patch authority: limited anchor clarification only after verifier confidence is
high.

Fallback: `skip_role_keep_original`

### 8. `secret_pov_guard`

Stage: output check

Default mode: optional check-only

Purpose: detect leaks of secrets, private thoughts, hidden narrator knowledge,
system/meta text, or information the current POV should not know.

Output schema: `structured_findings.v1`

Patch authority: delete or soften leak only when the protected span and user
intent allow it.

Fallback: `skip_role_keep_original`

### 9. `director_arbiter`

Stage: finding arbitration

Default mode: on when two or more AI readers ran, optional when only one reader
ran

Purpose: merge reader findings, reject conflicts, rank severity, and choose
whether patching is safe.

Output schema: `director_decision.v1`

Must return:

- `accepted_findings`
- `rejected_findings`
- `deferred_findings`
- `patch_plan_allowed`
- `reason`
- `downgrade_mode`

Must not:

- write final scene prose
- override verifier veto
- apply changes directly

Fallback: deterministic priority merge; if uncertain, no patch.

### 10. `patch_planner`

Stage: output enhance

Default mode: off in Audit, on only in Light Patch or higher

Purpose: convert accepted findings into bounded patch proposals for mutable
segments.

Output schema: `bounded_patches.v1`

Must return patches like:

```json
{
  "patches": [
    {
      "segment_id": "mutable_3",
      "patch_type": "replace",
      "start_hint": "short exact text before",
      "end_hint": "short exact text after",
      "original_quote": "text to change",
      "replacement": "bounded replacement",
      "reason": "why this patch is safer"
    }
  ]
}
```

Must not return a whole `final_output` in MVP modes.

Fallback: no patch, keep original.

### 11. `llm_verifier`

Stage: verification

Default mode: optional; deterministic verifier always runs first

Purpose: provide an extra semantic veto when deterministic checks pass but the
change is still risky.

Output schema: `verifier_decision.v1`

Must return:

- `safe_to_apply`
- `risk_reasons`
- `protected_integrity_ok`
- `user_intent_preserved`
- `meaning_not_worsened`

Fallback: deterministic verifier result only. If verifier is required by mode
and missing, revert to original.

### 12. `single_panel_table_read`

Stage: advanced review

Default mode: off for MVP

Purpose: one model simulates a lightweight table read panel and returns
structured findings. This is not full per-character Table Read.

Output schema: `table_read_panel.v1`

Must return:

- `character_notes`
- `world_notes`
- `plot_notes`
- `style_notes`
- `protected_risks`
- `recommended_findings`

Fallback: skip table read.

### 13. `full_table_read_director`

Stage: future deep/rehearsal mode

Default mode: future only

Purpose: coordinate character director and per-character readers for important
scenes.

Output schema: `full_table_read_decision.v1`

Fallback: skip deep table read and keep lower mode results.

### 14. `individual_character_reader`

Stage: future full table read

Default mode: future only

Purpose: inspect one character's dialogue, knowledge, posture, and subtext.
Each active character may have its own generated role instance and may use its
own role profile.

Output schema: `character_specific_findings.v1`

Fallback: skip that character reader only.

## Mode-To-Role Defaults

| Mode | AI calls | Patch behavior |
|---|---|---|
| `off` | none | original output |
| `guidance_only` | `input_guidance_compiler` before main model | no output patch |
| `audit_only` | selected readers only | no output patch |
| `standard_check` | router + selected readers + optional director | no output patch |
| `light_patch` | selected readers + director + patch planner + verifier | bounded patch or original |
| `single_panel` | single panel + director + optional patch planner | bounded patch or original |
| `deep_rehearsal` | future full table read roles | bounded patch or original |

Default first implementation target:

```text
protected_envelope_guardian
-> local_response_classifier
-> character_reader
-> plot_continuity_reader
-> style_reader
-> director_arbiter
-> patch_planner only when light_patch is enabled
-> no_regression_gate
-> trace_ledger
```

## Trace Requirements

Every turn trace must show:

- requested mode
- actual mode after downgrade
- roles selected
- roles skipped and why
- effective endpoint/model/temperature/prompt version for each AI role
- provider call success/failure
- findings returned
- findings accepted/rejected/deferred
- patches proposed
- patches applied
- verifier result
- whether final output is original or patched
- exact fallback reason when original is preserved

## First Implementation Cut

The first coding slice should implement the data structures and UI shape for
all role profiles, but only execute the safest subset:

1. `protected_envelope_guardian`
2. `local_response_classifier`
3. `trace_ledger`
4. `character_reader`
5. `plot_continuity_reader`
6. `style_reader`
7. `director_arbiter`
8. `no_regression_gate`

`patch_planner` should exist in settings and schema, but should remain disabled
until protected span masking and no-regression behavior are proven.

## Next Implementation Cut: Risu Context Access

After S0-S6, the next implementation slice should add
`risu_runtime_context_collector` in audit mode first:

1. Trace-only payload context snapshot.
2. Safe RisuAI API wrappers with per-source failure reasons.
3. Character/persona/current-chat read-only summaries.
4. Lorebook candidate and active-lore matching with regex error isolation.
5. Supa/Hypa/current-chat memory snapshot formatting.
6. Prompt injection as inspect-only source material for existing reader roles.
7. Trace UI rows for sources used, skipped, failed, and bounded char counts.
8. Pass-through verification proving check-only output remains byte-identical.
