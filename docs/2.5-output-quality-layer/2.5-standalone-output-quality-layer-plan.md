# 2.5 Standalone Output Quality Layer Plan

Status: active 2.5 design anchor

This document fixes the next direction after 2.2: build a standalone RisuAI
output quality layer first. Archive Center integration is deferred and must be
treated as an optional adapter, not as the foundation.

The design still includes the MDASH concept. "Standalone first" means the
runtime must not depend on Archive Center services; it does not mean removing
MDASH. The product should be a standalone, MDASH-inspired harness for RisuAI
input guidance, output review, Table Read, guarded patching, verification, and
trace.

The product should not be a simple merge of existing agent examples. Its
differentiator is not "more agents"; it is a controlled output quality pipeline
that builds contracts, checks the draft against those contracts, patches only
safe mutable regions, and falls back to the original output when safety is not
proven.

## Product Boundary

The 2.5 output layer must work without Archive Center.

Standalone inputs:

- current RisuAI request messages
- current user input
- recent visible chat
- character card / persona / scenario when accessible
- lorebook or activated context when accessible
- plugin settings
- provider presets

Optional future inputs:

- Archive Center memory packet
- world rules
- subjective character memory
- relationship state
- long-term hierarchy summaries

The optional inputs must be added through an adapter later. They must not shape
the first implementation.

## Core Principle

Do not rewrite the user's raw input.

Input preprocessing may add support context, but it must never replace,
summarize, sanitize, or reinterpret the user message as the new source of truth.

Output processing may improve the assistant draft, but it must never perform an
unbounded whole-response replacement. It must segment the output, protect
non-prose structures, patch only mutable prose when safe, verify the result, and
return the original draft on failure.

## MDASH Functional Interpretation

For 2.5, MDASH should be interpreted as a harness pattern, not as uncontrolled
multi-model rewriting.

The historical broad MDASH map was:

```text
Input MDASH
-> Memory Read MDASH
-> Main LLM
-> Output Check MDASH
-> Table Read
-> Memory Write MDASH
-> Committer
```

For this standalone RisuAI product, the active 2.5 subset should be:

```text
Input Enhance MDASH
-> RisuAI native prompt/context assembly
-> Main LLM
-> Output Check MDASH
-> Table Read MDASH when needed
-> Output Enhance MDASH
-> Verifier / protected output guard
-> RisuAI display/save
```

The following MDASH areas are not part of the first standalone release:

- Memory Read replacement
- Memory Write
- Committer
- historical turn rewrite
- DB or canonical state mutation
- autonomous retrieval authority

They can exist later as adapters or separate phases. In 2.5, MDASH means
role-separated review, structured findings, bounded patch proposals, verifier
veto, and run trace.

## High-Level Pipeline

```text
RisuAI original request
-> context reader
-> input preprocessing / pre-output guidance
-> main model draft
-> output review
-> patch planning
-> protected segment guard
-> verifier
-> patched output or original fallback
```

The feature is a quality-control layer, not a memory system and not a second
main model hidden behind RisuAI.

## Three Main Lanes

### 1. Input Preprocessing

Purpose: help the main model produce a better answer before it writes.

This lane reads the current RisuAI state and produces support-only guidance:

- scene guidance
- user intent contract
- character contract
- world contract
- plot and pacing contract
- continuity risks
- secret and POV boundaries
- output format reminders

It must not produce:

- rewritten user input
- canonical truth
- memory writes
- final response text that bypasses the main model

MDASH name: `Input Enhance MDASH`.

The safer 2.5 interpretation is not "normalized input replaces raw input".
Instead:

```text
raw user input stays unchanged
-> intent / constraint / scene pressure extraction
-> support-only guidance block
-> original request plus guidance goes to the main model
```

Useful outputs:

- `user_intent_contract`
- `style_constraints`
- `scene_pressure`
- `action_candidates`
- `ambiguity_notes`
- `ooc_or_meta_separation`
- `do_not_assume`

### 2. Output Post-Review

Purpose: inspect the main model draft after it exists.

Reviewers identify issues; they do not directly replace the answer.

Review lanes:

- Character Reader: voice, emotion, knowledge boundary, relationship behavior.
- World Reader: setting law, social rules, geography, institutions, magic or
  technology constraints.
- Plot Reader: scene goal, pacing, unresolved tension, next-turn openness.
- Continuity Reader: recent context, repeated scene drift, missing carryover.
- Secret Guard: hidden/private knowledge leakage and meta leakage.
- Structure Guard: image tags, status blocks, Chatindex, module tags, regex
  markers, JSON/code blocks, and user-requested output forms.

MDASH name: `Output Check MDASH`.

The output check decides:

- whether any fix is needed
- whether Table Read is warranted
- which reader roles should run
- whether patching is unsafe and should be skipped
- what evidence segment each issue points to

### 3. Output Material Processing

Purpose: decide whether and how the draft can be changed.

Steps:

```text
segment output
-> mark protected / mutable / inspect-only regions
-> merge reviewer findings
-> build minimal patches
-> apply patches only to mutable regions
-> verify protected regions and semantic safety
-> return patched output or original output
```

This lane is where the product should be stronger than ordinary agent bundles.
It treats output as structured material, not as a free text blob.

MDASH names:

- `Table Read MDASH`: role discussion/review, not roleplay continuation.
- `Output Enhance MDASH`: convert findings into bounded patch proposals.
- `Verifier`: deterministic and optional model-assisted veto layer.

Table Read must not write new scene prose directly. It should read the draft,
exchange structured opinions, and produce editorial constraints for Output
Enhance.

Output Enhance should prefer:

```json
{
  "patches": [
    {
      "segment_id": "mutable_3",
      "replacement": "patched prose for this segment only",
      "reason": "voice fit and user constraint repair"
    }
  ]
}
```

over a whole `final_output`.

## Contract Objects

The layer should compile request context into small explicit contracts.

### User Intent Contract

Captures what the current user input is asking for without rewriting it.

Examples:

- answer a question
- continue a scene
- react emotionally
- explain world lore
- resolve a tactical decision
- keep the current scene open
- avoid deciding for the user

### Character Contract

Per active character:

- known facts
- unknown facts
- private facts that must not leak
- current emotion
- current relationship posture
- voice constraints
- allowed actions this turn
- forbidden assumptions

### World Contract

Setting-level constraints:

- physical laws
- magic / technology rules
- faction rules
- geography and location constraints
- social law, taboo, rank, ritual, economics
- survival, resource, or progression systems

World rules must be abstract and setting-adaptive. They must not be hardcoded to
one genre such as fantasy, academy, company drama, apocalypse, or dungeon
progression.

### Plot Contract

Scene-level progression constraints:

- current scene goal
- immediate pressure
- unresolved tension
- active question
- what should remain open
- what should not be skipped

### Protected Output Map

The draft is segmented before any patch is allowed.

Protected examples:

- image generation tags
- status windows
- Chatindex / timestamp frames
- module tags
- regex-facing markers
- JSON/code blocks
- command-like blocks
- user-requested fixed templates

Mutable examples:

- normal prose paragraphs
- dialogue text
- descriptive transitions

Inspect-only examples:

- quoted user text
- logs or debug blocks
- external module outputs

## Safety Rules

- Fail open to the original output.
- No model call failure may erase or replace the main response.
- No post-review result becomes memory by itself.
- No reviewer is allowed to rewrite the whole response directly.
- No patch is accepted if protected regions changed.
- No patch is accepted if the revised output is suspiciously shorter.
- No patch is accepted if it changes language, format, scene, or user agency
  without explicit justification.
- No patch is accepted if it introduces unsourced facts.
- Streaming mode must degrade safely when clean pre-display replacement is not
  possible.

## Execution Modes

| Mode | Purpose | Output Change |
|---|---|---|
| `off` | no quality layer | none |
| `guidance_only` | input preprocessing only | none by post layer |
| `check_only` | output review and trace only | none |
| `light_patch` | review plus minimal safe patch | guarded partial patch |
| `single_panel` | one-call simulated table read | guarded partial patch |
| `full_table_read` | multi-agent high quality review | guarded partial patch |

Users choose the maximum mode. Runtime may downgrade for streaming, cost,
latency, missing provider settings, auxiliary request type, or safety concerns.

Approximate MDASH call budget:

| Mode | Extra calls before main | Extra calls after main | Notes |
|---|---:|---:|---|
| `guidance_only` | 0-1 | 0 | support-only guidance can be heuristic first |
| `check_only` | 0-1 | 1 | review only |
| `light_patch` | 0-1 | 1-2 | output check plus patch/verifier |
| `single_panel` | 0-1 | 2-3 | one moderator simulates Table Read |
| `full_table_read` | 0-1 | `2C + 5~9` | C = active character count |

Full Table Read is not a normal-turn default. It is for important scenes where
the user accepts latency and cost.

## Agent Roles

Minimum viable roles:

- Context Reader
- Guidance Writer
- Output Reviewer
- Structure Guard
- Patch Planner
- Verifier

Expanded roles:

- Scene Reader
- Character Director
- Individual Character Reader
- World Reader
- Plot Reader
- Continuity Reader
- Secret Guard
- Style/Rhythm Reader

The first implementation should avoid full multi-agent cost. Start with compact
roles and make the trace explain which roles actually ran.

## Relation To Example Plugins

Useful ideas:

- beforeRequest support context from multi-agent examples
- role separation from Serial Gradation
- provider presets and run inspector from agents examples
- auxiliary request bypass for memory, translation, image, module, and helper
  requests
- fail-open behavior

Do not copy:

- unbounded whole-output replacement
- blocking the main model to protect quota
- plugin-local memory as canonical truth
- hidden expensive calls on every turn
- hardcoded module names as the only routing authority
- monolithic single-file growth as the default architecture

## Architecture Direction

The implementation should be a standalone RisuAI plugin first.

Archive Center can later provide a read-only adapter, but the core should not
depend on Archive Center.js, MariaDB, ChromaDB, or Archive Center backend
routes.

### Single JS Plugin Constraint

The 2.5 deliverable must be a standalone plugin shipped as one RisuAI `.js`
file. It must not be implemented by expanding `Archive Center.js`, and it must
not require users to install Archive Center before it can run.

This is a distribution/runtime constraint, not permission to build another
unstructured giant file. The single JS file should keep clear internal modules
or namespaces for the core, adapter, provider registry, verifier, UI, and trace
ledger. Shared helpers should remain small, pure, and testable where possible.

Archive Center integration, if added later, must be an optional read-only
adapter inside this standalone plugin. Failure to reach Archive Center must
downgrade the feature, not block or replace the main RisuAI response.

### Per-Role AI Profile Constraint

Even though the deliverable is a single JS file, each review or processing lane
must be able to use its own AI profile. The plugin must not assume one global
LLM setting is correct for every role.

Each role profile should be configurable independently:

- enabled / disabled
- provider type
- endpoint
- API key or auth token reference
- model
- temperature
- max output tokens or chars
- timeout
- retry count
- system prompt
- task prompt template
- response schema expectation
- fallback policy

At minimum, separate profiles are needed for:

- Input Enhance / guidance compiler
- Character Reader
- Plot / Continuity Reader
- Style Reader
- World Reader
- Scene Anchor Guard
- Director Arbiter
- Patch Planner / Output Enhance
- Verifier
- Full Table Read roles when enabled later

The authoritative default role list, schemas, fallback policies, and mode-to-role
mapping are defined in `2.5-role-call-catalog.md`.

Profiles may inherit from a shared preset for convenience, but runtime behavior
must resolve and display the exact effective profile for each role. If a role's
required endpoint, key, model, or prompt is missing, that role must be skipped
with a traceable degraded reason. The plugin must not silently call an unrelated
global provider or replace the main output because a role profile is invalid.

Preferred shape:

```text
StandaloneOutputCore
  - request classifier
  - contract compiler
  - prompt builders
  - structured output parsers
  - segmenter
  - patch planner
  - verifier
  - run ledger

RisuAIAdapter
  - beforeRequest hook
  - afterRequest hook
  - current chat extraction
  - auxiliary request bypass
  - settings UI

ProviderRegistry
  - OpenAI-compatible
  - Anthropic
  - Gemini / Vertex
  - Ollama-compatible
  - custom endpoint

RoleProfileRegistry
  - per-role endpoint/model/prompt/temperature
  - preset inheritance
  - effective config resolver
  - missing-config downgrade reasons

OptionalArchiveAdapter
  - disabled by default
  - read-only context packet
  - no canonical writes
```

The final product name should not be tied to Archive Center or MDASH. The name
can be chosen later.

## Implementation Order

1. Define plugin shell, settings schema, role profile schema, and run ledger.
2. Implement role profile resolver with per-role endpoint/model/prompt/temperature.
3. Implement auxiliary request bypass and fail-open pass-through.
4. Implement protected segment parser.
5. Implement `Input Enhance MDASH` in `guidance_only` mode.
6. Implement `Output Check MDASH` in `check_only` mode.
7. Implement JS verifier with no LLM patching yet.
8. Implement `Output Enhance MDASH` as minimal patch planner for mutable prose
   only.
9. Add `light_patch` mode.
10. Add one-call `single_panel` Table Read MDASH.
11. Add trace UI for effective role profiles, contracts, findings, patches,
    verifier result, and
    downgrade reason.
12. Add full table read only after the lower modes are stable.
13. Add optional Archive Center read-only adapter after standalone behavior is
    proven.

## Done Criteria

- The deliverable is one standalone RisuAI `.js` plugin file.
- The plugin works without Archive Center.
- The plugin is not implemented as additions to `Archive Center.js`.
- Each role can have a distinct endpoint, model, temperature, and prompt.
- Missing per-role configuration degrades that role only and never replaces the
  main output.
- Raw user input remains unchanged.
- Main output is never erased by plugin/provider/backend failure.
- Image tags, status blocks, Chatindex, module tags, and regex markers survive.
- Check-only mode can explain issues without changing output.
- Light patch mode changes only mutable prose.
- Verifier can veto every unsafe patch.
- Streaming mode has honest degraded behavior.
- Trace shows requested mode, actual mode, roles run, calls used, skipped
  reasons, findings, patches, and fallback reason.
- The design remains modular enough to avoid another giant `Archive Center.js`.
