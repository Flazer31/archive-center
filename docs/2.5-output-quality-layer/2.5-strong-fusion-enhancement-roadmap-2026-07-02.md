# 2.5 Strong Fusion Enhancement Roadmap

Date: 2026-07-02

Status: active direction lock after external multi-agent comparison

## Purpose

This document corrects the implementation direction for the 2.5 standalone
RisuAI output quality layer.

The plugin must not become a cautious advisory tool that only reports issues.
The target is a strong multi-agent enhancement pipeline:

```text
RisuAI main output
-> protected segmentation
-> specialist AI revision lanes
-> strong candidate generation
-> Fusion Director consensus/conflict/gap scoring
-> Segment Composer
-> verifier-gated enhanced output
```

The specialist AIs are expected to improve the output, not merely comment on it.
Safety must be enforced at the apply/composer/verifier layer, not by making
every role timid.

## Direction Lock

These rules override weaker wording in older 2.5 docs.

- The product goal is improved final prose, not a trace report.
- `audit_only` exists for diagnosis, but the forward path is `fusion_enhance`.
- Specialist roles should produce concrete revision candidates when they see a
  fixable issue.
- A role should focus on its own lane and may be assertive inside that lane.
- Strong improvement attempts are allowed as long as they are structured,
  segment-bounded, and verifier-gated.
- The system should not force low-cost models. Low-cost models are the common
  target, but users may assign frontier or premium models to any role.
- The plugin should learn from multi-agent examples, but must not copy their
  code or their unsafe whole-output replacement behavior.

## What "Strong" Means

Strong enhancement means:

- Character roles actively repair voice, relationship tone, knowledge boundary,
  agency pressure, and emotional posture.
- Continuity roles actively repair missing bridges, timeline gaps, action order,
  final paragraph carryover, and scene logic.
- Style roles actively repair rhythm, repetition, stiffness, translationese,
  mechanical phrasing, and weak transitions.
- World/lore roles actively repair visible lore contradictions, world-rule
  mismatch, location drift, and terminology errors.
- Secret/POV roles actively repair private-knowledge leaks, narrator overreach,
  hidden-identity exposure, and POV boundary errors.
- Table Read roles act as higher-cost cross-lane review and escalation, not as
  silent direct rewriters.

Strong does not mean:

- unbounded full-output rewrite by default
- deleting protected markers, images, JSON, code blocks, or RisuAI structures
- inventing lore to fix uncertainty
- allowing one unstable model response to replace a good base draft
- treating low-severity stylistic taste as enough to rewrite the final output

## External Example Lessons

### Serial Gradation Agents for RP

Useful lesson:

- It makes each stage produce a usable improved draft, then passes that draft to
  the next stage.
- It treats output improvement as the actual product, not as optional notes.

Adopted direction:

- ROQL should add a stronger candidate-generation lane, where roles produce
  revision candidates that can materially improve the answer.

Rejected direction:

- Do not replace the entire RisuAI response with an unchecked serial rewrite.

### risu_agents

Useful lesson:

- It separates parallel pre-agent notes from sequential post-agent editing.
- Post agents directly apply to the current response.

Adopted direction:

- ROQL should keep parallel reader calls for independent analysis, but output
  application must be sequential and conflict-aware through the Segment
  Composer.

Rejected direction:

- Do not let a post agent freely return a full replacement without segment
  verification.

### Eros Tower

Useful lesson:

- Strong context, memory, and state improve output quality when injected before
  or used during post-processing.
- It clearly separates final-output post-processing from state commit.

Adopted direction:

- ROQL should later connect to Archive Center/MariaDB/ChromaDB as a richer
  context source, after the standalone version is stable.
- Current standalone work should keep context read-only and inspect-only.

Rejected direction:

- Do not make long-term memory/state management the immediate core of ROQL.

## Revised Architecture

### 1. Intake and Segmentation

Input:

- RisuAI main model output.
- Payload/system/recent-chat snapshot.
- Read-only RisuAI runtime context.
- Active lorebook and memory snapshots when available.

Output:

- base draft
- protected/mutable/inspect-only segment map
- context manifest with char/token estimates
- risk signals for routing

### 2. Strong Specialist Revision Lanes

Each role receives:

- the base draft
- only relevant context blocks
- mutable segment IDs
- protected segment rules
- role-specific improvement target

Each role returns:

- findings
- evidence quote
- revision lane
- local replacement candidate or segment-level rewrite candidate
- confidence, severity, and reason

Roles should not be told to stay quiet when they see a concrete improvement.
They should be told to be precise and useful.

### 3. Fusion Director

The Director is conservative about application, not about listening.

It must:

- accept strong candidate material from all roles
- score consensus, conflicts, gaps, and unique insights
- prefer secret/POV, continuity, character, and scene-anchor fixes over pure
  taste edits
- prevent unstable or fallback-missing roles from becoming the only apply
  source in Quality mode
- reject unsupported broad rewrites
- explain every accepted and rejected candidate in Trace

### 4. Segment Composer

The composer applies selected candidates in order:

1. protected preservation gate
2. exact or normalized segment match
3. overlap/conflict resolution
4. patch size and segment budget
5. candidate composition
6. JS verifier
7. enhanced return or original fallback

The composer is where safety lives.

### 5. Optional Whole-Response Polish Candidate

A future strong mode may allow a whole-response polish candidate, but it must
not directly replace the output.

Required path:

```text
whole_response_candidate
-> diff extraction
-> segment candidate conversion
-> Segment Composer
-> JS verifier
-> enhanced output
```

This borrows useful power from post-agent examples without accepting their
unsafe direct replacement behavior.

## Revised Patch Policy

The old behavior allowed low-severity, high-confidence single-role patches to
apply too easily. That can create visible quality wobble.

New Quality-mode policy:

- High severity: may apply with one strong role if evidence is exact and the
  verifier passes.
- Medium severity: may apply with one strong role if confidence is high, or if
  another role/table signal supports it.
- Low severity: trace by default; apply only when one of these is true:
  - two or more roles agree
  - Table Read supports it
  - it removes a mechanical artifact, repetition, or obvious broken phrasing
  - it is attached to a higher-priority lane such as secret/POV, continuity,
    character voice, or scene anchor
- Pure taste edits from one role should not apply in Full Quality.
- If the only successful role is unstable, reasoning-only-recovered, or has no
  fallback, the patch should be downgraded or held for Trace.

## Revised Work Order

### P5: Quality Patch Gate

Goal:

- Stop weak low-severity single-role patches from being applied in Full
  Quality while keeping strong medium/high fixes active.

Implementation:

- Add severity/lane/support-aware Director gate.
- Add `apply_strength` or equivalent Trace field.
- Mark low-only unsupported patches as `trace_only_low_support`.
- Keep audit visibility.

Acceptance:

- Full Quality still returns enhanced output when real issues exist.
- Stylistic single-role taste edits no longer replace the output by default.

### P6: Strong Revision Prompt Contract

Goal:

- Make roles produce useful revision candidates, not timid commentary.

Implementation:

- Rewrite role prompts around assertive lane ownership.
- Split audit prompt from fusion prompt.
- For fusion prompts, require a concrete candidate when an issue is fixable.
- Keep protected/inspect-only rules in the system prompt.

Acceptance:

- Character, continuity, style, world, secret/POV, and scene-anchor roles return
  actionable candidates more consistently.
- Empty findings decrease when visible issues exist.

### P7: Role Reliability and Fallback Gate

Goal:

- Prevent unstable roles from becoming single-source apply authority.

Implementation:

- Use model health in Director scoring.
- Add role result quality states: `clean`, `retry_recovered`, `reasoning_only`,
  `timeout`, `fallback_used`, `fallback_missing`.
- Downgrade apply authority for `retry_recovered` and `fallback_missing` in
  Quality mode.

Acceptance:

- Trace clearly shows whether a role was trusted for application or only for
  advisory evidence.

### P8: Segment Candidate Upgrade

Goal:

- Move beyond tiny phrase swaps when a segment needs stronger improvement.

Implementation:

- Allow bounded segment rewrite candidates for mutable prose segments.
- Keep segment-level hard caps.
- Compare candidate segment against original segment with verifier checks.
- Preserve final paragraph and protected blocks.

Acceptance:

- The plugin can improve awkward passages more naturally than exact phrase
  replacements alone.

### P9: Whole-Response Polish Candidate, Diff-Gated

Goal:

- Borrow the useful strength of post-agent polish without direct unsafe
  replacement.

Implementation:

- Add optional polish candidate role.
- Convert its output into segment diffs.
- Reject if diff is too broad, protected-touching, truncating, or format
  destructive.
- Feed surviving diffs into Segment Composer.

Acceptance:

- Whole-response polish can help prose quality, but the returned output is
  still verifier-gated segment composition.

### P10: Table Read as Fusion Escalation

Goal:

- Use Table Read when the scene is important or high-risk, not as a routine
  expensive call.

Implementation:

- Keep single-panel as lightweight support.
- Full Table Read remains explicit opt-in.
- Table findings become support signals for Director scoring.

Acceptance:

- Table Read improves decision quality without silently taking over output.

### P12: Beta Hardening

Goal:

- Prepare for user beta without reducing role capability.

Implementation:

- Presets: Safe, Balanced, Full Quality, Debug, Deep Rehearsal.
- Model fallback UX.
- Trace readability.
- Release zip.
- Known limitation list, including deferred identity/secret-memory issues that
  may require richer Archive Center context later.

Acceptance:

- Beta can be shared as a strong enhancement plugin with clear risks and
  recoverable fallback behavior.

## Current Next Step

P11A aggressive Fusion application is now the immediate direction before beta
hardening.

The next broad test should check whether Full Quality visibly rewrites more
than tiny phrase-level issues while still passing protected segment and JS
verifier gates.

This order keeps the desired direction:

```text
strong AI improvement attempts
+ strict deterministic application gate
= better final output without fragile random rewrites
```

## Implementation Notes

### 2026-07-02 P5 start

P5 must not be interpreted as "be more advisory." It is the opposite:

- Keep MDASH roles assertive.
- Keep OpenRouter Fusion-style support scoring visible.
- Keep Sakana Fugu-style candidate competition as the long-term direction.
- Move conservatism into the Director/Composer/Verifier layer only.

The first P5 code pass should:

- mark each finding with `apply_strength`
- keep weak low-severity unsupported candidates visible in Trace as
  `trace_only_low_support`
- allow high and medium findings to keep driving enhancement
- allow low findings when supported by consensus, Table Read, mechanical/repeat
  cleanup, or a high-priority lane with very high confidence
- make the Trace show why a finding was applied or held

### 2026-07-02 P6 start

P6 locks the prompt contract so fusion roles act as revision workers, not
advisory critics.

Implementation rule:

- Audit modes may diagnose concrete issues without patches.
- Fusion modes must produce bounded revision candidates when a visible lane
  issue is fixable inside one mutable segment.
- Candidate scope may be a phrase, sentence, or short contiguous passage.
- Specialists should attempt strong local repairs; the Director, Segment
  Composer, Quality Patch Gate, and Verifier decide whether a candidate may
  change the returned output.
- Protected spans, inspect-only spans, image tags, code blocks, JSON, RisuAI
  markers, and unsupported fact invention remain forbidden.

This preserves the intended architecture:

```text
MDASH-style specialist lanes
+ OpenRouter Fusion-style support scoring
+ Sakana Fugu-style bounded candidate competition
= strong enhancement attempts with deterministic safety gates
```

### 2026-07-02 P7 start

P7 is not a reduction of specialist role power. It is an apply-authority gate:
roles still produce strong candidates, but unreliable call states cannot apply a
patch alone in Quality profiles.

Implementation rule:

- `clean` and `fallback_used` are trusted states.
- `retry_recovered`, `reasoning_only`, `timeout`, `fallback_missing`, and
  `failed` are limited states.
- Limited states receive a confidence penalty in Director scoring.
- In Quality / Debug All profiles, limited states are trace-only unless the same
  issue has consensus or Table Read support.
- Trace must show role reliability state, score, penalty, and the reason a
  candidate was blocked or allowed.

This keeps the desired direction:

```text
strong role candidates
+ model/retry/fallback reliability gate
+ consensus/table support override
= fewer unstable single-model patches without returning to advice-only mode
```

### 2026-07-02 P8 start

P8 upgrades the Segment Composer so it can accept a bounded
`segment_rewrite` candidate when phrase-level replacement is too weak.

Implementation rule:

- `segment_rewrite` is still one mutable segment only, not whole-output rewrite.
- The model must quote the complete original mutable segment exactly in
  `evidence_quote`.
- The replacement must be the complete replacement segment text.
- A stricter per-segment size cap applies.
- The segment containing the original final paragraph tail cannot be rewritten.
- The existing JS verifier still checks protected spans, structural counts,
  length ratio, final paragraph preservation, duplicate blocks, and markdown
  balance before any enhanced output is returned.

This keeps the direction strong but bounded:

```text
phrase/sentence patches
+ optional small segment_rewrite candidates
+ Segment Composer competition
+ JS verifier
= more natural prose improvement without unsafe full response replacement
```

### 2026-07-02 P9 start

P9 adds a whole-response polish candidate without allowing unsafe whole-output
replacement.

Implementation rule:

- `whole_response_polisher` is a candidate generator, not a final responder.
- Full Quality, Debug All, and Deep Rehearsal presets enable the role; Balanced
  can use it only when the role is explicitly enabled.
- The role returns `whole_response_polish.v1` JSON with a full
  `polished_output` candidate.
- The plugin restores protected placeholders, runs a JS verifier precheck, and
  rejects candidates that change protected content, structure, markdown balance,
  length ratio, or final paragraph preservation.
- The candidate is then compared against the original segment map.
- Only changed mutable segments that preserve segment shape become
  `whole_response_diff_segment` / `segment_rewrite` patches.
- Candidates are rejected if they touch the final paragraph segment, change too
  many segments, or exceed the total changed-character budget.
- Surviving segment diffs enter the existing Fusion Director and Segment
  Composer with normal verifier gates.
- The whole polish output is never returned directly.

This keeps the strong enhancement direction:

```text
whole polish attempt
-> strict diff gate
-> small mutable segment candidates
-> Fusion Director / Segment Composer / JS verifier
= stronger prose polish without giving up output safety
```

### 2026-07-02 P10 start

P10 changes Table Read from routine reader participation into Fusion
escalation support.

Current behavior before P10:

- `single_panel_table_read` could be selected by the router and run in the same
  parallel reader batch as character/continuity/style readers.
- `full_table_read_director` and `individual_character_reader` already stayed
  behind `deep_rehearsal`, explicit opt-in, score gates, and non-streaming
  checks.

Implementation rule:

- In Fusion / Deep Rehearsal, single-panel Table Read waits for the P10
  escalation score instead of joining the first reader batch.
- Debug All and explicit `single_panel` mode may still run it directly for
  testing or intentional one-panel review.
- After initial specialist findings, the plugin computes important-scene and
  risk scores.
- If single-panel opt-in is enabled, the role is runnable, and the score is
  high enough, the single-panel pass runs as one extra support call.
- Its findings are appended, findings-level Fusion is recomputed, and the
  Director decision is recomputed before patch composition.
- Table Read support can add `table_read_support` to matching non-table
  findings when they share segment/domain/lane or exact evidence.
- Full Table Read remains explicit opt-in and deep-rehearsal-only.

This keeps the intended architecture:

```text
specialist findings
-> important/risk scoring
-> optional single-panel support
-> Director recompute with table support
-> optional full table read only in deep_rehearsal opt-in
= stronger consensus without silent expensive takeover
```

### 2026-07-02 P11A start

P11A changes the next direction from beta-only hardening to aggressive Fusion
application before P12 beta hardening.

Reason:

- The product goal is MDASH/Fusion/Fugu-style output improvement, not a
  conservative trace advisor.
- Calling more specialist LLMs than the example multi-agent plugins is not
  justified if the final output only changes one or two words.
- Safety must stay in protected segmentation, Segment Composer, and JS
  verifier gates. Specialist roles and the Director should not be made timid.

Implementation rule:

- Full Quality is allowed to apply more candidates per turn.
- Recovered JSON, reasoning-only retry recovery, or fallback-missing model
  health warnings reduce confidence but no longer automatically force
  trace-only output.
- Low-severity findings with concrete actionable patches may apply when their
  effective confidence clears the aggressive threshold.
- Whole-response polish becomes a normal strong candidate source in Quality
  routing, but its output is still never returned directly.
- Whole-response polish diffs may touch more mutable segments.
- Final paragraph / final beat wording can be improved as long as the JS
  verifier confirms the ending remains present and structurally safe.
- Table Read remains an escalation support source, not a silent final writer.

The intended path is now:

```text
specialist candidates
+ whole-response polish candidate
+ optional Table Read support
-> Fusion Director scoring
-> Segment Composer competition
-> JS verifier
= visibly improved final output, not advice-only notes
```

### 2026-07-02 P11B start

P11B stabilizes Whole-Response Polisher after P11A made it an aggressive
candidate source.

Problem observed in live Trace:

- The role was selected in Full Quality, but a heavy whole-draft polish call
  timed out before producing a candidate.
- The failure did not block direct specialist patches, but it made the P9 path
  unreliable and hard to diagnose.
- Whole-response polish is expensive because it must return a complete
  `polished_output` candidate; if output budget is too small or context is too
  large, the role is structurally likely to fail.

Implementation rule:

- Keep Whole-Response Polisher as a strong candidate source, not an advice
  role.
- Reduce nonessential context for the primary polish call.
- Full Quality / Debug / Deep Rehearsal presets raise the role's minimum output
  token and timeout settings so old saved values do not silently starve it.
- If the primary call times out, returns no final content, or fails JSON parsing,
  run one compact retry with only the core draft, focused segment map, and top
  findings.
- Compact retry may make only 1-3 high-impact changes and must still return a
  complete masked draft candidate.
- If the draft is too long for the configured output budget, skip whole polish
  and let direct segment candidates handle the turn instead of generating a
  likely-truncated unsafe candidate.
- Trace records primary/compact attempts, compact retry reason, and
  failed-before-candidate state.

This keeps the aggressive direction while moving risk back into deterministic
gates:

```text
whole polish primary
-> if timeout/json/content failure: compact retry
-> candidate pre-verifier
-> segment diff extraction
-> Segment Composer
-> final JS verifier
= whole-draft polish contributes only when it is complete and safe
```
