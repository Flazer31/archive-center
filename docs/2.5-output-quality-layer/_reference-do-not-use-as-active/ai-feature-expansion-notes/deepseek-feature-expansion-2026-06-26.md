## 1. 기존 플러그인과의 차별점

기존 플러그인은 크게 세 부류다:

| 부류 | 대표 예 | 한계 |
|---|---|---|
| **단순 후처리기** | 문법 교정, 번역 플러그인 | 전체 재작성, RP 맥락 무시, 구조 태그 파괴 |
| **단일 에이전트 리뷰어** | "응답을 검토해줘" 프롬프트 하나 | 역할 분리 없음, 자기 응답 자기 검토의 순환 |
| **멀티에이전트 예제** | risu_agents, Serial Gradation | 역할은 나누지만 결국 전체 재작성, protected segment 개념 없음, 실패 시 원문 보존 미흡 |

**이 플러그인의 본질적 차별점**은 "감독실(director's room)" 메타포다:

- **역할은 나누되, 펜은 하나**: 여러 전문가가 검토하지만, 최종 수정은 bounded patch. 전체를 다시 쓰지 않는다.
- **보호 구역 설정**: 이미지 태그, 시스템 태그, 코드블록은 성역이다. 건드리지 않는다.
- **신뢰도 기반 결정**: 각 에이전트가 "얼마나 확신하는지"를 함께 보고한다. 확신이 낮으면 원문 유지.
- **모드 사다리**: 비용/속도/스트리밍 여부에 따라 off → audit → light → full로 자연스럽게 오르내린다.

## 2. MDASH 스타일 멀티에이전트의 장점을 살리기 위한 기능

MDASH의 핵심은 **역할 분리 검토 → 구조화된 findings → bounded patch → verifier veto**다. 이 장점을 극대화하려면:

### 2.1 역할 간 교차 검증 (Cross-Examination Round)

- **목적**: 한 에이전트의 finding을 다른 에이전트가 2차 검증. 오탐을 줄이고 진짜 문제만 패치.
- **사용자 체감 효과**: "고쳤다가 더 이상해지는" 현상이 사라짐. 패치 품질 상승.
- **동작 흐름**:
  1. 모든 Reader가 1차 findings 제출
  2. confidence < 0.7인 finding은 다른 Reader에게 교차 확인 요청
  3. 2개 이상의 Reader가 동의한 finding만 패치 대상으로 승격
  4. 단독 finding은 trace에만 기록, 패치하지 않음
- **호출되는 에이전트**: 1차 전체 Reader + 교차 확인 대상 Reader (보통 1~2개)
- **필요한 LLM 호출 수**: +1~2 (기존 대비)
- **streaming 사용 시 동작**: 비스트리밍에서만 동작 (패치 모드 전용)
- **UI 설정**: "교차 검증 사용" 토글 (기본값: 켜짐)
- **구현 난이도**: 중간
- **MVP / 추후 버전 / 실험 기능**: 추후 버전 (light_patch 안정화 후)
- **기존 플러그인과의 차별점**: 기존 멀티에이전트는 각자 의견 내고 끝. 교차 검증은 없음.

### 2.2 역할별 신뢰도 가중치 (Role Confidence Weight)

- **목적**: 에이전트마다 "이 분야에 얼마나 강한지" 가중치를 두어 충돌 시 우선순위 결정.
- **사용자 체감 효과**: 세계관 충돌 vs 문체 취향이 부딪힐 때 세계관이 이긴다. 더 합리적인 결정.
- **동작 흐름**:
  1. 각 Reader가 finding과 함께 confidence(0.0~1.0) 반환
  2. finding의 우선순위 = confidence × role_weight
     - Secret Guard: weight 1.0 (절대적)
     - World Reader: weight 0.9
     - Character Reader: weight 0.85
     - Plot Reader: weight 0.7
     - Style Reader: weight 0.5
  3. 충돌 시 우선순위 높은 finding 채택
  4. 동률이면 원문 유지 (보수적)
- **호출되는 에이전트**: 충돌 발생 시 Moderator 1회
- **필요한 LLM 호출 수**: +0~1
- **streaming 사용 시 동작**: 해당 없음 (패치 모드 전용)
- **UI 설정**: 고급 설정에서 가중치 조정 가능
- **구현 난이도**: 낮음
- **MVP / 추후 버전 / 실험 기능**: MVP
- **기존 플러그인과의 차별점**: 기존은 충돌 해결 로직이 아예 없거나, 마지막 에이전트가 덮어씀.

## 3. 캐릭터 담당 AI — 무엇을 점검할까

### 3.1 Voice Fingerprint Match

- **목적**: 캐릭터의 말투, 어휘, 문장 길이, 감탄사 패턴이 최근 응답과 일관되는지 검사.
- **사용자 체감 효과**: "갑자기 캐릭터가 다른 사람처럼 말하는" 현상 방지.
- **동작 흐름**:
  1. 최근 N턴의 해당 캐릭터 대사에서 voice fingerprint 추출
     - 평균 문장 길이, 자주 쓰는 단어 top 20, 감탄사 목록, 존댓말/반말 비율, 이모지 사용 패턴
  2. 현재 응답의 캐릭터 대사와 fingerprint 비교
  3. 이탈도가 임계치 초과 시 finding + 대체 표현 제안
- **호출되는 에이전트**: Character Reader (voice 전문)
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only 모드에서 findings만 기록
- **UI 설정**: "말투 일관성 검사" 온오프, 민감도 슬라이더
- **구현 난이도**: 중간 (fingerprint 추출은 JS, 비교는 LLM)
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only의 Character Reader에 통합)
- **기존 플러그인과의 차별점**: 기존은 "캐릭터 같게 말해줘" 정도의 추상적 프롬프트. fingerprint는 정량적.

### 3.2 Knowledge Boundary Guard

- **목적**: 캐릭터가 알 수 없는 정보를 말했는지 검사. "그 캐릭터는 그걸 모르는데"를 잡아냄.
- **사용자 체감 효과**: 비밀 유지, POV 일관성. 추리/스파이/정치 RP에서 결정적.
- **동작 흐름**:
  1. Character Contract에서 캐릭터별 known_facts, unknown_facts, secrets 추출
  2. 응답에서 해당 캐릭터의 대사/행동이 unknown_facts나 secrets를 포함하는지 검사
  3. 위반 발견 시: 해당 문장을 finding으로 표시, "이 캐릭터는 이걸 몰라야 합니다" 사유
  4. 패치 제안: 정보를 제거하거나, "추측"이나 "우연"으로 프레이밍 변경
- **호출되는 에이전트**: Character Reader + Secret Guard (교차 확인)
- **필요한 LLM 호출 수**: 1~2
- **streaming 사용 시 동작**: check_only에서 findings 기록
- **UI 설정**: "지식 경계 검사" 온오프
- **구현 난이도**: 중간 (Contract 추출 품질에 의존)
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only)
- **기존 플러그인과의 차별점**: 기존에는 "비밀 유지"라는 추상적 지시만. known/unknown을 명시적 계약으로 관리.

### 3.3 Relationship Posture Consistency

- **목적**: 캐릭터 간 관계 역학이 갑자기 변하지 않았는지 검사.
- **사용자 체감 효과**: 적대 캐릭터가 갑자기 친근해지거나, 연인이 갑자기 냉랭해지는 부자연스러운 변화 방지.
- **동작 흐름**:
  1. 최근 N턴에서 캐릭터 쌍별 관계 posture 추출 (적대/우호/중립/복합, 신뢰도, 권력 균형)
  2. 현재 응답에서 관계 표현이 급격히 변했는지 검사
  3. 변화가 있으면: "정당한 서사적 이유가 있는가?" 확인
     - 갈등 해소, 배신, 고백 등 명시적 이벤트가 있으면 허용
     - 이유 없이 변했으면 finding
- **호출되는 에이전트**: Character Reader
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "관계 일관성 검사" 온오프
- **구현 난이도**: 중간
- **MVP / 추후 버전 / 실험 기능**: 추후 버전
- **기존 플러그인과의 차별점**: 관계 추적은 보통 장기 기억 시스템의 영역. 이 플러그인은 최근 N턴만으로 단기 관계 일관성 검사.

## 4. 세계관 담당 AI — 무엇을 점검할까

### 4.1 World Rule Compliance

- **목적**: 설정된 세계관 규칙(마법 체계, 기술 수준, 사회 규범, 물리 법칙) 위반 검사.
- **사용자 체감 효과**: "이 세계에선 그게 불가능한데"라는 위화감 제거.
- **동작 흐름**:
  1. 캐릭터 카드/로어북/최근 채팅에서 world rules 추출
     - 예: "이 세계의 마법은 대가를 치러야 한다", "화약은 발명되지 않았다", "귀족은 평민에게 말을 걸지 않는다"
  2. 응답에서 각 rule 위반 여부 검사
  3. 위반 발견 시: 어떤 rule을 위반했는지 명시, 대체 행동 제안
- **호출되는 에이전트**: World Reader
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "세계관 규칙 검사" 온오프, rule 커스텀 입력 필드
- **구현 난이도**: 중간 (rule 추출 품질이 관건)
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only)
- **기존 플러그인과의 차별점**: 기존은 "세계관에 맞게"라는 추상적 지시. rule을 명시적 목록으로 추출하여 하나씩 대조.

### 4.2 Environmental Consistency

- **목적**: 장면의 물리적 환경(시간, 날씨, 장소, 주변 물체)이 응답 중에 모순되지 않았는지 검사.
- **사용자 체감 효과**: "아까는 비 왔는데 지금은 맑음", "방에 없던 의자가 갑자기 생김" 같은 미세한 위화감 제거.
- **동작 흐름**:
  1. 직전 응답에서 환경 요소 추출 (시간대, 날씨, 장소, 주요 오브젝트, 인원)
  2. 현재 응답에서 환경 요소가 모순 없이 이어지는지 검사
  3. 모순 발견 시 finding
- **호출되는 에이전트**: World Reader (환경 전문)
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "환경 일관성 검사" 온오프
- **구현 난이도**: 낮음
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only)
- **기존 플러그인과의 차별점**: 대부분의 플러그인은 환경 일관성을 아예 검사하지 않음.

## 5. 플롯/서사 담당 AI — 무엇을 점검할까

### 5.1 Scene Goal Tracking

- **목적**: 현재 장면의 목적이 응답에서 진행되었는지, 방치되었는지, 너무 빨리 해결되었는지 검사.
- **사용자 체감 효과**: "할 말이 있었는데 대화가 엉뚱한 곳으로 샜다" 방지. 장면에 방향성 부여.
- **동작 흐름**:
  1. 직전 사용자 입력과 컨텍스트에서 scene goal 추출
     - 예: "사과하기", "정보 캐내기", "탈출로 찾기", "감정 표현하기"
  2. 응답에서 goal이 진행되었는지 평가
     - progressed: 목표를 향해 한 걸음 나아감
     - stalled: 아무 진전 없음
     - resolved: 너무 쉽게/빠르게 해결됨
     - diverted: 엉뚱한 주제로 샘
  3. stalled/diverted 시 finding + "이 goal을 진행하려면 어떻게 해야 하는지" 힌트
- **호출되는 에이전트**: Plot Reader
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "장면 목표 추적" 온오프
- **구현 난이도**: 중간
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only)
- **기존 플러그인과의 차별점**: 기존은 "플롯 일관성"을 추상적으로만 언급. goal tracking은 구체적.

### 5.2 Tension Budget

- **목적**: 응답이 긴장감을 적절히 소비하고 생성했는지 평가. 장면이 "식지" 않도록.
- **사용자 체감 효과**: 응답 후에도 "다음이 궁금하다"는 느낌 유지. 늘어지는 장면 방지.
- **동작 흐름**:
  1. 응답 전 tension 상태 추정 (갈등 미해결, 위험, 시간 압박, 감정 고조)
  2. 응답에서 tension 변화 측정
     - tension_spent: 해소된 긴장 (갈등 해결, 위험 제거, 감정 진정)
     - tension_created: 새로 생긴 긴장 (새 갈등, 복선, 의문, 위험 증가)
  3. tension_spent > tension_created 이고 장면이 끝나지 않았으면 finding
     - "장면의 긴장이 너무 빨리 해소되었습니다. 새 갈등이나 미해결 요소를 남겨주세요."
- **호출되는 에이전트**: Plot Reader
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "긴장감 추적" 온오프
- **구현 난이도**: 중간
- **MVP / 추후 버전 / 실험 기능**: 추후 버전
- **기존 플러그인과의 차별점**: 긴장감을 정량적 예산으로 다루는 접근은 기존에 없음.

### 5.3 Foreshadowing Echo

- **목적**: 이전 응답에서 뿌려진 복선이 현재 응답에서 자연스럽게 울리는지 감지.
- **사용자 체감 효과**: "아까 그 말이 여기서 이렇게 이어지네"라는 서사적 만족감. 치밀한 이야기.
- **동작 흐름**:
  1. 최근 N턴에서 "나중에 중요해질 수 있는" 요소 추출 (의문, 약속, 위협, 미완료 행동)
  2. 현재 응답에서 그 요소가 회수되거나 진전되었는지 검사
  3. 회수 기회가 있었는데 놓쳤으면 finding + "여기서 이 복선을 살릴 수 있습니다" 제안
- **호출되는 에이전트**: Plot Reader + Continuity Reader
- **필요한 LLM 호출 수**: 1~2
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "복선 추적" 온오프
- **구현 난이도**: 높음 (복선 인식이 어려움)
- **MVP / 추후 버전 / 실험 기능**: 추후 버전
- **기존 플러그인과의 차별점**: 복선 추적은 보통 장기 기억 시스템의 영역. 단기 N턴만으로도 가능한 복선 회수를 감지.

## 6. 문체/문장 담당 AI — 무엇을 점검할까

### 6.1 Prose Rhythm Analysis

- **목적**: 문장 길이, 단락 구조, 능동/수동 태의 리듬이 장면 분위기와 일치하는지 검사.
- **사용자 체감 효과**: 액션 신은 짧고 빠르게, 감정 신은 길고 깊게. 읽는 리듬이 내용과 일치.
- **동작 흐름**:
  1. 응답의 장면 유형 분류 (action, dialogue, introspection, description, transition)
  2. 장면 유형별 기대 리듬과 실제 리듬 비교
     - action: 평균 문장 길이 짧음, 단락 짧음, 능동태 많음
     - introspection: 평균 문장 길이 김, 단락 김, 수동태/비유 많음
  3. 불일치 시 finding + 리듬 조정 제안
- **호출되는 에이전트**: Style Reader
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "문장 리듬 검사" 온오프
- **구현 난이도**: 낮음 (JS로 문장 길이/단락 구조 분석 가능, LLM은 장면 분류만)
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only)
- **기존 플러그인과의 차별점**: 기존 문법 검사기는 오류만 찾음. 리듬은 "더 좋은 글"을 위한 제안.

### 6.2 Dialogue-to-Narration Ratio

- **목적**: 대사와 지문의 비율이 장면 유형에 적절한지 검사.
- **사용자 체감 효과**: 대화 장면에서 지문이 너무 많으면 늘어지고, 액션 장면에서 대사만 있으면 허전함. 적절한 균형.
- **동작 흐름**:
  1. 응답에서 대사 비율 계산 (따옴표/대사 마커 기준, JS)
  2. 장면 유형별 적정 비율과 비교
     - 대화 장면: 대사 50~70%
     - 액션 장면: 대사 10~30%
     - 내면 독백: 대사 0~10%
     - 전환/설명: 대사 0~20%
  3. 비율이 크게 벗어나면 finding
- **호출되는 에이전트**: Style Reader
- **필요한 LLM 호출 수**: 0~1 (비율 계산은 JS, 장면 분류만 LLM)
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "대사-지문 균형" 온오프, 장면별 목표 비율 커스텀
- **구현 난이도**: 낮음
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only)
- **기존 플러그인과의 차별점**: 단순 비율 계산이지만, 장면 유형과 연동하여 의미 있는 피드백.

### 6.3 Repetition Detector

- **목적**: 같은 단어, 구문, 문장 구조가 짧은 구간 내에 반복되는지 검사.
- **사용자 체감 효과**: "그녀는 미소지었다. 그는 미소지었다. 그녀도 미소지었다." 같은 단조로운 반복 제거.
- **동작 흐름**:
  1. JS로 응답 내 단어/구문 빈도 분석
  2. 짧은 구간(3문장 이내)에서 동일 어휘, 동일 문장 구조 반복 감지
  3. 임계치 초과 시 finding + 대체 표현 제안 (LLM)
- **호출되는 에이전트**: Style Reader
- **필요한 LLM 호출 수**: 0~1 (감지는 JS, 대체 표현 제안만 LLM)
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "반복 표현 감지" 온오프, 민감도
- **구현 난이도**: 낮음
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only)
- **기존 플러그인과의 차별점**: 기존 문법 검사기는 철자/문법만 봄. 어휘 반복은 보지 않음.

## 7. 연출/감정선 담당 AI — 무엇을 점검할까

### 7.1 Emotional Arc Validation

- **목적**: 응답 내에서 감정의 흐름이 자연스러운지 검사. 급격한 감정 변화는 이유가 있어야 함.
- **사용자 체감 효과**: "방금까지 화냈는데 갑자기 웃네?" 같은 부자연스러운 감정 점프 방지.
- **동작 흐름**:
  1. 응답을 시간순 세그먼트로 분할 (시작 → 중간 → 끝)
  2. 각 세그먼트의 주요 감정 톤 추출 (분노, 슬픔, 기쁨, 두려움, 놀람, 혐오, 중립)
  3. 감정 변화 그래프가 자연스러운지 평가
     - 자연: 분노→슬픔→체념 (단계적)
     - 부자연: 분노→기쁨 (이유 없는 급변)
  4. 부자연스러운 변화에 finding + "중간 감정 단계를 추가하거나 이유를 암시하세요"
- **호출되는 에이전트**: Performance Reader (감정 전문)
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "감정 흐름 검사" 온오프
- **구현 난이도**: 중간
- **MVP / 추후 버전 / 실험 기능**: MVP (check_only)
- **기존 플러그인과의 차별점**: 감정을 정적 상태가 아닌 동적 흐름(arc)으로 분석.

### 7.2 Show-vs-Tell Balance

- **목적**: "그는 슬펐다"라고 말하는(tell) 대신 "그의 어깨가 축 처졌다"라고 보여주는(show) 비율 검사.
- **사용자 체감 효과**: 더 몰입감 있는 소설적 연출. 독자가 감정을 추론하게 함.
- **동작 흐름**:
  1. 응답에서 tell 패턴 감지 (JS + LLM)
     - tell: "그는 ~했다" + 감정 형용사 직결 ("그는 화가 났다", "그녀는 기뻤다")
     - show: 행동/생리적 묘사로 감정 암시 ("주먹을 꽉 쥐었다", "입꼬리가 올라갔다")
  2. tell 비율이 임계치 초과 시 finding
  3. 핵심 감정 순간에 show로 변환 제안
- **호출되는 에이전트**: Performance Reader
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "Show-don't-tell 검사" 온오프, 강도 슬라이더
- **구현 난이도**: 중간
- **MVP / 추후 버전 / 실험 기능**: 추후 버전
- **기존 플러그인과의 차별점**: 문법 검사기가 아닌 "소설적 연출"을 평가하는 메타 인식.

### 7.3 Beat Timing

- **목적**: 응답 내 "비트"(행동, 대사, 침묵, 반응)의 타이밍이 자연스러운지 검사.
- **사용자 체감 효과**: 대화의 호흡이 자연스러움. 대답이 너무 빠르거나, 반응이 너무 느리지 않음.
- **동작 흐름**:
  1. 응답을 beat 단위로 분할 (캐릭터 A 행동 → 캐릭터 B 반응 → 대사 → 침묵 → 새 행동)
  2. beat 간 전환이 자연스러운지 평가
     - 질문 직후 즉시 긴 대답 → 부자연 (생각할 시간 필요)
     - 충격적 소식 직후 바로 농담 → 부자연 (감정 처리 시간 필요)
  3. 부자연 beat 전환에 finding + "여기에 짧은 침묵/반응 beat 추가"
- **호출되는 에이전트**: Performance Reader
- **필요한 LLM 호출 수**: 1
- **streaming 사용 시 동작**: check_only
- **UI 설정**: "비트 타이밍 검사" 온오프
- **구현 난이도**: 높음 (beat 분할이 주관적)
- **MVP / 추후 버전 / 실험 기능**: 추후 버전
- **기존 플러그인과의 차별점**: 연출의 "타이밍"을 분석하는 플러그인은 기존에 없음.

## 8. 충돌 해결 — 최종 결정자 설계

### Moderator with Priority Hierarchy

- **목적**: 여러 에이전트의 finding이 충돌할 때 최종 결정.
- **사용자 체감 효과**: "고치라고 했다가 말았다가" 하지 않음. 일관된 결정.
- **동작 흐름**:
  1. 모든 finding을 priority × confidence로 정렬
  2. 동일 segment에 대한 충돌 finding 발견 시:
     a. Secret Guard finding은 항상 최우선 (정보 누설은 무조건 수정)
     b. World > Character > Plot > Performance > Style 순
     c. 동일 priority면 confidence 높은 쪽
     d. confidence도 동일하면 원문 유지 (보수적)
  3. Moderator(경량 LLM)가 최종 판정: "A를 고치면 B가 깨진다" 같은 trade-off 평가
  4. Moderator도 확신이 없으면 원문 유지
- **호출되는 에이전트**: Moderator (경량/저비용 모델 권장)
- **필요한 LLM 호출 수**: 0~1 (충돌 시에만)
- **streaming 사용 시 동작**: 해당 없음 (패치 모드 전용)
- **UI 설정**: "충돌 해결 방식" 드롭다운 (우선순위 기반 / Moderator 판정 / 보수적 원문 유지)
- **구현 난이도**: 중간
- **MVP / 추후 버전 / 실험 기능**: MVP (우선순위 기반은 JS로 구현, Moderator는 추후)
- **기존 플러그인과의 차별점**: 기존은 충돌 해결 메커니즘 자체가 없음.

## 9. 선택적 에이전트 호출 — 비용/속도 최적화

### 9.1 Response Type Classifier

- **목적**: 응답 유형에 따라 필요한 Reader만 호출하여 불필요한 LLM 호출 제거.
- **사용자 체감 효과**: 빠른 응답. 액션 신에서 문체 검사로 시간 낭비하지 않음.
- **동작 흐름**:
  1. JS로 응답 특성 빠른 분석 (대사 비율, 문장 수, 길이)
  2. 응답 유형 분류:
     - action_scene: World + Plot + Performance (Character/Style 생략)
     - dialogue_heavy: Character + Style + Performance (World/Plot 생략)
     - introspection: Character + Style (World/Plot/Performance 생략)
     - exposition: World + Plot + Style (Character/Performance 생략)
     - mixed: 전체 호출
  3. 분류 결과를 trace에 기록
- **호출되는 에이전트**: Context Reader (분류만, 1회)
- **필요한 LLM 호출 수**: 1 (분류용) → 이후 Reader 호출 2~4개로 감소
- **streaming 사용 시 동작**: check_only에서도 분류 적용
- **UI 설정**: "선택적 에이전트 호출" 온오프, 응답 유형별 Reader 커스텀
- **구현 난이도**: 낮음
- **MVP / 추후 버전 / 실험 기능**: MVP
- **기존 플러그인과의 차별점**: 기존 멀티에이전트는 항상 전원 호출. 선택적 호출은 비용 절감.

### 9.2 Quick Scan vs Deep Read

- **목적**: 1차로 경량 모델이 빠르게 스캔하고, 문제 가능성이 있을 때만 본진 투입.
- **사용자 체감 효과**: 대부분의 "괜찮은" 응답은 빠르게 통과. 문제 있을 때만 시간 사용.
- **동작 흐름**:
  1. Quick Scan (경량 모델, 1회 호출): 응답 전체를 빠르게 훑으며 "문제 가능성" 점수 산정
  2. 문제 가능성 < 임계치: 통과 (원문 그대로)
  3. 문제 가능성 ≥ 임계치: Deep Read 모드로 전환, 전체 Reader 호출
- **호출되는 에이전트**: Quick Scanner (경량) → 필요 시 전체 Reader
- **필요한 LLM 호출 수**: 1 (대부분 통과) ~ 1+N (문제 시)
- **streaming 사용 시 동작**: Quick Scan만 수행, findings 기록
- **UI 설정**: "빠른 스캔 사용" 온오프, 임계치 슬라이더
- **구현 난이도**: 중간
- **MVP / 추후 버전 / 실험 기능**: 추후 버전
- **기존 플러그인과의 차별점**: 2단계 스캔은 비용 효율적. 기존은 항상 풀파워.

## 12. Killer Feature 제안

### 12.1 Before/After Diff with Reasoning

- **목적**: 패치된 내용을 원문과 나란히 보여주고, 각 변경의 이유를 표시.
- **사용자 체감 효과**: "뭐가 바뀌었는지 모르겠다"는 불신 해소. AI가 투명하게 설명.
- **동작 흐름**:
  1. 패치 적용 후 원문과 최종본 diff 생성
  2. 각 변경에 대해 어떤 Reader가 어떤 이유로 제안했는지 주석
  3. UI에 diff view 표시 (추가/삭제/변경 하이라이트)
  4. 사용자가 개별 변경을 수락/거절 가능 (선택적)
- **호출되는 에이전트**: 없음 (JS로 diff 생성)
- **필요한 LLM 호출 수**: 0
- **streaming 사용 시 동작**: audit 모드에서 findings만 diff 형태로 표시
- **UI 설정**: "변경 사항 설명 보기" 온오프
- **구현 난이도**: 중간 (diff 알고리즘 + UI)
- **MVP / 추후 버전 / 실험 기능**: MVP (Audit Only에서 findings diff로 시작, 패치 모드에서 full diff)
- **기존 플러그인과의 차별점**: **이게 Killer Feature다.** 기존 어떤 플러그인도 "무엇을 왜 바꿨는지" 투명하게 보여주지 않음. 사용자 신뢰의 핵심.

### 12.2 Director's Commentary

- **목적**: Audit Only 모드에서 findings를 "감독 코멘터리"처럼 자연어로 제공.
- **사용자 체감 효과**: 마치 영화 DVD의 감독 코멘터리처럼 "이 장면에서 캐릭터의 말투가 살짝 달라졌네요. 아마 방금 전 충격적인 사건 때문일 수 있어요." 같은 인사이트.
- **동작 흐름**:
  1. 모든 Reader findings를 수집
  2. Moderator가 findings를 자연스러운 코멘터리로 변환
  3. 사이드 패널에 "🎬 Director's Commentary"로 표시
  4. 긍정적 발견도 포함 ("이 대사는 캐릭터 목소리가 완벽하게 살아있네요")
- **호출되는 에이전트**: Moderator (1회)
- **필요한 LLM 호출 수**: +1
- **streaming 사용 시 동작**: audit 모드에서 코멘터리만 표시
- **UI 설정**: "감독 코멘터리" 온오프
- **구현 난이도**: 낮음
- **MVP / 추후 버전 / 실험 기능**: MVP (Audit Only 모드의 기본 표시 방식으로)
- **기존 플러그인과의 차별점**: findings를 딱딱한 JSON이 아닌 읽는 재미가 있는 코멘터리로. **사용자가 findings를 읽게 만드는 UX.**

### 12.3 "What Would ___ Say?" — 캐릭터별 리뷰 보기

- **목적**: 사용자가 특정 캐릭터의 Character Reader가 무슨 findings를 냈는지 따로 볼 수 있음.
- **사용자 체감 효과**: "내 캐릭터에 대해 AI가 뭐라고 했지?" 궁금증 해소. 캐릭터 애착을 플러그인 engagement로 전환.
- **동작 흐름**:
  1. Character Reader가 캐릭터별로 findings 생성
  2. UI에서 캐릭터 이름 옆에 알림 배지 ("리아: 2건")
  3. 클릭 시 "리아의 관점에서 본 이 응답" 표시
     - "리아의 말투가 평소보다 딱딱해요. 아마 긴장한 상황이라 그럴 수 있어요."
     - "리아는 이 정보를 몰라야 하는데, 대사에서 암시하고 있어요."
- **호출되는 에이전트**: Character Reader (이미 호출됨, 추가 호출 없음)
- **필요한 LLM 호출 수**: 0 (기존 findings 재가공)
- **streaming 사용 시 동작**: audit 모드에서 캐릭터별 findings 표시
- **UI 설정**: "캐릭터별 리뷰" 온오프
- **구현 난이도**: 낮음 (UI 작업)
- **MVP / 추후 버전 / 실험 기능**: MVP
- **기존 플러그인과의 차별점**: **사용자의 캐릭터 애착을 플러그인 engagement로 전환.** "내 최애 캐릭터가 뭐라고 했을까?"라는 심리를 이용.

## 최종 요약: 이 플러그인의 정체성

이 플러그인은 **"RP 감독실"** 이다.

- 문법 검사기가 아니다. **연출을 평가한다.**
- 전체 재작성기가 아니다. **문제 구간만 지적하고, 고칠지는 사용자가 결정한다.**
- 단일 에이전트가 아니다. **캐릭터, 세계관, 플롯, 문체, 감정 전문가가 각자 자기 분야만 본다.**
- 불투명하지 않다. **무엇을 왜 바꿨는지 diff와 코멘터리로 설명한다.**

**Killer Feature 3종:**
1. **Before/After Diff with Reasoning** — 신뢰
2. **Director's Commentary** — 읽는 재미
3. **"What Would ___ Say?"** — 캐릭터 애착 연동
