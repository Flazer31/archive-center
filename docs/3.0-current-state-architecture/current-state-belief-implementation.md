# 3.0 현재 상태·인물 인식·과거 기억 분리

## 기록 목적

이 작업은 2.5.0 안정판 이후의 3.0 후보 작업이다. 실사용 중 문제가 발견되어 소스를 2.5.0으로 되돌리더라도 무엇을 구현했고 어떤 위험을 막으려 했는지 다시 확인할 수 있도록 남긴다.

구현 시점 상태:

- 작업 트리 구현 완료
- Go 전체 내부 테스트 통과
- Go 백엔드 빌드 통과
- JavaScript 문법 검사 통과
- 실제 장기 세션 및 배포 환경 검증은 아직 필요

## 해결하려는 문제

기존 주입은 대체로 다음 순서였다.

`과거 기억 → 관계 지식 → 직접 근거 → 기타 상태 자료`

이 구조에서는 다음 혼란이 생길 수 있었다.

- 과거의 고중요도 기억이 현재 상태보다 먼저 모델의 주의를 끈다.
- 객관적인 현재 사실과 특정 인물의 믿음·오해가 구분되지 않는다.
- 현재 장면에 없는 인물의 사적 기억이 중요도만으로 난입한다.
- 사망→생존, 불신→신뢰 회복처럼 상태가 바뀌어도 이전값과 현재값이 함께 살아 있는 것처럼 보인다.
- 턴 삭제 후 삭제된 턴이 만든 현재값만 사라지고 직전 상태가 복원되지 않을 수 있다.

## 3.0 계약

평론가는 완료된 턴을 세 종류로 분리한다.

1. `narrative_events`: 이번 턴에 일어난 사건. 이력이며 현재값을 직접 덮어쓰지 않는다.
2. `state_claims`: 현재의 객관적 사실.
3. `belief_updates`: 특정 인물이 현재 믿거나 오해하거나 의심하는 내용.

각 항목은 최신 완료 턴에서 가져온 짧은 정확한 `evidence_excerpt`를 가져야 한다. 근거가 없으면 저장하지 않는다.

현재값 식별 기준:

`대상 종류 + 대상 + 상태 영역 + 인식 범위 + 인식 주체`

턴 번호는 의미상 진실을 결정하지 않는다. 저장 순서, 재처리, 삭제, 롤백을 위한 감사 순서로만 사용한다.

## 저장 동작

- 기존 `status_schema_registry`, `status_current_values`, `status_change_events`를 재사용한다.
- 같은 상태 식별 기준에는 현재값 하나만 유지한다.
- 값이 바뀌면 현재값을 교체하고 변경 이력에 이전값과 새값을 함께 기록한다.
- 상태 근거 발췌를 `direct_evidence_records`에도 포함하고 상태 변경에서 해당 직접 근거 ID를 참조한다.
- 더 오래된 턴을 재처리해도 더 새로운 현재값을 덮어쓰지 않는다.
- 완전히 같은 현재값과 동일 사건 재처리만 건너뛴다.
- 의미가 비슷하다는 이유로 기존 기억이나 상태를 자동 삭제하지 않는다.

## 주입 동작

2.5의 기존 기억 주입 순서를 유지한다. 현재 사실과 인물별 인식을 상시 앞에 붙이지 않는다.

`2.5 기억 주입 → ChromaDB 검색 결과와 현재값 대조 → 필요한 경우에만 [Continuity Correction]을 마지막에 추가`

- ChromaDB는 관련 기억 후보를 찾는다.
- MariaDB 현재값은 검색된 후보가 현재도 유효한지 판정하는 기준으로 사용한다.
- 최근 RisuAI 대화에 현재값이 이미 명확하면 보정문을 추가하지 않는다.
- 검색 기억이나 최근 대화가 종료된 이전값을 현재 사실처럼 주장할 때만 현재값을 보정한다.
- 현재 입력에서 직접 다루는 변경 상태가 검색 결과와 최근 대화 양쪽에서 빠진 경우에만 누락 보정을 허용한다.
- 인물별 믿음은 해당 인물이 현재 입력, 직전 대화, 최신 장면에 있을 때만 보정할 수 있다.

`[Continuity Correction]`은 새 사건을 만들거나 전개를 강제하는 자료가 아니다. 오류, 누락, 충돌을 막기 위한 짧은 후첨 보정문이며 전체 주입문의 마지막에 놓인다.

현재 상태가 이전값을 명시적으로 교체했다면, 해당 이전값을 그대로 주장하는 과거 기억은 중요도가 높아도 이번 주입에서 제외한다. DB 행 자체는 삭제하지 않는다.

현재 입력이 있는 일반 턴에서는 관련성이 없는 오래된 기억을 중요도만으로 `deep` 기억에 올리지 않는다. 관련성이 없는 보조 기억이 필요하면 오래된 중요도보다 최신 순서를 우선한다.

종료된 `KG valid_to` 범위와 아직 시작되지 않은 관계 지식은 현재 턴 주입에서 제외한다.

## 장면 밖 인물 보호

인물별 인식과 사적 기억은 소유자가 다음 중 하나에 있을 때만 주입한다.

- 현재 사용자 입력에 직접 등장
- 직전 대화에 등장
- 최신 장면 상태에 등장

현재 입력이나 직전 장면에서 제3자를 직접 언급했다면 관련 인물로 인정할 수 있다. 따라서 제3자에 대한 험담, 추적, 호출처럼 실제로 필요한 장면 전개까지 일률적으로 막지 않는다.

## 삭제와 롤백

삭제된 턴이 현재값을 만들었다면 남아 있는 변경 이력의 최신값으로 현재 상태를 다시 만든다. 삭제된 미래 상태를 남기지 않으며, 직전 상태 전체를 잃지도 않도록 한다.

## 변경 파일

새 파일:

- `go-service/internal/httpapi/narrative_state_contract.go`
- `go-service/internal/httpapi/narrative_state_contract_test.go`

연결 파일:

- `go-service/internal/httpapi/turn_extraction.go`
- `go-service/internal/httpapi/group_turn.go`
- `go-service/internal/httpapi/group_turn_test.go`
- `go-service/internal/httpapi/group_status_schema.go`
- `prompts/critic_system.txt`

`Archive Center.js`는 백엔드가 보정 필요성을 확정한 경우에만 `continuity_correction_text`를 읽고, 기존 2.5 주입문 뒤에 덧붙인다. JS는 상태 충돌을 자체 판정하지 않는다.

## 자동 검증 결과

완료된 검증:

- `go test ./internal/... -count=1`
- `go build ./cmd/archive-center-go`
- `node --check "Archive Center.js"`

검증한 주요 사례:

- 객관적 사실과 인물 믿음 분리
- 현재값 한 개 유지
- 이전값과 새값 변경 이력 보존
- 직접 근거 ID 연결
- 장면 밖 인물 인식 제외
- 종료된 이전값을 주장하는 과거 기억 제외
- 오래된 고중요도만으로 현재 장면 앞에 배치되지 않음
- 턴 삭제 후 남은 최신 상태 복원
- 동일 현재값과 동일 사건 재처리 중복 방지

## 실사용 검증 체크리스트

3.0 채택 전에 다음을 확인한다.

1. 30턴 이상 세션에서 현재 인물만 사적 기억에 등장하는지 확인한다.
2. 사망 오인→생존 확인 같은 반전 후 과거 사망 기억이 현재 사실을 덮지 않는지 확인한다.
3. 불신→신뢰 회복→재갈등처럼 같은 상태 영역이 여러 번 바뀌어도 최신값과 이력이 맞는지 확인한다.
4. 제3자가 현재 입력에서 언급된 경우 필요한 관련 기억은 차단되지 않는지 확인한다.
5. 최근 턴 삭제 후 직전 현재값이 복원되는지 확인한다.
6. 복사·이동·연결 세션에서 기존 2.5의 이어가기 기준과 충돌하지 않는지 확인한다.
7. 기존 세션을 재스캔한 뒤 상태값이 과도하게 생성되지 않는지 확인한다.
8. ChromaDB가 반환한 과거 고중요도 기억도 현재 상태 필터를 통과한 뒤에만 주입되는지 확인한다.

## 2.5.0으로 되돌릴 때

문제가 생기면 2.5.0 실행 파일과 프롬프트로 되돌릴 수 있다. 기존 기억·직접 근거·KG를 삭제할 필요는 없다.

3.0 후보가 추가한 범용 상태 행은 2.5.0에서 읽지 않으면 단순히 사용되지 않는다. 임의로 DB 전체를 초기화하지 않는다.

재적용할 때에는 다음 순서를 지킨다.

1. 평론가 JSON 계약 복원
2. 상태 정규화 및 현재값 저장기 복원
3. 직접 근거 연결 복원
4. 현재 사실·인물 인식 분리 주입 복원
5. 이전값 충돌 기억 필터와 장면 관련성 필터 복원
6. 롤백 후 현재값 복원 로직 복원
7. 자동 테스트 후 실사용 검증

## 기존 세션 적용 범위

업데이트 이후 진행되는 새 턴부터 새 상태 계약이 생성된다. 이전 턴 전체에 적용하려면 업데이트된 평론가를 사용하는 재스캔이 필요하다.

기존 데이터를 의미 유사도만으로 자동 정리하거나 삭제하지 않는다.
