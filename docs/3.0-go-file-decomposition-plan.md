# Archive Center 3.0 Go 대형 파일 분리 계획

작성일: 2026-07-11

## 목적

현재 동작을 바꾸지 않고 비대한 Go 파일을 같은 패키지 안의 책임별 파일로 나눈다.

이번 단계에서 하지 않는 것:

- HTTP 경로, 요청·응답 JSON, SQL, DB 스키마 변경
- 함수명·시그니처 변경
- 새 인터페이스나 공통 추상화 도입
- `httpapi`와 `store`를 새 하위 패키지로 이동
- JS 로직 이전이나 기능 개선을 파일 이동과 동시에 수행
- 테스트 기대값 변경

분리 후에도 패키지는 그대로 유지한다.

- `group_turn.go`, `group_narrative.go`, `turn_extraction.go` → `package httpapi`
- `mariadb.go` → `package store`

같은 패키지 안의 파일 이동은 기존 비공개 함수와 타입을 그대로 공유할 수 있으므로 첫 단계에서 순환 의존 위험이 가장 낮다.

### `group_turn.go` 1차 목표

- 먼저 SEQ-16.5~18 builder를 옮겨 런타임 핵심과 과거 검증 표면을 분리한다.
- 그 다음 rollback, render, vector, memory처럼 경계가 분명한 묶음을 옮긴다.
- 마지막에 `handlePrepareTurn`과 `handleCompleteTurnDecoded`를 별도 파일로 옮긴다.
- 두 대형 핸들러의 함수 내부 분해는 별도 작업으로 남긴다. 파일 이동과 로직 리팩터링을 동시에 하지 않는다.

## 테스트 분리

생산 코드 이동과 같은 커밋에서 테스트 내용을 고치지 않는다. 전체 생산 파일 이동이 끝난 뒤 테스트 파일만 다음처럼 옮긴다.

- `group_turn_test.go` → `group_turn_complete_test.go`, `group_turn_prepare_test.go`, `group_turn_rollback_test.go`, `prepare_turn_memory_test.go`, `prepare_turn_render_test.go`
- `group_narrative_test.go` → sessions/storylines/characters/world-rules/episodes/metrics 기준 파일
- 기존 SEQ 테스트 파일은 그대로 유지
- `turn_extraction_test.go`는 현재 289줄이므로 당장은 유지
- MariaDB 테스트는 이미 기능별 파일이 있으므로 새 테스트를 기존 도메인 파일명에 맞춰 추가

테스트 이동 시에도 assertion, fixture, 테스트 이름은 변경하지 않는다.

## 권장 실행 순서

1. 현재 1~5 안정화 변경을 GitHub push 없이 로컬 체크포인트로 고정
2. `group_turn.go`의 SEQ-16.5~18 builder만 기계적으로 이동
3. 전체 `gofmt`, `go vet ./...`, `go test ./... -count=1`
4. `group_narrative.go`를 라우트 도메인별로 이동 후 같은 검증
5. `mariadb.go`를 receiver 도메인별로 이동 후 MariaDB·rollback·migration 테스트 집중 실행
6. `turn_extraction.go`를 policy/config/critic/persist/vector 경계로 이동 후 같은 검증
7. `group_turn.go`의 runtime prepare/complete/rollback 묶음 이동
8. 대형 테스트 파일 이동
9. 파일별 줄 수와 함수 수를 다시 측정하고 문서 갱신

## 각 이동 단위의 필수 검증

매 이동마다 다음을 모두 통과해야 다음 묶음으로 진행한다.

```text
gofmt -l <이동한 Go 파일>
go vet ./...
go test ./... -count=1
```

추가 회귀 확인:

- `POST /prepare-turn` 응답 JSON 동일
- `POST /complete-turn` raw/derived/vector lane 동일
- 중복 pair와 idempotency 동작 동일
- `DELETE /rollback/{turn}` 범위 동일
- copy/move/lock/rollback SQL 결과 동일
- critic/embedding 설정 선택 source 동일
- Chroma upsert/search/delete 통합 테스트 동일

## 중단 조건

다음 중 하나라도 발생하면 다음 파일로 넘어가지 않는다.

- route 또는 JSON 필드 변화
- SQL 문자열·트랜잭션 범위 변화
- import cycle을 피하려고 새 인터페이스가 필요한 상황
- 기존 테스트 기대값을 바꿔야만 통과하는 상황
- 파일 이동과 무관한 기능 수정 필요 발견

이 경우 기능 수정은 별도 작업으로 기록하고, 파일 분리 diff에는 포함하지 않는다.

## 후속 작업: 백그라운드 성능 개선

이번 파일 분리는 실행 경로를 변경하지 않았으므로 런타임 속도는 기존과 동일하다. 분리된 경계를 이용한 성능 개선은 별도 작업으로 진행한다.

- prepare-turn, DB 조회, vector 검색, 감독관·평론가 호출, raw/derived/vector 저장 시간을 단계별로 계측
- 메인 출력과 raw 저장을 우선하고, 독립적인 파생 생성만 안전하게 백그라운드 처리
- 세션별 턴 순서, 멱등성, 삭제·리롤과의 충돌 방지 계약 유지
- timeout 단축이 아니라 병목 확인 후 query 축소, batch, 제한된 병렬 처리 적용
- 최적화 전후 저장 결과와 기억 주입 결과의 완전한 회귀 비교
