# Archive Center 3.0 구현 작업 원장

기준일: 2026-07-13  
기준 범위: `release-2.5.0-rc4` 기준 커밋 `26818fe` 이후  
현재 로컬 체크포인트: `82bfe09`

## 문서 목적

이 문서는 2.5.0 RC4 이후 3.0을 위해 실제로 수행한 작업을 하나의 원장으로
보존한다. 기능별 상세 설계는 기존 `docs/3.0-*.md` 문서를 따르되, 현재
구현 여부와 되돌린 시도는 이 문서를 우선 기준으로 삼는다.

상태 의미:

- **완료**: 코드와 자동 회귀 검증이 끝난 활성 구현
- **완료·실사용 확인 필요**: 자동 검증은 끝났지만 장기 세션이나 실제 RisuAI 환경 확인이 남음
- **되돌림**: 시도했으나 위험 또는 회귀 때문에 활성 코드에서 제거
- **미완료**: 설계 또는 기반만 있고 최종 기능은 아직 열지 않음

  - `314e348 chore: checkpoint before 3.0 stabilization work`
  - `7a58b3d chore: stabilize runtime before Go file split`

- 번들 Node.js를 자동 탐색해 JavaScript 런타임 검사를 조용히 건너뛰지 않게 했다.
- `node --check "Archive Center.js"`, `scripts/test-core-regressions.ps1`, `go test ./...`를 현재 변경의 기본 검증으로 사용한다.

## 2. Go 백엔드 구조 분리와 시간 계측

### 완료

- 비대해진 Go 파일을 같은 패키지 안에서 책임별 파일로 분리했다.

- prepare-turn과 complete-turn의 단계별 시간을 백엔드가 반환하고 대시보드가 표시한다.
- 확인된 일반 턴 예시에서는 준비 약 343ms, 저장·파생 생성 약 26.3초였고, 병목은 외부 평론가 LLM과 임베딩 호출이었다.

## 3. 요청 세션 고정과 JS/백엔드 중복 계산 제거

## 7. RisuAI 메시지 인덱스와 DB 턴의 직접 연결

### 완료

RisuAI active chat의 원본 배열 인덱스를 보존하고 다음 구조를 DB 논리 턴의
기준으로 사용한다.

| RisuAI 메시지 | DB 논리 턴 |
| --- | ---: |
| user `Index 0`, assistant `Index 1` | turn 1 |
| user `Index 2`, assistant `Index 3` | turn 2 |
| user `Index 4`, assistant `Index 5` | turn 3 |

적용 규칙:

- JS는 RisuAI의 원본 user/assistant 인덱스와 관찰 순서만 백엔드에 전달한다.
- 일반 세션의 `floor(user_index / 2) + 1` 계산과 최종 DB 턴 선택은 백엔드가 담당한다.
- 원본 인덱스를 사용할 수 없는 경우에만 백엔드가 관찰된 pair 순서를 대체 근거로 사용한다.
- 복사·이동·연결 세션은 backend baseline을 적용한다.
  - 기존 7턴을 빈 채팅에 가져온 뒤 로컬 turn 1은 DB turn 8이다.
  - 그 첫 로컬 턴을 삭제하면 turn 8부터만 삭제하고 turn 1~7은 보호한다.
- 화면의 같은 논리 턴이 DB에 이미 있으나 내용이 바뀌었으면 새 번호로 추가하지 않는다.
  - 기존 backend rollback 결정으로 해당 턴 이후를 정리한다.
  - 최신 user/assistant pair를 같은 턴 번호로 다시 저장한다.
- active chat이 완전히 빈 배열이 되어도 이전 rollback ledger 전체 제거를 검증할 수 있다.
- 빈 배열이라는 이유만으로 삭제하지 않으며, 검증 ledger 또는 세션 baseline 보호 계약을 통과해야 한다.
- 캐릭터의 `firstMessage`, `alternateGreetings`, active chat의 `fmIndex`를 읽어 실제 선택된 시작 메시지를 DB `turn 0`으로 저장한다.

관련 체크포인트:

- `c3acd74 fix: reconcile deletion before immediate next turn`
- `82bfe09 fix: align persistence with Risu message indexes`

자동 검증:

- 원본 인덱스 `0/1, 2/3, 4/5`가 turn `1, 2, 3`이 되는지 검사
- JS가 계산된 `turnIndex`를 만들거나 전송하지 않고 원본 인덱스만 전달하는지 검사
- 라이브 저장과 콜드 스타트 batch가 동일한 백엔드 계산 함수를 사용하는지 검사
- tail 삭제 후 남은 pair가 turn `1, 2`인지 검사
- `fmIndex`가 선택한 대체 시작문이 turn 0인지 검사
- 같은 논리 턴의 다른 출력이 append가 아니라 rollback 후 replace되는지 검사
- `timeline_copy`, `timeline_migrate`, `timeline_attach` 각각에서 기존 7턴 보호와 첫 로컬 턴 8 삭제를 검사
- 완전히 빈 active chat도 첫 턴 삭제를 ledger로 검증하는지 검사

## 9. ChromaDB 벡터 검색 정확성

### 완료

- 순위 기반 가짜 점수 `1.00, 0.95, 0.90...`를 제거했다.
- ChromaDB 거리와 저장 벡터를 보존하고 현재 질의 벡터와 실제 코사인 유사도를 계산한다.
- 실제 유사도로 후보를 다시 정렬하고 최소 관련도보다 낮은 후보를 제외한다.
- 적격 벡터 후보가 부족하면 어휘 관련성·최근 기억 경로가 남은 예산을 채운다.
- 진단 표면의 `similarity`와 `rerank_score`는 실제 점수가 있을 때만 의미를 갖는다.

관련 체크포인트:

- `8eb784c fix: make vector recall query sensitive`
- `efa44e2 fix: remove synthetic ANN score surfaces`

상세 문서: `3.0-vector-recall-query-sensitivity-fix.md`

### 완료·실사용 확인 필요

- 장기 세션에서 질의 변경에 따라 후보가 실제로 바뀌는지 라이브 검증이 남아 있다.
- 과거 데이터는 임베딩 모델과 ChromaDB 컬렉션 차원이 맞는 상태에서 재색인이 필요할 수 있다.

## 11. 원작 자료 라이브러리

### 확정된 제품 목표

- 1순위는 원작 로어북이 있는 기존 봇에서 부족한 자료와 고증을 보충하는 것이다.
- 2순위는 관련 로어북이 없는 봇에서도 같은 자료만으로 플레이할 수 있는 기본 설정 경로다.
- 기본 역할은 `supplement`이며, 로어북을 무시하고 원작 DB 전체를 반복 주입하지 않는다.
- `supplement/primary`는 세션 연결 UI에서 사용자가 명시적으로 선택하며,
  잘못된 저장값은 내부 `unknown`으로 차단한다. 로어북 개수나 봇 형태를 보고
  단독 모드로 자동 추측하지 않는다.
- 요청·장면 coverage 대조, 필드 coverage 색인, 실제 적용과 모드별 주입 정책은
  5단계까지 구현했다.
- 구현 시 JS는 RisuAI 상태 관측만 담당하고, coverage 판정·중복 제거·주입 계획은 백엔드가 담당한다.
- 기존 평론가 complete-turn 호출로 다음 턴용 coverage delta를 갱신하며, 최초 연결 또는 로어북 해시 변경 때만 전체 분석을 수행한다.
- 부족 범위는 `정적 보유`, `현재 요청 활성`, `현재 장면 필요`를 분리하고, 실제 보충은 `현재 장면 필요 - 현재 요청 활성`로 계산한다.
- coverage는 `covered/partial/missing/conflict/not_applicable/unknown`과 원작 ID, 로어북 ID, 요청 위치, 누락 필드, 신뢰도, reason code를 함께 기록해야 한다.
- 전체 작품에서 빠진 자료를 매 턴 주입하지 않고 현재 등장인물·장소·사건·세계 규칙에 필요한 부족분만 선택한다.

3. `coverage_field_index.v1` 필드 단위 coverage 색인: 완료 (2026-07-14)
   - 연결 작품의 승인 entity/alias/claim/timeline과 구조화 metadata leaf를
     전체 필드 인벤토리로 만든다.
   - 저장된 대화와 최신 실제 입력을 평탄화된 요청 `messages`에서 제외해
     system-role 및 user-role 로어북/프롬프트 문맥을 함께 보존한다.
   - binding마다 현재 snapshot 한 행과 field 행만 교체하며 이력을 누적하지 않는다.
     활성 문맥 hash와 작품 inventory hash가 같으면 DB 쓰기를 건너뛴다.
   - Chroma 후보 밖의 장면 필요 source도 `needed_sources`에서 진단하지만 고정 점수,
     가짜 후보, 의미 유사도는 생성하지 않는다.
   - ChromaDB와 embedding 설정이 없어도 필드 색인은 계산·저장된다.
   - 복사·이동 대상에서는 snapshot을 복제하지 않고 대상 세션 문맥으로 재생성한다.
   - `Archive Center.js` 변경과 추가 LLM 호출은 없으며 `injection_filtered=false`를
     유지한다.
4. `coverage_apply.v1` 실제 주입 필터: 완료 (2026-07-14)
   - 원시 Chroma 후보인 `selected`와 실제 프롬프트용 `injection_items`를 분리했다.
   - `partial`의 빠진 필드와 `missing`만 주입하고 `covered`, `not_applicable`,
     `unknown`, `conflict`는 자동 주입하지 않는다.
   - Chroma 후보 항목은 반환 순서와 raw distance/cosine을 보존한다.
   - 필드 색인이 후보 창 밖에서 찾은 장면 필요 항목은 별도 provenance로 보충하되
     Chroma rank, distance, cosine을 만들지 않는다.
   - 실제 주입 건수와 대시보드 표시를 `injection_items` 기준으로 통일했다.
   - 입력 원본 전체는 `reference_documents.raw_text`, 정리본은 기존
     entity/claim/timeline 테이블에 함께 유지한다.
   - 추출 excerpt가 실제 원문 chunk에 존재할 때만 `evidence_grounded=true`로
     저장하며, 원문에 없는 의역은 경고 후 원문 근거에서 제외한다.
   - `injection_items`에 정리본과 검증된 원문 excerpt 및 document/chunk provenance를
     함께 반환한다. 두 문장이 같으면 실제 프롬프트에는 한 번만 넣는다.
   - 매 턴 원본 문서 전체를 다시 읽지 않으며 검증된 짧은 excerpt만 사용한다.
   - 표식이 없는 기존 excerpt는 보존하되 자동 원문 주입하지 않는다. 승인된 기존
     자료는 재추출로 덮어쓰지 않으며, 새로 추출되는 후보부터 현재 원문 검증 계약을
     적용한다.
   - claim 근거를 수동 편집하면 기존 원문 검증 표식을 해제해 편집 문장을 원문으로
     오인하지 않는다.
   - `Archive Center.js` 변경과 추가 LLM 호출은 없다.
5. `supplement/primary/unknown` 운용 확정: 완료 (2026-07-14)
   - 작품 관계 역할인 `binding_role`과 운용 정책인 `reference_mode`를 분리했다.
   - 기존 연결과 신규 연결의 기본값은 `supplement`이며 기존 로어북·프롬프트에서
     확인되지 않은 `partial`/`missing` 필드만 보충한다.
   - `primary`는 사용자가 UI에서 직접 선택한 경우에만 적용한다. 같은 hard filter를
     통과한 실제 ChromaDB 후보를 원작 설정의 기본 맥락으로 사용할 수 있고 후보
     창은 최대 4개 넓어진다.
   - Chroma rank, raw distance, 실제 cosine을 재계산·정규화·고정하지 않는다.
   - 빈 레거시 값은 호환상 `supplement`, 명시적으로 잘못된 값은 `unknown`으로
     취급해 자동 주입하지 않는다.
   - 스키마 호환 migration, Store CRUD, binding preview/apply/update, 세션
     복사·이동, UI 선택과 대시보드 진단값에 같은 모드를 연결했다.
   - 단독 모드의 실제 Chroma 후보가 엔티티나 관계 claim을 선택하면, 같은 주체
     또는 명시적으로 언급된 엔티티에 연결된 승인 `relationship` claim을 최대
     4개까지 동반 후보로 만든다. 이 항목은 `primary_relation_expansion`으로
     표시하고 Chroma rank, distance, cosine을 생성하지 않는다.
   - `narrator_only` claim은 서술 정확도를 위해 사용할 수 있지만 캐릭터 지식으로
     전파하지 말라는 표식을 붙인다. `entity_scoped`와 비공개 claim은 현재 장면의
     허용된 knower가 없으면 계속 차단한다.

다음 작업은 기존 승인 자료의 원문 excerpt 무LLM 재검증과 실제 RisuAI에서
보완/단독 모드의 장기 회귀 검증이다.

### 완료된 기반과 관리 기능

- 작품, 연속성, 문서, 연표, entity, claim, 세션 binding용 MariaDB 스키마와 Store CRUD 기반을 추가했다.
- TXT, Markdown, JSON 파일 저장과 SHA-256 중복 방지를 구현했다.
- 평론가 기반 조각 추출 작업 큐와 자동 검수를 구현했다.
- 생성 자료 보관함에서 승인 자료를 인물, 장소, 물품, 세력·조직, 연표, 세계 규칙, 관계·사건으로 확인할 수 있다.
- 사용자 수정, 제외, 복구, 원문 근거, 충돌 진단, 완전 동일 중복 억제를 구현했다.
- 연표는 숫자만 파싱하지 않고 평론가의 연대·선후 관계 근거로 정렬하며 미해결 항목을 보존한다.
- 백엔드 미지원 패키지와 스키마 불일치를 UI에서 구분해 표시한다.
- 현재 RisuAI 세션에 작품·연속성을 연결하는 preview/apply/update/unlink API를 추가했다.
- 현재 시점, 공개 상한, 원작 분기 시점, 연결 역할과 미래 정보 정책을 UI에서 선택하며 최종 유효성은 백엔드가 판정한다.
- 연결 해제는 binding/runtime만 제거하고 재사용 원작 자료는 유지한다.
- 승인된 연표·인물/장소/물품·사실/설정을 `archive_center_reference_vectors` 전용 컬렉션에 색인한다.
- 재색인은 새 임베딩과 upsert가 성공한 뒤에만 이전 stale 문서를 지우며, 실패하면 기존 색인을 유지한다.
- 검색은 ChromaDB 응답 순서와 원시 distance를 그대로 반환한다. 순위 기반 정규화 유사도와 클라이언트 재정렬은 만들지 않는다.
- 반환된 저장 임베딩이 있을 때만 질의 임베딩과의 실제 cosine을 별도 진단값으로 계산한다.
- UI에서 색인 수와 모델을 확인하고 검색 문장별 실제 Chroma rank/distance/cosine을 비교할 수 있다.
- 일반 흐름에서는 평론가 자동 승인과 연표 정리가 끝난 직후 승인 자료를 같은 작업에서 자동 색인한다. 기존 승인 자료의 누락도 UI 상태 확인 시 한 번 자동 복구한다.
- 수동 재색인, 원시 ChromaDB 상태와 검색 결과 비교는 디버그 모드에서만 노출한다.
- 검색 직전 MariaDB의 현재 승인 ID를 대조해 재색인 전 제외된 오래된 벡터를 사용하지 않는다.
- 색인과 검색의 임베딩 provider/model이 다르면 같은 차원이어도 검색을 차단하고 재색인을 요구한다.
- 세션 recall preview는 연결된 binding별 실제 ChromaDB 순위와 원시 측정값을 보존한 채 시점·분기·공개 상한·인물별 지식 범위를 MariaDB에서 hard filter한다.
- 첫 턴 fast boot에서도 일반 과거 기억 검색은 생략하고, 연결된 원작 binding은 별도 예산으로 주입한다. JS는 백엔드의 `reference_text`를 실제 최종 보조 컨텍스트에 전달하고 대시보드에 반영 건수를 표시한다.
- 세션 binding의 존재 자체를 사용 선언으로 취급한다. 별도의 `사용`·`턴에 참고 자료 주입` 토글은 두지 않으며 연결 해제가 미사용 처리다. prepare-turn은 연결된 원작의 작은 참고 블록을 기존 세션 기억 뒤에 붙인다.
- 세션 복사·이동은 binding/runtime만 복제하고 migration 전용 매핑으로 롤백 범위를 제한한다. 턴 rollback은 binding을 보존하고 삭제 턴에서 나온 runtime 후보만 비운다.

## 16. 작업 원칙

- 현재 정상 동작을 신규 구조 이전의 재구현 대상으로 취급하지 않는다.
- RisuAI 내부에서만 관찰할 수 있는 메시지 인덱스, active chat, 최종 표시 문자열은 JS가 수집한다.
- DB 범위, migration, Store, Vector, 멱등성 결정은 백엔드가 소유한다.
- 삭제 문제를 JS 함수 추가만으로 우회하지 않고 기존 rollback decision과 cascade cleanup을 재사용한다.
- 콘텐츠가 같다는 이유만으로 다른 논리 턴을 합치지 않는다.
- 복사·이동·연결 baseline 이전의 DB 범위는 로컬 turn 0 또는 1로 오인해 삭제하지 않는다.
- 되돌린 시도는 문서에서 지우지 않고 되돌림 상태와 이유를 남긴다.
- GitHub 배포는 개별 실사용 검증이 끝난 체크포인트만 대상으로 한다.
