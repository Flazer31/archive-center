# Archive Center 3.0 구현 작업 원장

기준일: 2026-07-15
기준 범위: `release-2.5.0-rc4` 기준 커밋 `26818fe` 이후  
기준 부모 커밋: `9c6cac1`
현재 상태: 아래 원작 DB 변경과 3.0 문서 정리를 같은 로컬 체크포인트로 고정

## 문서 목적

이 문서는 2.5.0 RC4 이후 3.0을 위해 실제로 수행한 작업을 하나의 원장으로
보존한다. 기능별 상세 설계는 기존 `docs/3.0-*.md` 문서를 따르되, 현재
구현 여부와 되돌린 시도는 이 문서를 우선 기준으로 삼는다.

상태 의미:

- **완료**: 코드와 자동 회귀 검증이 끝난 활성 구현
- **완료·실사용 확인 필요**: 자동 검증은 끝났지만 장기 세션이나 실제 RisuAI 환경 확인이 남음
- **체크포인트 완료·실사용 확인 필요**: 코드와 자동 회귀를 같은 로컬 커밋으로 고정했지만 라이브 사용자 검증이 남음
- **부분 완료**: 목표 중 확인된 하위 계약만 구현됨
- **보류**: 사용자가 후속 개선으로 미뤘거나 실험 결과에 따라 폐기할 수 있음
- **되돌림**: 시도했으나 위험 또는 회귀 때문에 활성 코드에서 제거
- **미완료**: 설계 또는 기반만 있고 최종 기능은 아직 열지 않음

## 2026-07-15 계획 대비 현재 상태 재감사

이 절은 오래된 계획 문서의 빈 체크박스를 그대로 미완료로 세지 않고, 활성
`source`의 현재 커밋과 작업 트리를 실제 코드·테스트와 대조한 결과다. 현재
상태를 묻는 경우 이 절을 우선한다. 자동 검증 완료와 라이브 사용자 검증
완료를 섞어 "완료"라고 부르지 않는다.

### 이번 로컬 체크포인트에 포함된 원작 DB 변경

| 영역 | 현재 판정 | 확인 근거와 남은 조건 |
| --- | --- | --- |
| 원작 전용 예산 계약 | 체크포인트 완료·실사용 확인 필요 | `reference_injection_budget.v1`이 기본 기억 상한 `M`을 기준으로 보완형 `floor(M/2)`, 단독형 `M`을 원작 총예산으로 계산한다. 메인 기억 예산을 줄이지 않는 가산형 레인이다. `go test ./internal/httpapi -count=1` 통과. |
| 단독형 Primary Canon Base | 체크포인트 완료·실험 | 원작 총예산 안의 하위 상한으로 기반 자료와 장면 자료를 나누며 합계가 원작 총예산을 넘지 않는다. 이전 `5c5fb89` 구현은 `9c6cac1`에서 되돌렸고, 이번 새 구현은 자동 회귀는 통과했지만 라이브 확인 전이다. 필요하면 단독형만 폐기할 수 있다. |
| 원작 임의 예산 fallback 제거 | 체크포인트 완료·실사용 확인 필요 | 과거 `max_injection_chars<=0`일 때 임의로 3000자를 주던 경로를 제거했다. 명시적으로 해결된 원작 예산이 0이면 주입하지 않고, 첫 턴은 별도 전달된 기준값으로만 계산한다. |
| 특정 작품 예시 추출 프롬프트 제거 | 체크포인트 완료 | 특정 자료에서 유래한 예시 문구를 제거하고 일반적인 원문 근거·entity/claim 판정 규칙으로 바꿨다. |
| 원작 후보 개수 정책 통합 | 체크포인트 완료·실사용 확인 필요 | 승인 source 수와 요청 limit을 기준으로 실제 Chroma 후보 창을 만들고, 관계 확장의 별도 고정 4개 상한을 제거했다. 고정 점수·합성 Chroma 순위는 만들지 않는다. |
| 원작 recall 상태 분리 | 체크포인트 완료·실사용 확인 필요 | `failed/degraded/empty/ready`를 구분한다. 오류를 성공 `ready`로 바꾸지 않는다. 다만 공통 안정 error envelope까지 완료된 것은 아니다. |
| 원작 의존성의 메인 런타임 격리 | 체크포인트 완료·실사용 확인 필요 | 원작 전용 vector open/health 실패는 메인 시작, `/ready`, 저장, 장기 기억을 막지 않는다. `/ready`에는 `reference_vector_ready`와 `reference_vector_degraded`를 별도로 표시하고, recall은 실패 상태와 경고를 반환한다. 집중 테스트 통과. |
| Effective Input의 원작 표시 분리 | 체크포인트 완료·실사용 확인 필요 | 기존 `Assembled Auxiliary Context`와 별도로 `Original Work Reference Context` preview를 렌더링한다. 실제 payload 적용은 여전히 JS의 필수 host-adapter 역할이다. |
| 원작 대시보드 경고 | 체크포인트 완료·실사용 확인 필요 | 원작으로 연결했는데 vector/embedding/recall에 문제가 있으면 원작 전용 상태와 원인 표시가 보인다. 메인 ready와는 분리된다. |

주의: 원작 예산은 회계와 주입 레인은 분리됐지만, 사용자 설정값은 별도의
네 번째 숫자가 아니라 기존 기본 기억 상한에서 파생된다. 현재 계약은
`supplement=M/2`, `primary=M`이며 이를 "서로 독립적으로 입력하는 두 예산
설정"으로 기록하지 않는다.

### 명시적으로 보류한 항목

- Go가 만든 최종 주입 계획이 UI 상한보다 클 때 JS에서 다시 자르는 개선은
  현재 치명 오류가 재현되지 않아 후속 개선으로 보류했다. 새 JS 정책을
  추가하지 않는다.
- 원작 자료와 장기 기억 개체를 노드 그래프로 연결하거나 시각화하는 작업은
  원작 DB 완료 조건에서 제외하고 장기 기억 전체의 별도 설계로 보류했다.
- 단독형은 보조 기능의 실험 경로다. 메인 Archive Center를 침범하거나 장기
  기억 우선순위를 약화시키면 Primary Canon Base만 축소·폐기할 수 있다.

### 다음 작업 순서

- 0턴 시작문 개선은 보류한다. 여러 시작문을 선택하거나 분할 조합하는 호스트에서는
  원작 검색에 사용하기 전에 실제 선택된 시작 sequence를 식별하는 canonical contract가
  먼저 필요하다. 그 계약 없이 starting message, messages 배열, 0턴 저장물을 원작 검색
  질의로 사용하지 않는다.

1. 이번에 고정한 원작 격리·예산·UI 분리 체크포인트로 라이브 테스트용
   패키지를 만든다.
2. 옵트인 라이브 테스트에서 보완 모드를 먼저 검증한다. 실제 사용자 반응으로
   직전 입력과 최근 완성 대화에 맞는 `partial/missing` 자료만 보완하는지,

원작 고증 품질과 보완 효과는 개발자 한 명의 수동 확인으로 완료 판정하지

- 라이브 테스트는 명시적으로 참여한 사용자에게만 제공한다.
- 첫 실험은 기본 역할인 `supplement`만 대상으로 한다. `primary`는 별도 실험

- 자동으로 대화 원문이나 원작 문서를 외부 수집하지 않는다. 진단 자료는
  사용자가 확인하고 내보낸 상태·건수·reason code·preview를 기준으로 받는다.

  중복되지 않음`, `메인 기억을 방해하지 않음`, `출력이 원작과 무관하게
  이탈하지 않음`을 분리해서 기록한다.
- 원작 의존성 오류가 메인 시작, `/ready`, 저장, 장기 기억을 막으면 즉시 중단한다.
- 고정 점수·고정 작품명·고정 인물/세력 수나 무조건 성공 처리를 품질 보정으로

- 자료 자체의 오류는 ranking 조정으로 숨기지 않고 entity/claim/relationship을
  정정하고 재색인한 뒤 같은 사례를 다시 확인한다.
- 라이브 결과가 부족하면 `실사용 확인 필요`를 유지한다. 참가자 수나 기간만으로

## 7. RisuAI 메시지 인덱스와 DB 턴의 직접 연결

### 완료

RisuAI active chat의 원본 배열 인덱스를 보존하고 다음 구조를 DB 논리 턴의
기준으로 사용한다.

| RisuAI 메시지 | DB 논리 턴 |
| --- | ---: |
| user `Index 0`, assistant `Index 1` | turn 1 |
| user `Index 2`, assistant `Index 3` | turn 2 |
| user `Index 4`, assistant `Index 5` | turn 3 |

적용 규칙:

- JS는 RisuAI의 원본 user/assistant 인덱스와 관찰 순서만 백엔드에 전달한다.
- 일반 세션의 `floor(user_index / 2) + 1` 계산과 최종 DB 턴 선택은 백엔드가 담당한다.
- 원본 인덱스를 사용할 수 없는 경우에만 백엔드가 관찰된 pair 순서를 대체 근거로 사용한다.
- 복사·이동·연결 세션은 backend baseline을 적용한다.
  - 기존 7턴을 빈 채팅에 가져온 뒤 로컬 turn 1은 DB turn 8이다.
  - 그 첫 로컬 턴을 삭제하면 turn 8부터만 삭제하고 turn 1~7은 보호한다.
- 화면의 같은 논리 턴이 DB에 이미 있으나 내용이 바뀌었으면 새 번호로 추가하지 않는다.
  - 기존 backend rollback 결정으로 해당 턴 이후를 정리한다.
  - 최신 user/assistant pair를 같은 턴 번호로 다시 저장한다.
- active chat이 완전히 빈 배열이 되어도 이전 rollback ledger 전체 제거를 검증할 수 있다.
- 빈 배열이라는 이유만으로 삭제하지 않으며, 검증 ledger 또는 세션 baseline 보호 계약을 통과해야 한다.
- 캐릭터의 `firstMessage`, `alternateGreetings`, active chat의 `fmIndex`를 읽어 실제 선택된 시작 메시지를 DB `turn 0`으로 저장한다.

- JS가 계산된 `turnIndex`를 만들거나 전송하지 않고 원본 인덱스만 전달하는지 검사

- 구현 시 JS는 RisuAI 상태 관측만 담당하고, coverage 판정·중복 제거·주입 계획은 백엔드가 담당한다.
- 기존 평론가 complete-turn 호출로 다음 턴용 coverage delta를 갱신하며, 최초 연결 또는 로어북 해시 변경 때만 전체 분석을 수행한다.
- 부족 범위는 `정적 보유`, `현재 요청 활성`, `현재 장면 필요`를 분리하고, 실제 보충은 `현재 장면 필요 - 현재 요청 활성`로 계산한다.
- coverage는 `covered/partial/missing/conflict/not_applicable/unknown`과 원작 ID, 로어북 ID, 요청 위치, 누락 필드, 신뢰도, reason code를 함께 기록해야 한다.
- 전체 작품에서 빠진 자료를 매 턴 주입하지 않고 현재 등장인물·장소·사건·세계 규칙에 필요한 부족분만 선택한다.

5. `supplement/primary/unknown` 운용 확정: 완료 (2026-07-14)

   - 기존 연결과 신규 연결의 기본값은 `supplement`이며 기존 로어북·프롬프트에서
     확인되지 않은 `partial`/`missing` 필드만 보충한다.
   - `primary`는 사용자가 UI에서 직접 선택한 경우에만 적용한다. 같은 hard filter를
     통과한 실제 ChromaDB 후보를 원작 설정의 기본 맥락으로 사용할 수 있고 후보

   - 스키마 호환 migration, Store CRUD, binding preview/apply/update, 세션
     복사·이동, UI 선택과 대시보드 진단값에 같은 모드를 연결했다.
   - 단독 모드의 실제 Chroma 후보가 엔티티나 관계 claim을 선택하면, 같은 주체
     또는 명시적으로 언급된 엔티티에 연결된 승인 `relationship` claim을 최대
     4개까지 동반 후보로 만든다. 이 항목은 `primary_relation_expansion`으로
     표시하고 Chroma rank, distance, cosine을 생성하지 않는다.
   - `narrator_only` claim은 서술 정확도를 위해 사용할 수 있지만 캐릭터 지식으로
     전파하지 말라는 표식을 붙인다. `entity_scoped`와 비공개 claim은 현재 장면의
     허용된 knower가 없으면 계속 차단한다.

- 작품, 연속성, 문서, 연표, entity, claim, 세션 binding용 MariaDB 스키마와 Store CRUD 기반을 추가했다.

- 현재 시점, 공개 상한, 원작 분기 시점, 연결 역할과 미래 정보 정책을 UI에서 선택하며 최종 유효성은 백엔드가 판정한다.

- 승인된 연표·인물/장소/물품·사실/설정을 `archive_center_reference_vectors` 전용 컬렉션에 색인한다.
- 재색인은 새 임베딩과 upsert가 성공한 뒤에만 이전 stale 문서를 지우며, 실패하면 기존 색인을 유지한다.
- 검색은 ChromaDB 응답 순서와 원시 distance를 그대로 반환한다. 순위 기반 정규화 유사도와 클라이언트 재정렬은 만들지 않는다.
- 반환된 저장 임베딩이 있을 때만 질의 임베딩과의 실제 cosine을 별도 진단값으로 계산한다.
- UI에서 색인 수와 모델을 확인하고 검색 문장별 실제 Chroma rank/distance/cosine을 비교할 수 있다.
- 일반 흐름에서는 평론가 자동 승인과 연표 정리가 끝난 직후 승인 자료를 같은 작업에서 자동 색인한다. 기존 승인 자료의 누락도 UI 상태 확인 시 한 번 자동 복구한다.
- 수동 재색인, 원시 ChromaDB 상태와 검색 결과 비교는 디버그 모드에서만 노출한다.
- 검색 직전 MariaDB의 현재 승인 ID를 대조해 재색인 전 제외된 오래된 벡터를 사용하지 않는다.
- 색인과 검색의 임베딩 provider/model이 다르면 같은 차원이어도 검색을 차단하고 재색인을 요구한다.
- 세션 recall preview는 연결된 binding별 실제 ChromaDB 순위와 원시 측정값을 보존한 채 시점·분기·공개 상한·인물별 지식 범위를 MariaDB에서 hard filter한다.
- 첫 턴 fast boot에서도 일반 과거 기억 검색은 생략하고, 연결된 원작 binding은 별도 예산으로 주입한다. JS는 백엔드의 `reference_text`를 실제 최종 보조 컨텍스트에 전달하고 대시보드에 반영 건수를 표시한다.
- 세션 binding의 존재 자체를 사용 선언으로 취급한다. 별도의 `사용`·`턴에 참고 자료 주입` 토글은 두지 않으며 연결 해제가 미사용 처리다. prepare-turn은 연결된 원작의 작은 참고 블록을 기존 세션 기억 뒤에 붙인다.
- 세션 복사·이동은 binding/runtime만 복제하고 migration 전용 매핑으로 롤백 범위를 제한한다. 턴 rollback은 binding을 보존하고 삭제 턴에서 나온 runtime 후보만 비운다.

## 16. 작업 원칙

- 현재 정상 동작을 신규 구조 이전의 재구현 대상으로 취급하지 않는다.
- RisuAI 내부에서만 관찰할 수 있는 메시지 인덱스, active chat, 최종 표시 문자열은 JS가 수집한다.
- DB 범위, migration, Store, Vector, 멱등성 결정은 백엔드가 소유한다.
- 삭제 문제를 JS 함수 추가만으로 우회하지 않고 기존 rollback decision과 cascade cleanup을 재사용한다.
- 콘텐츠가 같다는 이유만으로 다른 논리 턴을 합치지 않는다.
- 복사·이동·연결 baseline 이전의 DB 범위는 로컬 turn 0 또는 1로 오인해 삭제하지 않는다.
- 되돌린 시도는 문서에서 지우지 않고 되돌림 상태와 이유를 남긴다.
- GitHub 배포는 개별 실사용 검증이 끝난 체크포인트만 대상으로 한다.
