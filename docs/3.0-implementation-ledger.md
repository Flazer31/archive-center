# Archive Center 3.0 구현 작업 원장

기준일: 2026-07-13  
기준 범위: `release-2.5.0-rc4` 기준 커밋 `26818fe` 이후  
현재 로컬 체크포인트: `82bfe09`

## 문서 목적

이 문서는 2.5.0 RC4 이후 3.0을 위해 실제로 수행한 작업을 하나의 원장으로
보존한다. 기능별 상세 설계는 기존 `docs/3.0-*.md` 문서를 따르되, 현재
구현 여부와 되돌린 시도는 이 문서를 우선 기준으로 삼는다.

상태 의미:

- **완료**: 코드와 자동 회귀 검증이 끝난 활성 구현
- **완료·실사용 확인 필요**: 자동 검증은 끝났지만 장기 세션이나 실제 RisuAI 환경 확인이 남음
- **되돌림**: 시도했으나 위험 또는 회귀 때문에 활성 코드에서 제거
- **미완료**: 설계 또는 기반만 있고 최종 기능은 아직 열지 않음

- 번들 Node.js를 자동 탐색해 JavaScript 런타임 검사를 조용히 건너뛰지 않게 했다.

## 7. RisuAI 메시지 인덱스와 DB 턴의 직접 연결

### 완료

RisuAI active chat의 원본 배열 인덱스를 보존하고 다음 구조를 DB 논리 턴의
기준으로 사용한다.

| RisuAI 메시지 | DB 논리 턴 |
| --- | ---: |
| user `Index 0`, assistant `Index 1` | turn 1 |
| user `Index 2`, assistant `Index 3` | turn 2 |
| user `Index 4`, assistant `Index 5` | turn 3 |

적용 규칙:

- JS는 RisuAI의 원본 user/assistant 인덱스와 관찰 순서만 백엔드에 전달한다.
- 일반 세션의 `floor(user_index / 2) + 1` 계산과 최종 DB 턴 선택은 백엔드가 담당한다.
- 원본 인덱스를 사용할 수 없는 경우에만 백엔드가 관찰된 pair 순서를 대체 근거로 사용한다.
- 복사·이동·연결 세션은 backend baseline을 적용한다.
  - 기존 7턴을 빈 채팅에 가져온 뒤 로컬 turn 1은 DB turn 8이다.
  - 그 첫 로컬 턴을 삭제하면 turn 8부터만 삭제하고 turn 1~7은 보호한다.
- 화면의 같은 논리 턴이 DB에 이미 있으나 내용이 바뀌었으면 새 번호로 추가하지 않는다.
  - 기존 backend rollback 결정으로 해당 턴 이후를 정리한다.
  - 최신 user/assistant pair를 같은 턴 번호로 다시 저장한다.
- active chat이 완전히 빈 배열이 되어도 이전 rollback ledger 전체 제거를 검증할 수 있다.
- 빈 배열이라는 이유만으로 삭제하지 않으며, 검증 ledger 또는 세션 baseline 보호 계약을 통과해야 한다.
- 캐릭터의 `firstMessage`, `alternateGreetings`, active chat의 `fmIndex`를 읽어 실제 선택된 시작 메시지를 DB `turn 0`으로 저장한다.

관련 체크포인트:

- `c3acd74 fix: reconcile deletion before immediate next turn`
- `82bfe09 fix: align persistence with Risu message indexes`

자동 검증:

- 원본 인덱스 `0/1, 2/3, 4/5`가 turn `1, 2, 3`이 되는지 검사
- JS가 계산된 `turnIndex`를 만들거나 전송하지 않고 원본 인덱스만 전달하는지 검사
- 라이브 저장과 콜드 스타트 batch가 동일한 백엔드 계산 함수를 사용하는지 검사
- tail 삭제 후 남은 pair가 turn `1, 2`인지 검사
- `fmIndex`가 선택한 대체 시작문이 turn 0인지 검사
- 같은 논리 턴의 다른 출력이 append가 아니라 rollback 후 replace되는지 검사
- `timeline_copy`, `timeline_migrate`, `timeline_attach` 각각에서 기존 7턴 보호와 첫 로컬 턴 8 삭제를 검사
- 완전히 빈 active chat도 첫 턴 삭제를 ledger로 검증하는지 검사

## 9. ChromaDB 벡터 검색 정확성

### 완료

- 순위 기반 가짜 점수 `1.00, 0.95, 0.90...`를 제거했다.
- ChromaDB 거리와 저장 벡터를 보존하고 현재 질의 벡터와 실제 코사인 유사도를 계산한다.
- 실제 유사도로 후보를 다시 정렬하고 최소 관련도보다 낮은 후보를 제외한다.
- 적격 벡터 후보가 부족하면 어휘 관련성·최근 기억 경로가 남은 예산을 채운다.
- 진단 표면의 `similarity`와 `rerank_score`는 실제 점수가 있을 때만 의미를 갖는다.

관련 체크포인트:

- `8eb784c fix: make vector recall query sensitive`
- `efa44e2 fix: remove synthetic ANN score surfaces`

상세 문서: `3.0-vector-recall-query-sensitivity-fix.md`

### 완료·실사용 확인 필요

- 장기 세션에서 질의 변경에 따라 후보가 실제로 바뀌는지 라이브 검증이 남아 있다.
- 과거 데이터는 임베딩 모델과 ChromaDB 컬렉션 차원이 맞는 상태에서 재색인이 필요할 수 있다.

## 11. 원작 자료 라이브러리

### 완료된 기반과 관리 기능

- 작품, 연속성, 문서, 연표, entity, claim, 세션 binding용 MariaDB 스키마와 Store CRUD 기반을 추가했다.
- TXT, Markdown, JSON 파일 저장과 SHA-256 중복 방지를 구현했다.
- 평론가 기반 조각 추출 작업 큐와 자동 검수를 구현했다.
- 생성 자료 보관함에서 승인 자료를 인물, 장소, 물품, 세력·조직, 연표, 세계 규칙, 관계·사건으로 확인할 수 있다.
- 사용자 수정, 제외, 복구, 원문 근거, 충돌 진단, 완전 동일 중복 억제를 구현했다.
- 연표는 숫자만 파싱하지 않고 평론가의 연대·선후 관계 근거로 정렬하며 미해결 항목을 보존한다.
- 백엔드 미지원 패키지와 스키마 불일치를 UI에서 구분해 표시한다.
- 현재 RisuAI 세션에 작품·연속성을 연결하는 preview/apply/update/unlink API를 추가했다.
- 현재 시점, 공개 상한, 원작 분기 시점, 연결 역할과 미래 정보 정책을 UI에서 선택하며 최종 유효성은 백엔드가 판정한다.
- 연결 해제는 binding/runtime만 제거하고 재사용 원작 자료는 유지한다.
- 승인된 연표·인물/장소/물품·사실/설정을 `archive_center_reference_vectors` 전용 컬렉션에 색인한다.
- 재색인은 새 임베딩과 upsert가 성공한 뒤에만 이전 stale 문서를 지우며, 실패하면 기존 색인을 유지한다.
- 검색은 ChromaDB 응답 순서와 원시 distance를 그대로 반환한다. 순위 기반 정규화 유사도와 클라이언트 재정렬은 만들지 않는다.
- 반환된 저장 임베딩이 있을 때만 질의 임베딩과의 실제 cosine을 별도 진단값으로 계산한다.
- UI에서 색인 수와 모델을 확인하고 검색 문장별 실제 Chroma rank/distance/cosine을 비교할 수 있다.
- 검색 직전 MariaDB의 현재 승인 ID를 대조해 재색인 전 제외된 오래된 벡터를 사용하지 않는다.
- 색인과 검색의 임베딩 provider/model이 다르면 같은 차원이어도 검색을 차단하고 재색인을 요구한다.

- 과거 기억이 현재 사실을 덮거나, 현재 장면에 없는 인물의 주관 기억이 서사의 중심으로 난입하지 않도록 소유자·현재 장면 근거를 사용한다.
- 객관적 상태와 인물별 믿음, 직접 근거, 상태 변화 연결 계약과 자동 테스트가 있다.

장기 세션에서 관계 반전, 생사 정정, 비밀 공개, 오해 해소가 누적될 때의 실사용 검증은 남아 있다.

## 16. 작업 원칙

- 현재 정상 동작을 신규 구조 이전의 재구현 대상으로 취급하지 않는다.
- RisuAI 내부에서만 관찰할 수 있는 메시지 인덱스, active chat, 최종 표시 문자열은 JS가 수집한다.
- DB 범위, migration, Store, Vector, 멱등성 결정은 백엔드가 소유한다.
- 삭제 문제를 JS 함수 추가만으로 우회하지 않고 기존 rollback decision과 cascade cleanup을 재사용한다.
- 콘텐츠가 같다는 이유만으로 다른 논리 턴을 합치지 않는다.
- 복사·이동·연결 baseline 이전의 DB 범위는 로컬 turn 0 또는 1로 오인해 삭제하지 않는다.
- 되돌린 시도는 문서에서 지우지 않고 되돌림 상태와 이유를 남긴다.
- GitHub 배포는 개별 실사용 검증이 끝난 체크포인트만 대상으로 한다.
