# Archive Center 3.0 JS/Backend Duplicate Computation Removal

## Meaning

This work does not delete duplicate database values. It removes repeated work
where Go has already returned a complete result and JavaScript performs the same
read or formatting operation again during the same turn.

The transition rule is always:

1. Prefer a complete, version-compatible backend result.
2. Keep the existing JavaScript path as a fallback for an absent, old, partial,
   degraded, or unreachable backend.
3. Remove the fallback only after release-level compatibility evidence exists.

## Ownership boundary

JavaScript continues to own:

- observing RisuAI hooks, active chat, payloads, and displayed output;
- applying an injection to the actual RisuAI message array;
- confirming the final displayed assistant string;
- local UI rendering and localization;
- compatibility fallback when the backend result is incomplete.

The backend owns:

- MariaDB and ChromaDB reads;
- selection, ranking, and bounded recall results;
- rendered store-backed memory sections;
- persistence and migration range decisions;
- stable error codes and backend timing.

## Completed scope

The prepare-turn runtime support contract is
`session_state.runtime_support.v1`. It carries endpoint-equivalent records for:

- active states, including content;
- storylines, including context, key points, and tensions;
- character states, including status, relationships, personality, and speech;
- pending threads, including description, type, owner, target, and lifecycle
  turns.

JavaScript now reuses these records before any individual endpoint fallback,
but only when the RisuAI compatibility read plan contains exactly the same one
session. Copy, connect, and move compatibility plans keep the existing
multi-session merge path.

The removed repeated work is:

- `/active-states/{session}` after prepare-turn already read active states;
- `/storylines/{session}` after prepare-turn already read storylines;
- `/characters/{session}` after prepare-turn already read character states;
- `/pending-threads/{session}` after prepare-turn already read pending threads;

Compatibility behavior remains:

- old or partial backend packets do not claim a complete section;
- a declared active-state count without usable content triggers the old endpoint;
- backend unavailability retains all existing JavaScript reads and formatting;
- a multi-session compatibility plan retains the existing endpoint merge reads;
- current-turn overlays and RisuAI display-name sanitation still use the existing
  local formatter over the reused records.

The old compact session-state shape was not reused. The backend contract was
expanded first so that no content disappeared during the transition.

## Items deliberately retained

- Final injection budget assembly remains in JavaScript because supervisor
  output and actual RisuAI payload placement are only known after prepare-turn.
- Storyline, character, speech-style, and pending-thread formatting remains in
  the host adapter. These functions use current-turn overlays and RisuAI-local
  display-token rules that the backend cannot observe. The repeated Store/API
  reads are removed; host-only rendering is not treated as backend duplication.
- World-rule inherited-scope reads and formatting remain because the prepared
  session snapshot is session-only and cannot represent the active inherited
  location chain. Its `complete_sections.world_rules` value is deliberately
  false.
- Effective Input final parity remains in JavaScript because only the plugin can
  inspect the payload actually returned to RisuAI.

Presentation ViewModels are tracked separately from turn orchestration.
Dashboard ViewModel migration was completed on 2026-07-11:

- `POST /dashboard/view-model` receives RisuAI-only observations from the host
  adapter and returns `dashboard.viewmodel.v1`.
- The backend owns status normalization, warning/failure severity, card and row
  composition, summary counts, first-turn display rules, persistence-lane
  status, duplicate-save presentation, and backend timing summaries.
- JavaScript supplies only observations that the backend cannot see directly,
  then localizes labels/detail codes and renders the returned cards.
- Dashboard ViewModel failure does not fall back to the old JavaScript
  calculator. The UI shows a small unavailable card instead, preventing two
  competing sources of dashboard truth.
- Turn processing, persistence, rollback, session migration, and canonical
  output selection are not changed by this presentation endpoint.

Timeline and Explorer ViewModel migration was completed on 2026-07-12:

- `POST /presentation/view-model` receives host-only observations and returns
  `presentation.viewmodel.v1`.
- The backend owns timeline item normalization, pending-item visibility,
  turn grouping, representative previews, source counts, session action
  eligibility, empty/load-more states, Explorer tab totals, session rows, and
  active-session sync state.
- JavaScript owns localization, DOM markup, scroll restoration, button event
  wiring, and edit forms. It no longer groups timeline turns or calculates
  Explorer navigation totals.
- Timeline detail responses receive the same backend-produced item ViewModel,
  so detail and list rendering use one normalization contract.
- Existing Store reads, edit/delete requests, session copy/move/connect, and
  rollback behavior are unchanged.

Rollback and session-routing range ownership was moved on 2026-07-12:

- JavaScript still observes RisuAI message changes, assistant removal, visible
  completed pairs, and local routing baselines. It does not own the final DB
  deletion range.
- `POST /rollback/decision` calculates whether deletion is allowed, the final
  `from_turn`, copied/moved/attached-session protection, and the effective
  completed-turn count.
- An allowed decision issues a short-lived one-use token bound to the session
  and exact turn. The active JavaScript rollback path sends that token to the
  existing delete route; invalid, mismatched, expired, or reused tokens are
  rejected before Store mutation.
- `POST /session-routing/turn-resolution` owns the `existing backend turns +
  post-route local turns` calculation used by active-chat recovery, backfill,
  and tail reconciliation.
- The existing migration preview/complete ledger remains the authority for
  whether source rows exist, whether the target is empty, and which rows are
  copied. This change does not broaden migration or deletion scope.
- Legacy direct rollback API compatibility remains available for older plugin
  builds, but the current JavaScript path does not execute DB rollback without
  a backend decision token.

## Completion criteria

- Backend runtime-support fields are covered by a direct contract test.
- JavaScript source-contract tests verify backend-first and endpoint-fallback
  ordering.
- JavaScript syntax validation passes.
- `go test ./... -count=1` passes.
- No database deletion, merge, migration, or turn-range behavior changes.
