# Archive Center 3.0 JS/Backend Duplicate Computation Removal

## Meaning

This work does not delete duplicate database values. It removes repeated work
where Go has already returned a complete result and JavaScript performs the same
read or formatting operation again during the same turn.

The transition rule is always:

1. Prefer a complete, version-compatible backend result.
2. Keep the existing JavaScript path as a fallback for an absent, old, partial,
   degraded, or unreachable backend.
3. Remove the fallback only after release-level compatibility evidence exists.

## Ownership boundary

JavaScript continues to own:

- observing RisuAI hooks, active chat, payloads, and displayed output;
- applying an injection to the actual RisuAI message array;
- confirming the final displayed assistant string;
- local UI rendering and localization;
- compatibility fallback when the backend result is incomplete.

The backend owns:

- MariaDB and ChromaDB reads;
- selection, ranking, and bounded recall results;
- rendered store-backed memory sections;
- persistence and migration range decisions;
- stable error codes and backend timing.

## First completed slice: active-state read reuse

`/prepare-turn` already builds `continuity_pack.items` with complete active-state
records containing `state_type`, `turn_index`, and `content`. JavaScript then
called `/active-states/{session}` again before orchestration.

JavaScript now:

- extracts complete active-state records from the prepared continuity pack;
- records `prepare_turn_bundle` as the source;
- skips the second endpoint read when the bundle is complete;
- falls back to the existing endpoint when the bundle is absent or declares
  rows without supplying usable content.

The compact `session_state.active_states` list is deliberately not used because
it omits content. Using it would reduce prompt quality.

## Items deliberately retained

- Final injection budget assembly remains in JavaScript because supervisor
  output and actual RisuAI payload placement are only known after prepare-turn.
- Storyline and world-rule local formatting remains because current-turn
  supervisor overlays may not exist in the earlier backend packet.
- Character and pending-thread compatibility reads remain until backend packets
  carry equivalent cross-session compatibility evidence and complete fields.
- Effective Input final parity remains in JavaScript because only the plugin can
  inspect the payload actually returned to RisuAI.

## Next safe candidates

1. Add complete compatibility-aware character and pending-thread records to the
   prepare response, then prefer them before individual reads.
2. Return an explicit backend display model for timeline grouping while leaving
   localized labels and DOM rendering in JavaScript.
3. Return stable dashboard severity and reason codes so JavaScript does not
   reinterpret backend status text with regular expressions.

Each candidate requires output-parity tests before its JavaScript fallback can
be removed.
