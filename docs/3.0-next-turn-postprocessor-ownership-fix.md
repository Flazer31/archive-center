# Next-turn postprocessor ownership fix

## Symptom

The first turn was stored, but a genuine second turn could roll back turn 1 and be stored again as turn 1.

## Evidence

The session audit recorded a rollback whose request source was `postprocessor_final_replace`. This showed that persistence and the retired Milvus code were not the failing path. The client had classified a genuine next-turn request as a secondary postprocessor request.

## Cause

While the active chat still ended with the previous assistant message, the ownership decision checked the stale post-output context before checking whether the payload user text matched the fresh input-hook text.

## Fix

The ownership order is now:

1. verify against the active chat user tail;
2. verify against the fresh input-hook user text;
3. only then classify an unmatched request as a post-output secondary request.

No module name, prompt phrase, or output tag is used for this distinction.

## Regression coverage

- A genuine next turn is accepted when its payload matches the input-hook text even if the active chat still ends with the previous assistant output.
- A real postprocessor request that does not match fresh user input remains classified as `post_output_secondary_request`.

## Existing missed-turn cold-start recovery

Session normalization can only repair raw turns that the plugin supplies from the live RisuAI chat. A normalization audit showed `repair_entry_count: 0` and an empty target turn list even though the visible chat had later turns. The active-chat resolver had been inferring the current chat through the character object.

The resolver now prefers RisuAI's official asynchronous `getChatFromIndex(characterIndex, chatIndex)` API. The existing `getCharacter()` and database snapshot paths remain as compatibility fallbacks. Cold start, active-chat backfill, and deletion reconciliation all share this resolver.
