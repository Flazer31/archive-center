# Archive Center 3.0 원작 자료 라이브러리와 세션 연결 설계

상태: 스키마·Store CRUD·파일 입력·평론가 추출·자동 검수·생성 자료 관리 UI, 세션 binding preview/apply/update/unlink·수동 시점 선택 UI, 작품 전용 벡터 색인·정확 검색 진단 구현. 시점 hard filter recall preview, prepare-turn 주입, 세션 이전 통합 검증은 미완료

## 목표

원작이 있는 작품의 인물, 장소, 물품, 사건, 세계 규칙을 채팅 세션과
분리해 한 번만 저장하고 여러 세션에서 재사용한다. 각 세션은 필요한
작품을 명시적으로 연결하고, 현재 원작 시점 이전의 자료만 검색한다.

이 기능은 기존 장기 기억을 대체하지 않는다.

- 세션 기억은 해당 플레이에서 실제로 일어난 일을 기록한다.
- 원작 자료는 작품에 원래 존재하는 참고 사실을 제공한다.
- 둘이 충돌하면 현재 유저 입력과 세션에서 확정된 사실이 우선한다.
- 원작 자료는 세션의 사건이 실제로 일어났다고 주장하는 canonical writer가
  아니다.

## 필수 경계

1. 작품 자료를 기존 `chat_session_id` 행으로 복사하지 않는다.
2. 작품은 안정적인 `work_id`로 식별하고 RisuAI의 유동적인 `char_*` 값에
   종속시키지 않는다.
3. 작품 연결은 기본적으로 수동이다. 제목 추측만으로 자동 연결하지 않는다.
4. MariaDB가 원문 출처, 정리된 사실, 시점과 연결 상태의 권위 저장소다.
5. ChromaDB는 후보 검색만 담당하고 MariaDB 확인을 통과한 자료만 주입한다.
6. 세션 삭제와 턴 롤백은 작품 자료를 삭제하지 않는다.
7. 작품 자료 삭제는 연결된 세션을 명시적으로 해제한 뒤에만 허용한다.
8. 미래 사건과 미공개 정체는 현재 시점 경계를 넘어서 주입하지 않는다.
9. 세션에서 원작과 달라진 사실은 원작 자료보다 우선한다.
10. 웹 자료는 출처와 수집 시각을 남기고 광고, 메뉴, 댓글, 탐색 링크는
    사실 자료로 저장하지 않는다.

## 저장 공간

같은 MariaDB 안에 별도 테이블을 두되 기존 세션 기억 테이블과 소유권을
분리한다. 운영상 별도 데이터베이스 서버를 하나 더 요구하지 않는다.

키와 삭제 규칙:

- 공개 키는 UUID를 사용하고 내부 API도 같은 키를 사용한다.
- 작품 하위 자료는 `work_id` 외래키를 가진다.
- 현재 DB에는 별도 session master 테이블이 없으므로 binding의
  `chat_session_id`에는 인덱스를 두되 존재하지 않는 session 외래키를
  만들지 않는다.
- 일반 세션 삭제는 binding/runtime만 지운다. 명시적인 전체 DB 초기화는
  작품 라이브러리까지 지울 수 있지만 대상 수를 preview에 따로 표시한다.

### `reference_works`

작품 라이브러리의 최상위 항목이다.

| 필드 | 의미 |
|---|---|
| `work_id` | 작품의 단일 기본 키, UUID |
| `title` | 대표 작품명 |
| `work_type` | novel, anime, manga, game, mixed, custom |
| `default_language` | 자료 기본 언어 |
| `status` | draft, indexing, ready, disabled |
| `metadata_json` | 제작자, 발매 정보 등 선택 정보 |
| `created_at`, `updated_at` | 관리 시각 |

### `reference_continuities`

소설판, 애니메이션판, 리메이크, 사용자 대체 설정처럼 서로 섞으면 안 되는
연속성을 분리한다.

| 필드 | 의미 |
|---|---|
| `continuity_id` | 연속성 기본 키, UUID |
| `work_id` | 소속 작품 |
| `continuity_key` | unique key, 예: `light_novel_main` |
| `label` | UI 표시명 |
| `parent_continuity_id` | 공통 원전이 있을 때 선택 사용 |
| `status` | active, archived |

### `reference_documents`

사용자가 넣은 TXT, Markdown, JSON, 웹 문서 등 출처 단위 기록이다.

| 필드 | 의미 |
|---|---|
| `document_id` | 문서 기본 키, UUID |
| `work_id`, `continuity_id` | 작품과 연속성 |
| `source_type` | manual_text, file, web |
| `source_uri` | 파일명 또는 URL |
| `content_hash` | 중복 입력 방지 |
| `raw_retention` | full, local_only, derived_only |
| `raw_text` | 정책상 허용될 때만 저장 |
| `import_status` | pending, parsed, reviewed, rejected |
| `provenance_json` | 제목, 수집 시각, 입력자, 라이선스 메모 |

동일 작품·연속성·`content_hash`는 기본적으로 한 번만 입력한다.
unique 기준은 `(work_id, continuity_id, content_hash)`다.

### `reference_timeline_nodes`

권·화 번호만 강제하지 않고 사건 기준 시점을 표현한다. 시작 이전,
중간 시작, 완결 후, 분기와 크로스오버를 다룰 수 있어야 한다.

| 필드 | 의미 |
|---|---|
| `node_id` | 시점 노드 기본 키, UUID |
| `work_id`, `continuity_id` | 작품과 연속성 |
| `node_key` | 예: `before_bell_joins`, `minotaur_rescue` |
| `label` | 사람이 읽는 사건명 |
| `ordinal` | 같은 연속성 안의 기본 정렬값 |
| `parent_node_id` | 분기점 또는 포함 관계 |
| `branch_key` | 본편, 대체 분기 등을 구분 |
| `kind` | prelude, event, arc, post_canon, branch |

`ordinal`은 검색 경계를 위한 내부 순서이며 작품 속 달력 시간으로 간주하지
않는다. 복잡한 분기는 `parent_node_id`와 `branch_key`로 제한한다.
`(continuity_id, branch_key, node_key)`는 unique로 두고
`(continuity_id, branch_key, ordinal)`을 조회 인덱스로 둔다.

### `reference_entities`와 `reference_entity_aliases`

인물, 장소, 물품, 세력의 대표 이름과 별칭을 작품 단위로 관리한다. 기존
세션 Entity와 자동 병합하지 않는다. 주입 시 이름 대조에만 사용하며,
세션의 Alias Repair가 작품 자료를 변경해서는 안 된다.

`reference_entities`는 UUID `entity_id`를 기본 키로 사용한다. 별칭은
`(work_id, continuity_id, normalized_alias)` 범위에서 찾되, 같은 별칭이
여러 인물을 가리키면 자동으로 하나를 선택하지 않는다.

### `reference_claims`

검색과 주입의 최소 사실 단위다.

| 필드 | 의미 |
|---|---|
| `claim_id` | claim 기본 키, UUID |
| `work_id`, `continuity_id` | 작품과 연속성 |
| `document_id` | 출처 문서 |
| `claim_type` | character, relationship, world_rule, event, item, location |
| `subject_entity_id` | 선택적 주체 |
| `claim_text` | 짧게 정리된 사실 |
| `evidence_excerpt` | 필요한 최소 근거만 저장 |
| `temporal_scope` | timeless, bounded, event |
| `valid_from_node_id`, `valid_to_node_id` | 사실이 성립하는 원작 구간 |
| `reveal_from_node_id` | 스포일러 없이 알려져도 되는 최초 지점 |
| `knowledge_scope` | public_world, entity_scoped, narrator_only |
| `branch_key` | 적용 분기 |
| `confidence` | 추출 신뢰도 |
| `review_status` | pending, approved, rejected |

`valid_from`은 사실의 발생 시점이고 `reveal_from`은 독자나 등장인물에게
그 사실을 공개해도 되는 시점이다. 둘을 하나로 합치지 않는다.

`entity_scoped` claim은 별도 `reference_claim_knowers` 행으로 그 사실을
아는 원작 인물을 지정한다. 현재 장면 인물이 knower가 아니면 그 인물의
대사·생각 근거로 쓰지 않는다. `narrator_only`는 설정 검증에는 사용할 수
있지만 등장인물 지식으로 주입하지 않는다.

주요 조회 인덱스는
`(work_id, continuity_id, review_status, branch_key, reveal_from_node_id)`와
`(subject_entity_id, claim_type)`다.

### `session_reference_bindings`

채팅 세션과 작품 라이브러리를 연결한다.

| 필드 | 의미 |
|---|---|
| `binding_id` | 연결 기본 키, UUID |
| `chat_session_id` | RisuAI 세션 ID |
| `work_id`, `continuity_id` | 연결 작품과 연속성 |
| `binding_role` | primary, crossover, reference_only |
| `enabled` | 현재 검색 사용 여부 |
| `anchor_mode` | manual, assisted |
| `current_node_id` | 현재 원작 시점 |
| `reveal_ceiling_node_id` | 주입 가능한 최대 공개 지점 |
| `divergence_node_id` | 원작에서 갈라진 시점, 선택 |
| `future_policy` | block, preview_only |
| `priority` | 여러 작품 연결 시 정렬값 |
| `revision` | 동시 수정 충돌 방지 |

unique 기준은 `(chat_session_id, work_id, continuity_id)`다. 하나의 세션에
여러 작품을 연결할 수 있으므로 크로스오버를 별도 예외 처리하지 않는다.

### `session_reference_runtime`

매 턴 쌓는 기억이 아니라 연결마다 한 행만 갱신하는 현재 상태다.

- 마지막으로 확인한 원작 시점 후보
- 후보를 만든 세션 턴과 근거
- 사용자가 확정했는지 여부
- 마지막 검색 claim ID와 진단값

자동 추론은 이 행의 후보만 갱신한다. 사용자가 확정한 `current_node_id`와
`reveal_ceiling_node_id`를 자동으로 앞으로 이동시키지 않는다.

## 세션 연결 흐름

1. 사용자가 원작 라이브러리에서 작품과 연속성을 선택한다.
2. 현재 시작 지점 또는 "시작 이전/모름"을 선택한다.
3. 백엔드는 연결 preview에서 작품·연속성 소속, 승인된 시점 노드,
   분기 일치, 현재·공개 상한·분기 시점의 순서와 기존 연결 충돌을 검증한다.
4. 사용자가 적용하면 `session_reference_bindings` 한 행을 만든다.
5. 허용 claim 수와 차단될 미래 claim 수는 recall preview 단계에서 추가한다.
6. `prepare-turn` 검색 후보 적용은 recall hard filter 검증 후 별도로 활성화한다.

작품명 자동 감지는 연결 추천까지만 가능하다. 사용자의 적용 없이 DB 연결을
생성하지 않는다.

## 시점과 분기 규칙

### 일반 진행

- `reveal_from_node_id`가 현재 공개 상한보다 뒤면 hard block한다.
- `valid_from`/`valid_to` 밖의 상태 사실은 현재 상태로 주입하지 않는다.
- 과거 사건 설명은 현재 시점 이전이면 참고할 수 있다.
- 현재 지점이 `unknown`이면 `temporal_scope=timeless`이고 공개 검수가 끝난
  claim만 허용한다. 사건, 관계 변화, 정체 공개 자료는 시점 확정 전까지
  차단한다.

### 원작에서 벗어난 세션

`divergence_node_id` 이후에는 원작의 후속 사건을 미래 사실로 강제하지
않는다. 인물의 기존 성격, 이미 알려진 세계 규칙, 과거 사건만 참고하고
세션에서 새로 확정된 관계·생사·소유 상태가 우선한다.

### 크로스오버

작품마다 독립적인 현재 지점과 공개 상한을 가진다. 한 작품의 시점 값을
다른 작품의 `ordinal`과 비교하지 않는다. 검색 결과에는 항상 `work_id`와
작품명을 붙여 출처가 섞이지 않게 한다.

## 검색과 주입

검색 순서:

1. 현재 세션의 enabled binding을 읽는다.
2. 작품·연속성·branch·공개 상한으로 MariaDB 후보 범위를 제한한다.
3. 현재 유저 입력, 직전 등장인물, 현재 장소의 별칭으로 검색문을 만든다.
4. 별도 ChromaDB 컬렉션 `archive_center_reference_vectors`에서 작품별 후보를
   찾는다.
5. MariaDB claim을 다시 읽어 시점, 공개, review 상태를 재검증한다.
6. 현재 장면 인물과 claim knower 범위를 대조한다.
7. ChromaDB가 반환한 원시 거리와 순서를 보존한 뒤, 현재 등장인물과 원작
   시점은 후속 recall 단계의 별도 적격성 필터로 적용한다.
8. 현재 세션 사실과 충돌하는 claim은 제외하거나 참고 경고로 낮춘다.

작품 벡터는 기존 세션 벡터와 별도 컬렉션에 둔다. 따라서 세션 삭제,
롤백, 복사 정리에서 작품 벡터가 함께 삭제되지 않는다. 임베딩 모델 차원
변경도 두 컬렉션을 각각 명확하게 재색인한다.

Go에서는 일반 기억용 `VectorStore`와 원작용 `VectorStore` 인스턴스를 서로
다른 필드와 컬렉션으로 연다. 일반 세션 삭제·롤백 경로는 원작용 인스턴스를
호출하지 않으며, 두 컬렉션 이름이 같으면 설정 검증에서 시작을 차단한다.

MariaDB CRUD도 기존 필수 `Store`에 추가하지 않고 선택적
`ReferenceLibraryStore`로 둔다. MariaDB authority 구현만 이 계약을
제공하며 noop이나 쓰기 금지 store에서는 기능이 명시적으로 비활성화된다.
따라서 원작 기능이 아직 설정되지 않은 기존 런타임의 Store 계약은 바뀌지
않는다.

주입 위치는 기존 Archive Center 보조 컨텍스트의 마지막에 작은
`[Original Work Reference]` 블록으로 둔다. 우선순위는 다음과 같다.

1. 현재 유저 입력과 최근 대화
2. 현재 세션에서 확정된 사실과 기억
3. 시점 검증을 통과한 원작 참고 자료

블록에는 "세션에서 달라진 사실이 원작보다 우선"이라는 규칙을 포함한다.
원작 자료만으로 세션의 새 사건을 만들거나 미래 전개를 강제하지 않는다.

## 복사·연결·이동·삭제와의 관계

- 세션 복사: binding과 runtime 현재값을 대상 세션에 복사하고 원작 본문은
  참조만 공유한다.
- 세션 이동: migration preview에 binding 수를 표시하고 대상에 이전한다.
- 세션 연결: 대상에 없는 binding만 preview 후 추가한다. 같은 작품의 다른
  연속성이 이미 있으면 자동 병합하지 않는다.
- 턴 롤백: 원작 자료는 건드리지 않는다. 자동 시점 후보가 삭제된 턴에서
  나왔다면 runtime 후보만 이전 근거로 되돌린다.
- 세션 삭제: binding과 runtime만 삭제한다.
- 작품 삭제: 활성 binding이 하나라도 있으면 차단하고 연결 해제 preview를
  먼저 요구한다.

## 입력과 검수

첫 구현은 사용자가 직접 넣는 TXT, Markdown, JSON만 지원한다. LLM 평론가는
문서를 claim, entity, timeline 후보로 정리하지만 바로 검색 가능 상태로
승격하지 않는다.

사용자가 직접 모든 필드를 작성하게 만들지 않는다. 입력 방식은 세 단계로
나눈다.

### 빠른 추가

사용자는 작품명, 판본 또는 "잘 모름", 파일만 넣는다. 평론가가 다음을
자동으로 제안한다.

- 대표 인물·장소·물품과 별칭
- 작품의 큰 사건 순서와 사람이 읽는 시점 이름
- 인물 성격, 관계, 세계 규칙, 중요 사건 claim
- 비밀 정보의 공개 시점과 아는 인물 후보
- 중복 문서와 기존 claim 충돌

사용자는 전체 표를 편집하지 않고 "이 자료 묶음 적용" 또는 충돌 항목만
선택한다.

### 안내형 추가

평론가가 확신하지 못한 경우에도 자유 서술을 요구하지 않는다. UI가
"어느 사건보다 전/후인가", "소설판인가 애니판인가", "이 사실을 누가
아는가"처럼 2~3개의 선택지를 제시한다. 답을 모르면 `unknown`으로 남길
수 있다.

### 고급 수동 편집

정확한 자료를 직접 관리하려는 사용자만 entity, timeline, claim 필드를
수정한다. 기본 설치에서는 접어둔 고급 기능으로 제공한다.

자동 추출 신뢰도가 높아도 원문을 곧바로 주입하지 않는다. 후보 묶음은
검수 대기 상태로 저장하고, 사용자가 일괄 승인하거나 보수적 자동 승인
정책을 명시적으로 켜야 `approved`가 된다. 시점 판정에 실패한 사건성
claim은 `timeless`로 낮추지 않고 차단 상태로 남긴다.

1. 원문 입력과 hash 기록
2. 구조 분석과 광고·메뉴·링크 목록 제거
3. claim/entity/timeline 후보 생성
4. 중복과 충돌 표시
5. 사용자 검수 또는 보수적 자동 승인
6. 승인 claim만 벡터화

자료는 작품 전체를 한 번에 완성할 필요가 없다. 현재 플레이와 관련된
인물·에피소드 문서부터 점진적으로 추가할 수 있으며, 새 자료가 들어와도
기존 세션 연결과 사용자가 확정한 공개 상한은 자동 변경하지 않는다.

웹 URL 직접 수집은 이 흐름이 안정화된 뒤 추가한다. 웹 수집 시에는 본문
영역 판별, robots/접근 실패, 출처 표시, 원문 보관 정책을 별도 처리한다.

## 구현 순서

### 2026-07-12 구현 상태: 파일 자동 추출과 생성 자료 보관함

다음 첫 입력 흐름을 구현했다.

1. 설정의 `원작 자료` 화면에서 작품과 이야기 흐름을 선택한다.
2. TXT, Markdown, JSON 파일을 읽어 원문과 SHA-256을 MariaDB에 저장한다.
3. 평론가 설정을 사용해 백그라운드 `reference_extract` 작업을 시작한다.
4. 문서를 조각 단위로 분석해 연표, 인물·장소·물품, 사실·설정 후보를 만든다.
5. 평론가가 근거와 품질을 자동 판정해 사용할 자료를 확정한다.
6. 확정된 자료는 UI의 기본 `생성된 자료` 탭에 즉시 표시한다. 사용자가 항목을 하나씩 승인하는 절차는 두지 않는다.

구현 API:

- `GET/POST /reference-works`
- `GET/POST /reference-works/{work_id}/continuities`
- `POST /reference-works/{work_id}/documents`
- `POST /reference-works/{work_id}/documents/{document_id}/extract`
- `GET /reference-jobs/{job_id}`
- `GET /reference-works/{work_id}/library`
- `GET /reference-works/{work_id}/review-candidates`
- `POST /reference-works/{work_id}/review`

보호 규칙:

- 같은 문서 hash의 재입력은 기존 문서를 반환한다.
- 같은 문서의 실행 중 추출 작업은 새 작업을 만들지 않고 기존 `job_id`를 반환한다.
- 검수 변경은 URL의 `work_id`에 속한 후보에만 적용한다.
- 재추출은 이미 승인하거나 제외한 후보의 검수 결정을 `pending`으로 되돌리지 않는다.
- 조각이 달라도 기존 연표 키와 인물 별칭을 읽어 claim의 주체와 시점을 연결한다.
- 평론가 API 키는 작업 상태 응답에 저장하지 않는다.
- 승인 후보의 벡터화와 세션 연결은 구현했다. prepare-turn 주입은 시점 hard filter 검증 전까지 수행하지 않는다.

자동 검수 정책:

- 신규 파일 추출이 끝나면 같은 평론가 설정으로 후보 묶음을 한 번 더 검수한다.
- 원문 근거가 분명한 작중 사실은 `approved`로 자동 전환한다.
- 메뉴·각주·광고·편집 흔적·제작진 정보·현실 모티브·중복 후보는 `rejected`로 자동 전환한다.
- 원문에 추정·추측 표현이 있거나 시점·충돌이 불명확하면 `pending`으로 남긴다.
- 시대 제목은 원문이 별도 조직을 명시하지 않는 한 세력 entity로 승인하지 않는다.
- 같은 사건이 연표와 사건 claim으로 중복되면 연표를 우선하고 중복 claim은 제외한다.
- 불확실한 후보는 내부 `pending` 상태로 남겨 자동 생성 자료에서 제외한다.
- 자동 결정은 원작 자료 검수 상태만 바꾸며 일반 채팅 기억이나 세션 DB에는 쓰지 않는다.
- 검수 결과에는 `review_source`, `review_reason`, `reviewed_at`을 기록한다.
- 일반 UI는 `생성된 자료`와 `자료 추가`만 제공한다. 내부 판정 기록은 오류 추적과 향후 관리 도구를 위해 보존한다.
- `생성된 자료`는 승인된 항목만 반환하는 전용 API를 사용하고 `전체`, `연표`, `인물·장소·물품`, `사실·설정`으로 나누어 보여준다.
- 감사 열이 추가되기 전에 처리된 기존 후보는 상태를 유지하고 `기존 검수 · 상세 기록 없음`으로 표시한다.

### 2026-07-12 구현 상태: 자료 정확성 관리

AI 주입 전에 원작 자료를 정정할 수 있도록 다음 관리 계약을 추가했다.

- 생성 자료 화면을 `인물`, `장소`, `물품`, `세력·조직`, `연표`, `세계 규칙`, `관계·사건`, `삭제됨`으로 분리한다.
- 각 항목에 원본 문서명과 원문 근거를 표시한다. 원문 전체는 목록 API에서 다시 반환하지 않는다.
- 수정한 항목은 `review_source=user_edit`와 함께 승인 상태로 저장한다. 재추출 upsert는 승인·제외 항목을 덮어쓰지 않으므로 사용자 수정이 잠긴다.
- 일반 삭제는 행을 제거하지 않고 `user_excluded`로 전환한다. 검색·주입에서는 제외하고 `삭제됨` 화면에서 복구할 수 있다.
- 복구는 `user_restore`로 기록하며 사용자 확정 자료로 취급한다.
- 같은 작품·흐름·종류·주체·정규화 문장이 완전히 같은 claim은 동일한 안정 ID를 사용한다. 후보 순서나 문서 조각 번호가 바뀌어도 새 중복 행을 만들지 않는다.
- 기존의 완전 동일 중복은 보관함 조회에서 한 항목만 사용한다. 이름이나 문장이 비슷하다는 이유만으로 합치지 않는다.
- 추출 claim에는 같은 속성을 가리키는 `fact_key`를 기록한다. 같은 fact key에 서로 다른 내용이 승인되면 충돌 확인 대상으로 표시한다.
- 자동 검수에는 기존 승인 자료를 읽기 전용 비교 기준으로 함께 제공한다. 기존 사실과 충돌하는 신규 후보는 자동 승인하지 않고 `pending`으로 남긴다.
- 중복·충돌·사용자 확정 개수와 해당 항목 표시는 보관함 진단에 포함한다.
- 평론가가 각 조각에서 생성한 `ordinal`을 그대로 시간 순서로 간주하지 않는다. 승인 연표 전체와 원문 근거를 평론가가 다시 비교해 명시된 연도·시대·부모 사건·선후 관계를 기준으로 정렬한다.
- 숫자 연도만 파싱하는 규칙은 사용하지 않는다. 판타지 연호와 상대적 사건 순서를 지원하고, 근거가 불명확한 항목은 `unresolved_ids`로 남긴다.
- 확정된 순서는 10, 20, 30 간격의 내부 순서값으로 저장하고 UI에는 1, 2, 3의 연속 표시 순서를 제공한다.
- 자동 추출·검수 완료 후 시간순 정리를 실행한다. 기존 자료는 `연표 시간순 정리` 명령으로 같은 작업을 실행할 수 있다.
- 시간순 정리는 사건 내용을 삭제하거나 합치지 않는다. 같은 사건 후보와 상충하는 연대는 `duplicate_groups`, `conflict_groups`로 반환하고 별도 진단 대상으로 유지한다.

관리 API:

- `GET /reference-works/{work_id}/library`
- `PATCH /reference-works/{work_id}/library/{kind}/{item_id}`
- `DELETE /reference-works/{work_id}/library/{kind}/{item_id}`: 복구 가능한 제외만 수행
- `POST /reference-works/{work_id}/library/timeline/normalize`

영구 삭제는 관계 주체, 연표 범위, 세션 anchor를 끊을 수 있으므로 일반 UI에 제공하지 않는다.

1. 스키마와 Store interface, MariaDB CRUD, migration을 추가한다. 완료.
2. 작품·연속성·문서·claim 파일 입력 및 검수 API와 테스트를 만든다. 완료.
3. 세션 binding preview/apply/update/unlink와 삭제 보호를 만든다. 완료.
4. timeline node와 수동 current/reveal/divergence anchor UI를 만든다. 완료. assisted 후보 갱신은 recall 단계에서 구현한다.

9. 복사·연결·이동·롤백·세션 삭제 통합 테스트를 추가한다.

## 필수 자동 테스트

- 같은 문서 hash와 멱등성 키의 동시 입력이 한 문서만 생성하는지 확인
- 연결되지 않은 `work_id` 검색 결과가 0건인지 확인
- unknown anchor에서 timeless claim만 통과하는지 확인
- reveal ceiling 뒤의 고득점 벡터가 MariaDB hydration에서 차단되는지 확인
- entity-scoped claim이 허용되지 않은 등장인물에게 주입되지 않는지 확인
- 같은 세션에 두 작품을 연결해도 각 ordinal과 branch를 독립 계산하는지 확인
- 세션 복사·이동·연결 preview와 apply가 binding/runtime만 다루는지 확인
- 턴 롤백과 세션 삭제 후 작품 행과 작품 벡터 수가 변하지 않는지 확인
- 활성 binding이 있는 작품 삭제가 mutation 전에 차단되는지 확인
- 기능 off와 binding 0개에서 기존 prepare-turn 응답이 동일한지 확인
- 서로 다른 두 질의가 실제 ChromaDB 결과 순서를 바꾸는지 확인
- 검색 응답에 순위 기반 고정 유사도나 재정렬 점수가 생기지 않는지 확인
- 재색인 실패 시 기존 원작 벡터가 삭제되지 않는지 확인

## 출구 기준

- 원작 자료 행과 벡터는 세션 삭제·롤백으로 삭제되지 않는다.
- 연결되지 않은 작품은 어떤 경우에도 검색되지 않는다.
- 현재 공개 상한 이후 claim은 벡터 점수가 높아도 주입되지 않는다.
- 시점이 unknown이면 timeless가 아닌 claim은 주입되지 않는다.
- 인물별 비밀 claim은 허용된 knower 외의 인물 지식으로 사용되지 않는다.
- 세션에서 확정된 변화가 원작의 과거 상태로 되돌아가지 않는다.
- 크로스오버 작품별 시점이 서로 독립적으로 유지된다.
- 동일 문서 재입력은 hash와 멱등성 키로 중복 생성되지 않는다.
- 모든 주입 문장에 작품, claim, 문서 출처를 역추적할 수 있다.
- 기능이 꺼져 있거나 연결이 없으면 기존 2.5/3.0 기억 흐름과 결과가 같다.
