# Archive Center 3.0 Runtime Boundary and Compatibility Plan

작성일: 2026-07-11

상태: 조사 및 작업 계획만 기록. 이 문서 작성 단계에서는 런타임 코드, 설치 스크립트, CI, 패키지를 변경하지 않는다.

현재 상태 정정(2026-07-11): 이 문서의 회귀 기준선은 정상 기능을 다시 구현하라는 뜻이 아니다. 현재 구현 여부와 실제 남은 작업 판정은 `3.0-verified-current-state-audit.md`를 우선한다. 특히 RisuAI 요청 타입은 기존 `model` 전용 게이트를 유지하며, `model` 요청을 프롬프트 내용으로 다시 보조 요청으로 분류하는 분석기를 백엔드에 그대로 확대 이전하지 않는다.

2026-07-12 갱신: 대형 Go 파일 분리, 중복 세션 조회 축소, 대시보드·타임라인·탐색 ViewModel 이전, 삭제·리롤·복사·연결·이동 기준의 백엔드 소유권 정리가 완료됐다. 아래 순서는 설계 당시 기록으로 보존하며 현재 잔여 작업은 `3.0-verified-current-state-audit.md`를 따른다. 원작 자료 신규 기능은 `3.0-reference-work-library-and-session-binding-design.md`를 기준으로 한다.

## 목적

3.0 작업에서 JS 로직을 백엔드로 옮기다가 기존에 정상 동작하던 입력 인식, 기억 주입, 저장, 삭제, 복사, 연결, 이동, 번역, 후처리 흐름이 깨지는 일을 막는다.

또한 현재 발견된 ChromaDB 버전/API 경로 불일치와 Go 빌드 도구 버전 차이를 구현 전에 하나의 호환성 기준으로 고정한다.

## 절대 원칙

1. 이동 전과 이동 후의 결과가 같다는 회귀 증거 없이 기존 JS 판정 코드를 제거하지 않는다.
2. JS는 RisuAI 관찰 및 적용 어댑터로 남는다.
3. 백엔드는 판정, 계획, DB 작업, 표시 모델을 소유한다.
4. 기존 멱등성 보호를 삭제하고 새 체계로 교체하지 않는다. 새 턴 계약은 기존 동일 턴/동일 pair 보호를 보강한다.
5. 백엔드 장애 시 RisuAI의 사용자 입력과 실제 출력은 보존한다.
6. ChromaDB 장애는 MariaDB 원문 저장 성공을 실패로 바꾸지 않는다.
7. 버전/API 불일치 때문에 ChromaDB 컬렉션이나 MariaDB 데이터를 자동 삭제하지 않는다.

## RisuAI와 백엔드 경계

### JS에 남아야 하는 역할

- `beforeRequest`, `afterRequest`, `input`, `output`, `display`, `process` 훅 관찰
- 현재 캐릭터, 현재 채팅, active chat tail, poller 후보 수집
- RisuAI 메시지 배열에 백엔드 주입 계획 적용
- 실제 화면에 반환된 최종 문자열 확인
- 채팅 전환, 재생성, 삭제, 중단, 복사/연결/이동 UI 이벤트 관찰
- 설정 UI, 대시보드 DOM, 타임라인/탐색 DOM 렌더링
- 백엔드 실패 시 원문 통과와 로컬 상태 표시

### 백엔드가 소유할 역할

- 요청 소유권 판정
- 입력 후보와 출력 후보의 canonical 선택
- 기억 검색, 현재 상태 대조, 주입 블록/예산 계획
- DB 저장 원문 판정
- 삭제/롤백/복사/연결/이동의 DB 기준 범위 계산
- 안정적인 오류 코드, 상세 원인, 재시도 가능 여부
- 대시보드/타임라인/탐색 ViewModel

### 함께 처리해야 하는 역할

| 항목 | JS | 백엔드 |
|---|---|---|
| 요청 소유권 | RisuAI 관측값 수집 | 판정과 근거 반환 |
| 입력 출처 | payload/active chat/poller 후보 전달 | canonical 후보 선택 |
| 번역/후처리 | 실제 후보와 최종 표시값 전달 | 비교 및 저장 후보 판정 |
| canonical 출력 | 실제 표시값 일치 확인 | 저장 원문 판정 |
| 오류 | 연결 상태와 로컬 실패 관찰 | 코드, 상세, severity, retryable 반환 |
| UI | DOM 및 로컬 상태 렌더링 | ViewModel 반환 |

## 이동 전 회귀 기준선

각 fixture는 이전 JS 결과와 새 백엔드 shadow 결과를 동시에 기록한다.

- 요청 소유권 판정과 판정 근거
- 선택된 사용자 입력과 출처
- 선택된 assistant 출력과 출처
- 주입 블록, 순서, 글자 예산, 제외 이유
- canonical 저장 원문
- raw/derived/vector 저장 결과
- 삭제/롤백/복사/연결/이동 dry-run 범위
- 오류 코드, severity, retryable, 사용자 표시 분류

필수 시나리오:

- 일반 메인 채팅
- user-role 지시문이 섞인 프롬프트
- 선택형 입력
- Gemini context caching
- native `afterRequest`
- poller 복구
- native와 poller가 모두 관찰한 동일 출력
- 스트리밍 중단 및 늦은 `afterRequest`
- 리롤
- GigaTrans 정상/깨진 태그
- 일반 후처리 모듈
- `/del`, `/cut`, `/del N`, 반복 1턴 삭제
- 복사/연결/이동 후 첫 턴과 첫 턴 삭제
- 백엔드, MariaDB, ChromaDB 각각의 단독 장애

기존 결과와 다르면 새 경로를 기본값으로 승격하지 않는다.

## 턴 상관관계 계약

기존 complete-turn의 동일 턴/동일 user+assistant pair 멱등성 보호는 유지한다.

추가 후보:

```text
contract_version
client_build
request_id
turn_token
client_chat_id
client_turn_revision
tail_fingerprint
selected_input_source
selected_output_source
backend_capabilities
```

`turn_token`은 prepare와 complete를 연결하고 native/poller 중복 관찰을 같은 턴으로 묶는 보조 키다. 기존 원문 pair 비교와 turn index 보호를 대체하지 않는다.

상태는 한곳에서 전부 소유하지 않는다.

```text
JS trace: observed -> injected -> output_observed
Backend ledger: prepared -> completed | aborted | reconciled
```

JS가 complete 요청에서 관찰 상태와 `turn_token`을 전달해 두 흐름을 연결한다.

## 실패 처리 계약

| 실패 | 사용자 응답 | 저장 판정 | 재시도 |
|---|---|---|---|
| prepare timeout | 원래 입력으로 통과 | 아직 없음 | 다음 턴 또는 명시 재호출 |
| complete timeout | 화면 출력 유지 | 미확정 | 같은 turn token으로 가능 |
| MariaDB 실패 | 화면 출력 유지, 저장 실패 경고 | 실패 | 가능 |
| ChromaDB 실패 | 화면 출력과 MariaDB 저장 유지 | 부분 성공 | vector만 재시도 |
| critic 실패 | raw 저장 유지 | 부분 성공 | derived만 재시도 |
| canonical 판정 실패 | 실제 표시 원문 보존 | 보수적 저장/대기 | 중복 방지 후 가능 |

오류 응답은 최소한 `code`, `severity`, `retryable`, `component`, `detail`을 가진다. JS는 이 값을 다국어 사용자 문구로 변환한다.

## 비밀정보 설정 경로

현재 JS는 prepare/complete 요청마다 embedding API key를 전달한다. 이를 단순히 env 전용으로 바꾸면 UI 설정 우선과 원격 백엔드 사용이 깨질 수 있다.

후보 구조:

1. UI에서 credential을 입력하거나 변경한다.
2. 별도 설정 API가 백엔드에 credential을 등록한다.
3. 백엔드는 credential handle 또는 설정 revision을 반환한다.
4. 턴 요청은 handle/provider/model만 전달한다.
5. env는 fallback으로 유지한다.
6. 원격 HTTP 연결에는 credential 전송 경고를 표시하고 HTTPS/Tailscale 보호 경로를 권장한다.

credential은 응답, trace, dashboard, 로그에 원문으로 반환하지 않는다.

## 승격 중단 조건

다음 중 하나라도 발생하면 새 경로를 기본값으로 올리지 않는다.

- 기존 정상 입력이 auxiliary request로 오분류됨
- 실제 사용자 입력 대신 user-role 프롬프트가 저장됨
- native/poller 중복 저장
- 후처리 초안과 최종본이 함께 저장됨
- 연결/이동/복사 후 기준 턴 손실
- 1턴 삭제가 DB 전체 삭제로 확대됨
- MariaDB 성공이 ChromaDB 실패 때문에 전체 실패로 표시됨
- ChromaDB 버전/API 불일치를 ready로 표시함
- 기존 DB 경로를 놓치거나 새 DB를 조용히 생성함

## 문서 정리 대상

- `//@api 3.0`은 RisuAI 플러그인 API 버전이므로 유지한다.
- Archive Center 제품 3.0 후보 문서에는 제품 후보임을 명시한다.
- RC4 설정 제목에 남은 RC3 문구는 실제 구현 단계에서 정리한다.
- Milvus 문서는 역사/미래 참고로 유지하되 현행 기본값처럼 읽히지 않게 분류한다.
- ChromaDB 현행 기준은 MariaDB canonical truth + ChromaDB retrieval lane + `/api/v2`다.

## 저장 요청 멱등성과 타임아웃 복구 (2026-07-11 확인)

현재 JS 구현은 중복 요청 가능성이 실제로 있다.

- `bridgeFetch()`의 `R.nativeFetch` 경로는 `Promise.race()`로 응답 대기만 끝낸다.
- 이 경로에는 `AbortController`가 없으며 timeout용 `setTimeout`도 성공/실패 후 해제하지 않는다.
- `/complete-turn`은 `bridgeFetchWithRetry(..., 1)`로 같은 저장 POST를 즉시 한 번 더 보낼 수 있다.
- 같은 payload는 write-ahead retry queue에도 먼저 들어가고, drain 시 다시 `/complete-turn`으로 전송된다.
- 현재 본문 쌍 조회와 queue dedupe는 이미 완료된 순차 중복에는 유효하지만, 첫 요청이 아직 처리 중일 때 들어오는 동시 중복을 보장하지 않는다.

따라서 저장성 요청은 다음 계약으로 바꾼다.

- [ ] `/complete-turn`, rollback, migration complete/reindex 등 저장성 POST는 응답 timeout만으로 자동 재전송하지 않음
- [ ] JS가 `request_id`와 안정적인 `idempotency_key`를 최초 요청 때 한 번 생성하고 queue 복구까지 그대로 유지
- [ ] 백엔드는 idempotency key에 unique 제약을 두고 `processing/completed/failed` 상태와 최종 응답을 저장
- [ ] 같은 key가 처리 중이면 새 처리를 시작하지 않고 `processing`을 반환
- [ ] 같은 key가 완료됐으면 저장된 최종 응답을 반환
- [ ] timeout 후 JS는 새 POST보다 먼저 처리 상태 조회 API를 호출
- [ ] 상태를 확인할 수 없는 경우에만 같은 idempotency key로 복구 요청
- [ ] timeout timer는 모든 성공/실패 경로에서 `clearTimeout` 처리
- [ ] prepare/search처럼 비저장 요청과 complete/rollback/migration 같은 저장 요청의 retry 정책을 분리
- [ ] 느린 첫 요청, timeout 직후 queue drain, 브라우저 재연결, 늦은 응답을 동시에 재현하는 동시성 테스트 추가
