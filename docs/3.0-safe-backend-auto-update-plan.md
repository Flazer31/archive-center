# Archive Center 3.0 Safe Backend Auto Update

## Feasibility

Automatic backend updates are implementable. The current backend already:

- checks the latest GitHub release through `GET /update/check`;
- chooses an OS/architecture package;
- downloads through `POST /update/download`;
- verifies the package against the release `SHA256SUMS` entry;
- stages the verified archive without applying it.

The missing part is a safe apply and rollback helper. A running backend must not
overwrite itself directly.

## Required architecture

### Backend

- Check release metadata and compatibility.
- Download to a staging directory.
- Verify the expected SHA-256 before any install action.
- Write an apply manifest containing current version, target version, asset,
  digest, executable path, service name, and health-check URL.
- Expose update state: available, downloading, verified, applying, restarted,
  rolled back, or failed.

### External updater helper

- Start only after an explicit user Apply action or explicit unattended-update
  opt-in.
- Wait for the backend process to stop.
- Extract into a new version directory, never over the active directory.
- Preserve user environment files, prompts, and runtime configuration according
  to the package migration contract.
- Switch the active executable/version pointer atomically.
- Restart the Windows process, systemd service, or Termux launcher.
- Call `/ready`; if readiness fails, restore the previous pointer and restart
  the previous version.

### Plugin UI

- Show current and latest versions.
- Separate Download from Apply.
- Explain when restart is required.
- Show verification, restart, readiness, and rollback results in plain language.
- Never report success from `/health` alone; successful apply requires `/ready`.

## Data preservation rules

The updater must never package, replace, move, or delete:

- MariaDB data directories or database contents;
- ChromaDB data directories or collections;
- Windows `.runtime` data;
- `.env.full.local` or protected environment files;
- user-created prompt overrides and explicit local settings.

Backend binaries and bundled static files are versioned. Data and configuration
remain outside the version directory and are referenced by stable paths.

## Security rules

- Accept assets only from the configured GitHub repository and selected release.
- Require an exact `SHA256SUMS` match before extraction.
- Reject path traversal and symbolic-link escape during archive extraction.
- Do not execute scripts contained inside the downloaded archive.
- Record the installed asset name, digest, release tag, and previous version.
- Add release-signature verification later if signed release metadata is adopted;
  SHA-256 alone proves file integrity only relative to the trusted sums file.

## OS application strategy

| OS | Apply mechanism |
| --- | --- |
| Windows | Small updater executable or PowerShell helper launched outside the backend process, then restart through the existing launcher/service contract |
| Ubuntu/Linux | Versioned release directory plus atomic active pointer and systemd restart |
| Termux | Versioned release directory plus launcher restart without touching MariaDB/Chroma data |

3. Implement Linux/systemd helper and rollback test.
4. Implement Windows helper and locked-executable replacement test.
5. Implement Termux launcher integration.

The 2.5 behavior remains download-and-verify only. Automatic application begins
with the 3.0 updater helper after its rollback tests pass.

## 2026-07-16 implementation checkpoint

The 3.0 Windows package now establishes the updater needed for later releases.
The bounded behavior is intentionally simple:

- the settings UI explicitly downloads a release and schedules it for the next
  package start;
- the Go backend derives the digest from the selected release and rejects a
  client override, a non-HTTPS asset URL, and a release that is not newer;
- the external Go helper verifies the archive and managed-file manifest before
  changing package files, then records a rollback journal;
- the Windows launcher runs the helper before local services, requires the
  updated backend's main `/ready`, and commits or restores the previous managed
  files;
- `.runtime`, `.updates`, local env files, database/vector paths, caches, and
  secret/data file types are not managed update targets;
- v1 rejects additions or changes to managed migration SQL and
  `mariadb-schema.exe`, so file rollback cannot leave the database on a schema
  that the restored backend did not approve;
- a missing helper or rejected pending archive does not block the normal
  package when updater state proves that no package mutation occurred;
- an unprovable or partially applied state stops startup instead of running a
  mixed package;
- `client_only` does not apply a pending package because it cannot perform the
  local backend readiness gate.

The v1 managed-file contract does not delete files. If a future package omits a
previously managed path, the update is rejected before mutation. A deletion
contract requires a separately reviewed v2 design.

Windows next-start application is the only launcher integration in this
checkpoint. Linux/systemd and Termux application adapters, signed release
metadata, and real upgrade E2E runs with preserved MariaDB/Chroma data remain
future release gates. The updater does not change Archive Center memory,
storage, reference-work, or prompt-assembly policy.
