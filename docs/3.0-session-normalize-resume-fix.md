# 3.0 Session Normalize Resume Fix

## Observed Problem

Session Normalize enabled every force-rebuild option on every run. A retry
therefore sent already successful turns back to the Critic, repeated raw world
rule audits and hierarchy generation, and forced vector reindexing.

At roughly 20 seconds per Critic call, 1,000 turns can require more than five
hours. Retrying an 80-percent-successful run replayed the successful 80 percent
instead of concentrating on the failed remainder.

## Corrected Default

- Existing canonical memories mark their turns as completed.
- A retry selects only turns that still lack derived memory.
- World-rule, episode, and hierarchy backfills fill missing ranges without
  replacing completed ranges.
- Vector reindex is not forced by Session Normalize.
- If MariaDB candidates and ChromaDB documents are already consistent, reindex
  returns `vector_index_already_current` without embedding or upsert calls.
- Full forced rebuilding remains possible only through explicit force options
  with `resume_existing=false`.

## Safety Boundary

Session Normalize remains non-destructive. It does not delete raw chat logs,
trim turns, perform rollback, or clear user databases.

## HypaMemory Handoff

HypaMemory import and Session Normalize are separate operations. After a
HypaMemory import job has completed and its generated memories are visible in
Archive Center, HypaMemory can be disabled. Archive Center then uses its own
MariaDB and ChromaDB records during normal turns. Session Normalize is only
needed when raw chat turns or derived artifacts are missing; it is not required
for every HypaMemory handoff.

## Regression Evidence

- A 0-to-120-turn rescan resumes without duplicate Critic calls.
- Session Normalize metadata rerun makes zero Critic calls for completed turns.
- An already-current vector index performs no duplicate upsert.
- Full Go tests and core RisuAI adapter regressions pass.
