# Archive Center 3.0 Turn Correlation Audit

## Purpose

Verify that one RisuAI main output is stored once, in the session that started
the request, when native `afterRequest`, active-chat recovery, late callbacks,
rerolls, postprocessors, and chat navigation overlap.

This audit does not introduce a new turn token. Existing request and content
identity protections are retained unless a concrete uncovered case requires a
smaller fix.

## Current protections

| Case | Protection | Result |
| --- | --- | --- |
| Native `afterRequest` | complete-turn key contains session, turn, user hash, and assistant hash | Covered |
| Native callback and poller see the same output | watcher is stopped by native callback; recovered output records a grace marker | Covered |
| Native callback arrives after poller recovery | recent recovery marker ignores the late native callback | Covered |
| Same request is still processing | backend idempotency ledger joins the in-flight request and returns the same response | Covered |
| HTTP timeout after backend accepted the request | client queries `/complete-turn/request-status` instead of blindly repeating the write | Covered |
| Same user/assistant pair is replayed at another turn | backend exact-pair guard skips duplicate raw and derived writes | Covered |
| Postprocessor replays a final output | latest-turn replacement plus exact-pair guard keeps one canonical pair | Covered |
| Reroll reuses an old callback | active-chat pair alignment and stale assistant replay guard prevent the old output from becoming the new turn | Covered |
| User switches chats while output is finishing | established origin CID is now retained; only a provisional session may resolve to a fresh active CID | Fixed in this audit |
| RisuAI turn is deleted and immediately regenerated | raw Risu message indexes select the logical turn; conflicting DB tail is rolled back before replacement | Covered |
| First local turn after copy, move, or attach is deleted | backend routing baseline maps local turn 1 after the imported DB tail and protects all imported turns | Covered |
| Active chat becomes completely empty after deleting its first turn | persisted rollback ledger verifies full-tail removal; an unverified empty chat cannot delete DB rows | Covered |
| Character uses a selectable alternate greeting | active chat `fmIndex` selects `alternateGreetings` or `firstMessage` and stores it as turn 0 | Covered |

## Session routing correction

Before this audit, `resolveAfterRequestWriteSessionId()` preferred any different
active CID. That behavior was useful when a provisional session ID became a real
CID during the first request, but it could also redirect an already established
CID to a different chat opened by the user.

The corrected rule is:

- If the request started from a provisional, non-CID identity and RisuAI now
  exposes a real CID, use the real CID.
- If the request already started from a CID, preserve that CID through
  completion even when another chat becomes active.

This change does not alter turn numbering, deletion ranges, copy/connect/move
baselines, canonical output selection, or database schemas.

## RisuAI message index correction

The active chat adapter now preserves the original RisuAI message array index.
For a normal chat, user indexes `0, 2, 4` and assistant indexes `1, 3, 5` map to
Archive Center turns `1, 2, 3`. Content equality is no longer allowed to push a
regenerated logical turn to the next backend number.

Copy, move, and attach sessions do not use the local number directly. The
backend applies the saved routing baseline. For example, seven imported turns
plus local user/assistant indexes `0/1` resolve to backend turn 8. Deleting that
pair resolves the rollback floor to turn 8, preserving imported turns 1 through
7.

If the deletion watcher misses the short interval before the regenerated
output, persistence performs the same reconciliation before complete-turn: it
reads the canonical pair at the structurally selected turn, requests the
existing backend rollback decision for a conflict, then stores the new pair at
the same turn.

node --check "Archive Center.js"

## Remaining live-only edge case

RisuAI host callbacks cannot be fully simulated by the Go test suite. A release
candidate should still receive one live check where a long response is started,
the user opens another existing chat before completion, and the finished pair is
confirmed only in the originating session. This is a release check, not an
unimplemented code path.
