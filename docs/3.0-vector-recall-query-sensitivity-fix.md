# 3.0 벡터 기억 검색의 질의 민감도 수정

## 확인된 2.5 결함

- ChromaDB가 반환한 `distances`를 읽고도 검색 결과 계약에 보존하지 않았다.
- 실제 유사도 대신 `1.0 - rank * 0.05`를 `similarity`와 `rerank_score`로 표시했다.
- 벡터 결과가 하나라도 준비되면 어휘 관련성, 최근 기억, 중요 기억 보완 경로를 모두 중단했다.
- 이 조합 때문에 질의와 무관한 초기 기억이 반환되어도 정상 벡터 검색처럼 표시되고 주입을 독점할 수 있었다.

## 3.0 수정 계약

1. ChromaDB 검색은 거리와 저장 벡터를 함께 반환받는다.
2. 백엔드는 현재 질의 벡터와 저장 벡터의 코사인 유사도를 직접 계산한다.
3. ChromaDB가 반환한 순서를 그대로 신뢰하지 않고 실제 코사인 유사도로 다시 정렬한다.
4. 요청한 수보다 넓은 후보를 받은 뒤 재정렬하고 최종 `top_k`만 선택한다.
5. 실제 점수가 없거나 최소 관련도보다 낮은 후보는 기억과 직접 근거 및 세계 규칙 주입에서 제외한다.
6. 벡터 검색이 실패하거나 적격 후보가 부족하면 남은 예산을 어휘 관련성 및 최근 기억 경로가 채운다.
7. `similarity`는 실제 계산값만 의미하며 순번 기반 합성 점수를 사용하지 않는다.
8. `ann_candidate_snapshot`은 canonical 문서 앞부분을 후보처럼 꾸미지 않고 실제 벡터 검색 결과만 표시한다.
9. 아직 질의가 적용되지 않은 인덱스 문서는 `similarity=null`, `query_matched=false`로 표시한다.

## 보호 경계

- MariaDB가 원문과 파생 자료의 기준 데이터라는 기존 원칙은 유지한다.
- ChromaDB는 후보 선택과 순위 가속만 담당한다.
- 벡터 검색 실패가 턴 저장 실패로 번지지 않는다.
- 기억 선택 변경은 삭제, 리롤, 복사, 연결, 이동의 턴 계산과 무관하다.

## 회귀 기준

- 서로 다른 질의가 서로 다른 저장 벡터를 1위로 선택해야 한다.
- ChromaDB가 고정된 잘못된 순서로 후보를 반환해도 백엔드 재정렬 결과는 질의 벡터를 따라야 한다.
- 낮은 관련도의 오래된 기억은 주입되지 않고, 현재 질의와 겹치는 어휘 기억이 대신 선택되어야 한다.
- 실제 점수가 없는 후보는 벡터 정상 결과로 처리하지 않아야 한다.
