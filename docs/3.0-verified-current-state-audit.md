# Archive Center 3.0 현재 상태 코드 감사

작성일: 2026-07-11

## 2026-07-11 안정화 1~5 재검증 결과

이번 변경은 3.0 백엔드 이전 자체가 아니라, 이전 작업을 시작하기 전에 필요한 기존 동작 안정화다.

1. **전체 Go 테스트 복구**
   - 고정된 개인 PC 경로를 임시 fixture로 교체했다.
   - 과거 JS 문자열만 찾던 검사를 현재 공개 동작 계약에 맞췄다.
   - 기억 재생성 응답은 `code`와 사람이 읽는 `detail`을 분리했다.
   - `go test ./... -count=1` 전체 통과를 출구 기준으로 사용한다.
2. **저장성 POST 중복 방지**
   - `/turns`, `/turns/complete`, `/effective-inputs`, `/complete-turn`은 JS에서 자동 즉시 재시도하지 않는다.
   - `/complete-turn`은 같은 세션·턴·입출력으로 안정적인 멱등성 키를 만든다.
   - 백엔드는 처리 중 요청을 합치고, JS 대기열은 `/complete-turn/request-status` 확인 후에만 복구한다.
   - 상태 조회는 단순 완료와 HTTP 성공을 구분한다.
3. **Go 1.26.5 이상 및 CI**
   - `go.mod` 최소 버전을 1.26.5로 올렸다.
   - 패키징은 Go 1.26.5 이상을 허용한다.
   - CI는 JS 문법, gofmt, vet, 전체 테스트, 빌드, govulncheck를 실행한다.
4. **ChromaDB 1.5.9 및 API v2 통일**
   - Windows/POSIX/Termux 설치 기준을 Python 패키지 `chromadb==1.5.9`와 `/api/v2`로 통일했다.
   - 백엔드는 시작 전에 heartbeat와 컬렉션 API를 검사한다.
   - `/ready`도 실행 중 Chroma 연결을 다시 검사한다.
   - CI에서 실제 ChromaDB 1.5.9를 실행해 생성·upsert·검색·삭제를 검증한다.
5. **`model` 내부 재분류 축소**
   - RisuAI의 non-model 타입 제외는 그대로 유지한다.
   - `model` 요청은 프롬프트 마커나 active-tail 불일치만으로 차단하지 않는다.
   - 이미 출력된 초안을 후처리 결과로 교체하는 `post_output_secondary_request` 보호는 유지한다.

주의: 삭제·리롤·복사·연결·이동, 번역·후처리 canonical 선택, 기존 기억 주입 순서는 이번 작업에서 재작성하지 않았다.

기준:

- 기준 커밋: `26818fe` (`release-2.5.0-rc4` 병합)
- 검사 대상: 기준 커밋 위의 현재 `source` 미커밋 변경 포함
- 사용자 실사용 확인과 코드/자동 테스트 확인을 구분한다.
- 정상 기능을 3.0 신규 구현 과제로 다시 적지 않는다.

## 확인된 실제 결함

### 1. 저장성 POST의 처리 중 중복 가능성

`R.nativeFetch` 경로는 `Promise.race`로 응답 대기만 끝내며 실제 요청을 취소하지 않는다. timeout timer도 해제하지 않는다. `/complete-turn`은 write-ahead queue와 retry 경로가 있어 첫 요청이 아직 처리 중일 때 같은 POST가 다시 들어갈 수 있다.

기존 pair 중복 방지는 완료된 저장을 다시 넣는 상황에는 유효하지만 동시 처리 중 중복을 완전히 막지는 않는다.

필요 작업:

- 저장성 POST 자동 즉시 재시도 금지
- 안정적인 idempotency key
- backend 처리 상태 ledger
- timeout 후 상태 조회
- timer 정리

### 2. `model` 내부 내용 기반 재분류 잔존

최상위 타입 게이트는 `model`만 허용하지만 이후 다음 조건으로 다시 차단할 수 있다.

- payload user tail 없음
- post-output secondary request 판정
- auxiliary marker
- payload/active tail mismatch

이는 과거 `auxiliary_module_request` 오판과 같은 계열의 위험이다. 3.0에서 이 분석기를 더 크게 백엔드로 옮기는 것은 현재 목표와 맞지 않는다. 먼저 RisuAI가 준 non-model type 제외를 기준으로 하고, `model` 안에서는 저장할 최종 입력/출력 선택에 필요한 최소 검증만 남기는 방향을 검토한다.

### 3. 전체 Go 테스트 21개 실패

`go test ./... -count=1` 실측 결과:

- 고정된 `M:\risulongmemory\...` 경로: 1개
- 과거 JS 문자열/함수명/버전 marker 검사: 19개
- `/explorer/memories/regenerate`의 `code` 계약 불일치: 1개

내부 패키지 테스트 `go test ./internal/... -count=1`은 통과한다.

### 4. ChromaDB 배포 기준 혼재

- backend 기본과 관리형 실행기: `/api/v2`
- 루트 및 일부 live env: `/api/v1`
- Windows RC4: ChromaDB 1.5.9
- POSIX/Termux: `chromadb>=0.5,<1.0`
- live test: 무고정 설치

이는 실제 배포 호환성 작업이다.

### 5. Go/CI 기준 미완성

- 현재 번들 Go: 1.26.4
- 목표 보안 patch: 1.26.5
- GitHub Actions workflow 없음
- `govulncheck` 로컬 도구 없음

## 실제 3.0 남은 작업

우선순위 순서:

1. 깨진 전체 테스트 21개를 현재 공개 동작 계약으로 복구
2. 저장성 POST 처리 중 중복 방지 계약 구현
3. Go 1.26.5 도구와 자동 CI/release gate 고정
4. ChromaDB 1.5.9 및 `/api/v2` 관리형 배포 기준 통일
5. `model` 내부 내용 기반 요청 재분류를 최소화하고 오판 회귀 테스트 추가
6. 대형 Go 파일을 같은 패키지 안에서 동작 변경 없이 분리
7. 이미 백엔드가 반환하는 주입/표시 모델과 중복되는 JS 계산만 제거
8. 대시보드·타임라인 표시 데이터 조립을 backend ViewModel로 점진 이전
9. 삭제·복사·연결·이동 기능은 유지하고 중복된 JS 기준 계산만 backend owner로 축소
10. 현재 상태·인물 인식 보정의 장기 실사용 검증
11. 원작 자료 전용 DB, 작품 연결, 시점 제한 검색을 별도 신규 기능으로 설계
12. OS별 설치·업데이트 및 기존 DB 유지 통합 검증

이 항목들은 구조 이동 시 회귀 테스트로만 사용한다.
