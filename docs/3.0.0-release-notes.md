# Archive Center 3.0.0 릴리스 노트

작성일: 2026-07-16
비교 기준: GitHub `main`의 Archive Center 2.5.0 RC4 배포선
(`26818fe`) → Archive Center 3.0.0

## 릴리스 방향

3.0.0은 Archive Center의 기존 장기 기억 기능을 중심에 두고, 정책과
저장 책임을 Go 백엔드로 더 명확히 옮긴 안정화 릴리스다. 원작 DB는 메인
기억 기능을 대체하지 않는 선택형 보완 기능이며, 장애가 발생해도 기본
시작·ready·저장·장기 기억을 막지 않도록 별도 경로로 격리했다.

## 2.5.0과 달라진 점

| 영역 | 2.5.0 계열 | 3.0.0 |
| --- | --- | --- |
| 런타임 책임 | JavaScript에 일부 정책·표시 계산·턴 범위 계산이 함께 존재 | 대시보드·타임라인·탐색 ViewModel과 삭제·리롤·복사·연결·이동 범위, 원작 예산 정책을 Go 백엔드가 소유하도록 이동 |
| 턴과 세션 연결 | RisuAI 메시지와 DB 턴의 경계 사례가 삭제 직후 진행, 후처리, 세션 이동에서 흔들릴 수 있음 | 원본 RisuAI 메시지 인덱스와 origin session을 보존하고, 백엔드가 논리 턴과 migration baseline을 계산하도록 보강 |
| 중복 처리 | 같은 완료 요청이 겹치는 경우의 보호가 제한적 | `/complete-turn`에 안정 키, 처리 중 요청 합치기, 완료 응답 재사용을 추가. 현재 ledger는 프로세스 메모리와 10분 TTL 범위 |
| 벡터 검색 | 순위에서 만든 고정 유사도 표면과 사용하지 않는 Milvus 실험 코드가 남아 있었음 | 고정 점수 표면을 제거하고 실제 Chroma distance/cosine과 질의 민감도를 사용. 제품 경로를 MariaDB canonical truth + ChromaDB retrieval로 정리 |
| 관리형 런타임 | 이전 Go/Chroma 기준과 패키징 검증 사용 | Windows full package에서 Go 1.26.5 미만을 차단하고, 관리형 ChromaDB 1.5.9와 `/api/v2`를 기준으로 빌드·CI를 정리 |
| 원작 DB | 기본 제품 기능으로 제공되지 않음 | 작품·문서·entity·claim·timeline·session binding의 수집, 검수, 검색, 주입 기반을 선택 기능으로 추가 |
| 원작 주입 예산 | 해당 없음 | 메인 기억 예산과 별도인 `reference_injection_budget.v1` 사용. 보완 모드는 메인 기억 예산의 절반, 단독 모드는 같은 크기의 독립 예산을 사용하며 메인 기억 예산을 차감하지 않음 |
| 원작 모드와 실패 격리 | 해당 없음 | `supplement`, `primary`, `unknown`을 명시적으로 구분. 원작 전용 DB·검색·추출 장애는 기본 Archive Center의 시작·ready·저장·장기 기억을 막지 않고, 원작 사용 시에는 대시보드에 별도 경고를 표시 |
| 입력 확인 UI | 원작 자료가 장기 기억의 Assembled Auxiliary Context와 섞여 보일 수 있음 | Effective Input에서 원작 DB 주입을 별도 영역으로 분리해 메인 기억 조립 결과와 구분 |
| Windows 업데이트 | 패키지를 교체하는 수동 배포 중심 | GitHub release 확인, SHA-256 검증, 다음 시작 시 외부 helper 적용, `/ready` 확인 뒤 commit, 실패 시 rollback을 수행하는 안전 업데이트 경로 추가 |
| 업데이트 자산 | 전체 설치 패키지만 사용 | 전체 설치 패키지와 약 20MB 규모의 경량 Windows Update Package를 분리. 경량 자산은 MariaDB·ChromaDB runtime과 사용자 `.env*`, DB·vector·cache·secret/data를 포함하거나 덮어쓰지 않음 |

## 원작 DB의 현재 위치

- 원작 DB는 Archive Center의 메인 장기 기억 기능보다 우선하지 않는다.
- 보완 모드는 현재 대화와 직전 입력에서 부족한 원작 정보만 검색해 별도
  예산으로 지원한다.
- 단독 모드는 봇 자체에 원작 설정이 없는 경우를 위한 선택형 실험 기능이다.
- 승인된 원문 excerpt와 구조화 claim을 함께 보존하고, 검색 결과가 비었거나
  실패했을 때 무관한 내용을 강제로 주입하지 않는다.
- 인물·세력 관계의 정확도는 자료 품질과 승인 데이터에 좌우된다. 실제 사용자
  피드백을 통한 원문 보강, 관계 정정, 재색인이 계속 필요하다.

## 자동 업데이트 안전 계약

- 업데이트는 실행 중인 프로세스를 직접 덮어쓰지 않고 다음 시작에 적용한다.
- 다운로드한 ZIP과 manifest의 SHA-256을 확인한 뒤 staging한다.
- 관리 파일만 journal에 기록해 교체하며, 사용자 설정과 데이터 경로는 보호한다.
- 새 백엔드가 `/ready`를 통과하면 commit하고, 시작 또는 ready 검증이 실패하면
  이전 버전으로 자동 rollback한다.
- v1 업데이트는 파일 rollback과 DB schema가 어긋나는 일을 막기 위해 migration
  SQL 추가·변경과 `mariadb-schema.exe` 변경이 포함된 자산을 적용 전에 거부한다.

## 릴리스 검증 기록

- `go test ./... -count=1` 전체 통과.
- `node --check "Archive Center.js"` 통과.
- 관련 PowerShell 스크립트 구문 검사 통과.
- production updater CLI에서 apply/status/commit, 명시적 rollback, 변조 자산
  사전 거부를 확인.
- 3.0.0 전체 패키지 복제본에 3.0.1 경량 자산을 적용하는 Windows 로컬 E2E에서
  `.env.full.local`과 `.runtime` 보존을 확인.
- 의도적으로 잘못된 DB 설정을 사용한 다음 시작 검증에서 자동 rollback과
  3.0.0 재시작을 확인하고, 정상 설정에서는 3.0.1 commit을 확인.
- 위 업데이트 검증은 복제 패키지와 격리 포트에서 수행했으며 사용 중인 원본
  패키지와 DB는 수정하지 않았다.
- GitHub `v3.0.0` 공개 뒤 `/releases/latest`에서 stable release와 세 자산을
  확인하고, 공개 URL에서 경량 업데이트 ZIP과 체크섬을 다시 내려받아 로컬
  원본과 SHA-256이 일치함을 확인했다.

## 알려진 제한과 후속 검증

- 자동 적용 updater는 현재 Windows 패키지에만 구현됐다. Linux/systemd와
  Termux는 기존 수동 업데이트 경로를 사용해야 한다.
- POSIX 패키지의 Go 하한 진단과 toolchain manifest 기록은 Windows와 같은
  수준으로 정리되지 않았다.
- `/complete-turn` 멱등 ledger는 재시작 뒤 유지되는 영속 ledger가 아니다.
- 메인 최종 주입 예산과 주 요청 소유권의 일부 기존 계산은 아직 JavaScript에
  남아 있다. 3.0.0에서는 기존 기능 보호를 위해 추가 이동하지 않았다.
- 원작 DB의 장기 session binding, 복사·이동·삭제, 시점·공개 범위와 단독 모드
  품질은 라이브 사용자 피드백으로 계속 검증한다.
- 100턴 이상 장기 진행, 중단된 normalize 재개, 여러 후처리 조합은 추가
  실사용 회귀 검증이 필요하다.

## 배포 자산과 플랫폼 범위

- Windows x64 전체 설치 패키지와 별도 경량 자동 업데이트 패키지.
- Linux x64 및 Linux arm64 자동 설치 패키지.
- macOS Intel 및 macOS Apple Silicon 자동 설치 패키지.
- Termux arm64 자동 설치 패키지.
- POSIX 다섯 패키지는 Go 1.26.5로 목표 `GOOS/GOARCH`를 교차 빌드했고,
  필수 실행 파일·3.0.0 버전·비밀값 및 사용자 DB 부재를 검사했다. GitHub 공개
  뒤 다섯 ZIP을 다시 내려받아 통합 체크섬과 SHA-256이 일치함을 확인했다.
- Windows 외 플랫폼은 실제 대상 기기 실행 검증이 아니라 교차 빌드와 기존
  설치 스크립트 계약 검증 범위다. 다음 시작 외부 helper를 사용하는 자동 적용은
  Windows만 지원하며, Linux·macOS·Termux 패키지 제공 여부와는 별개다.

## 호환성 메모

3.0.0 자동 업데이트는 사용자 `.env*`, `.runtime`, MariaDB·ChromaDB 데이터,
vector·cache·secret/data 경로를 관리 업데이트 대상에서 제외한다. 전체 설치
패키지는 신규 설치 또는 수동 복구용이고, 기존 설치의 자동 업데이트에는
별도 Windows Update Package를 사용한다.
