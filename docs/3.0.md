# Archive Center 3.0 통합 문서

기준일: 2026-07-16
비교 기준: Archive Center 2.5.0 RC4 (`26818fe`) 이후 3.0 작업
정식 릴리스: [Archive Center 3.0.0](https://github.com/Flazer31/archive-center/releases/tag/v3.0.0)

## 1. 문서 목적과 판정 기준

이 문서는 Archive Center 3.0의 계획, 구현 기록, 개별 결함 수정, 원작 자료
라이브러리 설계, 자동 업데이트, 릴리스 노트를 하나로 합친 현재 기준 문서다.
3.0 구현 여부와 남은 작업을 판단할 때 이 문서를 우선한다.

모든 버전에 적용되는 RisuAI host와 백엔드의 영구 경계는
[`permanent-risu-host-backend-boundary.md`](permanent-risu-host-backend-boundary.md)를
따른다. 이 문서는 3.0 전용 문서가 아니므로 통합 대상에서 제외한다.

상태 의미:

- **완료**: 활성 코드와 자동 회귀 검증이 존재한다.
- **완료·라이브 확인 필요**: 자동 검증은 통과했지만 실제 RisuAI 장기 진행 확인이 남았다.
- **부분 완료**: 확인된 하위 계약만 구현됐다.
- **실험**: 기본 기능과 격리된 선택 기능이며 결과에 따라 축소·폐기할 수 있다.
- **보류**: 사용자가 후속 개선으로 미뤘다.
- **되돌림**: 위험 또는 회귀 때문에 활성 코드에서 제거했다.
- **미완료**: 설계 또는 일부 기반만 존재한다.

## 2. 3.0의 제품 방향

3.0은 Archive Center의 기존 장기 기억 기능을 중심에 두고 정책, 선택, 범위,
저장과 표시 계산을 Go 백엔드가 더 명확히 소유하도록 정리한 안정화 릴리스다.

핵심 원칙:

1. 메인 장기 기억의 시작, `ready`, 저장, 삭제, 복사·이동·연결을 우선 보호한다.
2. JavaScript는 RisuAI 관찰, 실제 payload 적용, 최종 표시 확인, DOM과 현지화를 담당한다.
3. Go는 정책, 검색, 예산, canonical 선택, 턴·DB 범위, 저장과 ViewModel을 담당한다.
4. MariaDB는 canonical truth이고 ChromaDB는 후보 검색용 vector backend다.
5. ChromaDB나 파생 생성 실패가 성공한 MariaDB 원문 저장을 실패로 바꾸지 않는다.
6. 원작 DB는 메인 기능을 대체하지 않는 선택형 보완 기능이다.
7. 원작 전용 장애는 메인 Archive Center의 시작·저장·장기 기억을 막지 않는다.
8. 고정 유사도, 고정 작품명, 고정 인물·세력 수, 무조건 성공 처리를 품질 보정으로 추가하지 않는다.
9. 기존 사용자 환경, MariaDB·ChromaDB 데이터, `.runtime`을 업데이트 자산으로 덮어쓰지 않는다.

## 3. 2.5.0과 3.0.0의 차이

| 영역 | 2.5.0 계열 | 3.0.0 |
| --- | --- | --- |
| 런타임 책임 | JavaScript에 일부 정책·표시·범위 계산이 함께 존재 | ViewModel, DB 범위, 원작 예산 등 백엔드 소유권 확대 |
| 턴 연결 | 삭제 직후 진행, 후처리, 세션 이동의 경계가 흔들릴 수 있음 | RisuAI 원본 메시지 인덱스와 origin session을 보존하고 백엔드가 논리 턴 계산 |
| 중복 처리 | 처리 중 동일 완료 요청 보호가 제한적 | `/complete-turn` 안정 키, 처리 중 요청 합치기, 완료 응답 재사용 |
| 벡터 검색 | 순위 기반 고정 유사도 표면과 Milvus 실험이 남음 | 실제 Chroma distance/cosine과 질의 민감도를 사용하고 Milvus 제거 |
| 관리형 기준 | Go/Chroma 버전과 패키징 기준이 혼재 | Go 1.26.5, 관리형 ChromaDB 1.5.9, `/api/v2` 기준 |
| 원작 자료 | 제품 기능으로 제공되지 않음 | 작품·문서·entity·claim·timeline·binding 수집, 검수, 검색, 주입 기반 추가 |
| 원작 예산 | 해당 없음 | 메인 기억과 별도인 `reference_injection_budget.v1` 추가 |
| 업데이트 | 수동 패키지 교체 중심 | 지원 패키지별 다음 시작 적용, `/ready`·`/version` 확인, 실패 rollback |
| 플랫폼 자산 | Windows, Linux, macOS, Termux | Windows x64, Linux x64/arm64, macOS Intel/Apple Silicon, Termux arm64에 같은 안전 적용 계약 사용 |

## 4. RisuAI host와 Go 백엔드 경계

### 4.1 JavaScript 소유 범위

- `input`, `beforeRequest`, `afterRequest`, streaming, display 등 RisuAI hook 관찰
- 현재 캐릭터·채팅·메시지 인덱스·active chat·표시 출력 관찰
- 백엔드가 만든 주입 계획을 실제 RisuAI 메시지 배열에 적용
- 실제 화면에 표시된 최종 assistant 문자열 확인
- 설정, 대시보드, 타임라인, 탐색 UI의 DOM·현지화·이벤트 연결
- 백엔드 미도달 시 원문 보존과 최소 transport queue

### 4.2 Go 백엔드 소유 범위

- 요청 소유권과 canonical 입력·출력 후보 선택
- 기억, 상태, 원작 자료의 검색·ranking·hard filter
- prompt/context 조립과 예산 계산
- 논리 턴, rollback, 삭제, 복사·이동·연결 baseline
- MariaDB·ChromaDB 저장, migration, 멱등성, retry 상태
- 대시보드·타임라인·탐색 ViewModel
- 오류 code, severity, retryable, component, detail

### 4.3 3.0에 의도적으로 남긴 JavaScript 작업

- 실제 RisuAI payload 배치와 최종 Effective Input parity
- 실제 표시 문자열 확정
- 현재 턴 overlay와 RisuAI-local 표시명 처리
- inherited world-rule scope처럼 백엔드 snapshot이 완전하지 않은 기존 호환 읽기
- 메인 최종 주입 예산의 `assembleInjectionWithBudget`
- 주 요청 소유권 일부의 `buildMainRequestOwnershipDecision`

마지막 두 항목은 migration debt다. 현재 치명 회귀가 재현되지 않아 3.0.0에서
추가 이전하지 않았으며, 새 JavaScript 정책을 덧붙이는 근거로 사용하지 않는다.

### 4.4 백엔드 packet 재사용

`session_state.runtime_support.v1`은 prepare-turn에서 이미 읽은 active state,
storyline, character state, pending thread를 반환한다. JavaScript는 같은 단일
세션에 대해 이 packet이 완전할 때 개별 endpoint를 다시 읽지 않는다.
오래됐거나 부분적인 packet, backend unreachable, 다중 세션 호환 merge에서는
기존 읽기 경로를 사용한다.

## 5. 핵심 런타임 구현

### 5.1 회귀 안전망

`scripts/test-core-regressions.ps1`과 전체 Go 테스트는 다음을 실제 사용자 DB와
비밀정보 없이 검사한다.

- 일반 입력과 출력 adapter
- raw/derived/vector 부분 성공
- 동일 턴 중복 저장 방지
- 삭제·리롤과 명시적 `/del`, `/cut` 보호
- 복사·이동·연결 baseline
- 후처리 최종 출력
- RisuAI 원본 인덱스와 논리 턴
- 원작 기능 off 또는 binding 0개일 때 기존 prepare-turn 보존

번들 Node를 찾아 JavaScript 검사를 조용히 건너뛰지 않도록 했다.

### 5.2 Go 파일 분리와 timing

대형 Go 파일을 별도 패키지로 재작성하지 않고 같은 패키지 안에서 책임별로
분리했다. HTTP route, JSON, Store interface, SQL과 schema는 구조 이동과 함께
바꾸지 않았다.

주요 분리 범위:

- turn ownership, input source, prepare, complete, idempotency
- injection planner와 canonical resolver
- session routing과 rollback decision
- dashboard, timeline, explorer ViewModel
- MariaDB 책임별 Store 파일

prepare-turn과 complete-turn은 단계별 backend timing을 반환한다. 확인된 병목은
주로 외부 critic LLM과 embedding 호출이며 파일 분리 자체로 정책을 바꾸지 않았다.

### 5.3 요청 세션과 canonical 출력

- 요청이 시작된 CID를 완료 시점까지 유지한다.
- 출력 중 다른 채팅을 열어도 저장 대상을 새 채팅으로 바꾸지 않는다.
- 후처리 모듈이 같은 턴의 초안을 최종본으로 교체하면 최종본 하나를 canonical 원문으로 저장한다.
- fresh input-hook과 payload user 입력이 일치하면 실제 다음 턴으로 우선 판정한다.
- active chat만으로 누락 턴을 광범위하게 복구한 시도는 오탐 위험 때문에 되돌렸다.

`/complete-turn`은 안정 키, 처리 중 동일 요청 합치기, 완료 응답 재사용을 제공한다.
현재 ledger는 프로세스 메모리와 10분 TTL 범위이며 재시작 뒤 유지되는 durable
ledger는 아니다.

### 5.4 RisuAI 인덱스와 DB 논리 턴

| RisuAI pair | DB 턴 |
| --- | ---: |
| user 0 / assistant 1 | 1 |
| user 2 / assistant 3 | 2 |
| user 4 / assistant 5 | 3 |

- JavaScript는 원본 zero-based index와 관찰 pair 순서를 전달한다.
- 백엔드는 일반적으로 `floor(user_index / 2) + 1`을 계산한다.
- 원본 index가 없을 때만 관찰 pair 순서를 대체 근거로 사용한다.
- 복사·이동·연결 세션은 migration ledger의 durable mapped-turn baseline을 우선한다.
- 기존 7턴을 가져온 세션의 첫 로컬 턴은 DB turn 8이며, 삭제도 turn 8부터만 수행한다.
- 같은 논리 턴의 화면 내용이 바뀌면 새 번호로 append하지 않고 backend rollback 후 같은 턴을 교체한다.
- 완전히 빈 active chat도 검증 ledger와 보호 baseline 없이 전체 삭제하지 않는다.
- 실제 선택된 `firstMessage` 또는 `alternateGreetings`는 turn 0으로 저장할 수 있다.

여러 시작 메시지가 조합되는 host에서 turn 0 전체를 원작 검색 질의로 사용하는
계약은 보류했다. 첫 원작 검색은 실제 첫 user 입력을 우선하며 의미 있는 검색
신호가 없으면 무관한 자료를 강제로 주입하지 않는다.

### 5.5 삭제·리롤·복사·이동·연결

- `POST /rollback/decision`이 허용 여부, `from_turn`, 보호 범위와 짧은 일회용 token을 반환한다.
- token은 session과 정확한 turn에 묶이며 만료·재사용·불일치 시 Store mutation 전에 거부된다.
- `POST /session-routing/turn-resolution`이 imported prefix와 로컬 턴을 결합한다.
- durable 기준은 `session_migrations`, `session_migration_row_map`, mapped `chat_logs`다.
- 사유는 `timeline_copy`, `timeline_migrate`, `timeline_attach`로 보존한다.
- 화면이 짧아졌다는 추정만으로 trim guard를 켜지 않는다. 실제 input hook에서 명시적 slash command를 관찰한 경우만 보호한다.

### 5.6 세션 정상화 재개

중단된 normalize는 이미 성공한 memory, evidence, world rule과 vector 단계를
처음부터 반복하지 않는다. stage/substage와 진행 상태를 보존하고 누락된 작업부터
재개한다. embedding 차원이나 필수 설정 오류는 대량 처리 전에 preflight에서
실패시킨다.

### 5.7 ViewModel

- `dashboard.viewmodel.v1`: 상태 정규화, severity, card/row, 요약, timing 표시
- `presentation.viewmodel.v1`: timeline grouping, preview, count, action eligibility, explorer totals

JavaScript는 localization, DOM, scroll, button wiring과 edit form만 담당한다.
ViewModel 실패는 저장이나 DB mutation으로 이어지지 않는다.

## 6. 현재 상태·인물 믿음·과거 기억

완료된 턴은 다음 세 종류로 분리한다.

1. `narrative_events`: 이번 턴 사건 이력
2. `state_claims`: 현재 객관적 사실
3. `belief_updates`: 특정 인물이 믿거나 오해하거나 의심하는 내용

모든 항목은 최신 완료 턴의 짧은 `evidence_excerpt`가 있어야 한다. 현재값 key는
`대상 종류 + 대상 + 상태 영역 + 인식 범위 + 인식 주체`이며, 같은 key에는
현재값 하나만 유지한다. 변경 시 이전값과 새값을 change history에 남기고 직접
근거 ID를 연결한다. 오래된 턴 재처리가 더 최신 현재값을 덮지 않는다.

주입 순서는 기존 기억을 우선 유지한다.

```text
기존 2.5 기억 주입
-> Chroma 후보와 MariaDB 현재값 대조
-> 필요한 경우에만 마지막에 [Continuity Correction]
```

보정문은 새 사건이나 미래 전개를 강제하지 않는다. 종료된 이전 상태를 현재
사실로 주장하는 기억을 이번 주입에서 제외할 수 있지만 DB 원문은 삭제하지 않는다.
인물별 사적 기억은 그 인물이 현재 입력, 직전 대화 또는 최신 장면에 있을 때만
주입한다. 턴 삭제 시 남아 있는 변경 이력의 최신값으로 현재 상태를 복원한다.

이 기능은 자동 테스트를 통과했지만 관계 반전, 생사 정정, 비밀 공개, 오해 해소가
누적되는 장기 세션 실사용 확인이 남아 있다.

## 7. 벡터 검색

### 7.1 ChromaDB 질의 민감도

- 순위에서 만든 `1.00, 0.95, 0.90...` 고정 유사도를 제거했다.
- Chroma의 raw distance와 저장 vector를 보존한다.
- 저장 vector가 있을 때 현재 query vector와 실제 cosine을 별도 계산한다.
- 실제 유사도로 적격 후보를 고르고 최소 관련도 미만을 제외한다.
- vector 후보가 부족하면 어휘 관련성과 최근 기억 경로가 남은 예산을 채운다.
- 진단의 `similarity`와 `rerank_score`는 실제 값이 있을 때만 의미를 가진다.

### 7.2 Milvus 정리

사용하지 않던 Milvus/Milvus Lite 코드, 설정, SDK, smoke command와 dependency를
제거했다. 제품 vector 구조는 MariaDB canonical truth와 ChromaDB retrieval로
고정했다. `milvus.db`, `milvus_data` 검사는 과거 산출물이 배포물에 섞이지 않게
하는 artifact guard로만 남았다.

## 8. 원작 자료 라이브러리

### 8.1 역할과 격리

원작 DB의 기본 역할은 원작 로어북이 있는 챗에서 부족한 고증만 보완하는 것이다.
로어북이 없는 경우를 위한 단독 모드는 선택형 실험이다. 원작 기능이 꺼졌거나
binding이 없으면 기존 Archive Center 응답과 저장 흐름을 바꾸지 않는다.

원작 전용 Store·Chroma·embedding·recall 실패는 메인 시작, `/ready`, 원문 저장,
장기 기억을 막지 않는다. 연결된 원작에 문제가 있을 때는 원작 전용 경고와
`failed/degraded/empty/ready` 상태를 대시보드에 표시한다. 실패를 성공 `ready`로
바꾸지 않는다.

### 8.2 예산 계약

`reference_injection_budget.v1`은 사용자가 설정한 기본 기억 상한을 `M`으로 사용한다.

| 모드 | 원작 총예산 `R` |
| --- | ---: |
| `supplement` | `floor(M / 2)` |
| `primary` | `M` |
| 비활성, binding 없음, unknown-only, `M=0` | `0` |

원작 예산은 메인 기억 예산에 가산되는 별도 레인이다. 메인 기억을 줄이거나
원작 자료와 한 예산으로 다시 섞지 않는다. 단독형의 Primary Canon Base는 `R`
안의 하위 예산이며 `primary_used + scene_used <= R`을 지킨다. 임의의 3000자
fallback은 제거했다.

새 빈 세션 첫 턴은 과거 세션 기억 fast boot를 계속 생략할 수 있지만, 연결된
원작 binding은 별도 원작 예산으로 검색한다.

### 8.3 모드

- `supplement`: 기본값. 현재 요청과 scene에 필요한 `partial/missing` 필드만 보완한다.
- `primary`: 사용자가 UI에서 직접 선택한 경우만 원작을 기본 설정원으로 사용한다.
- `unknown`: 잘못된 명시값. 자동 주입하지 않고 경고한다.

`binding_role`과 `reference_mode`는 별도다. 로어북 개수나 봇 형태로 단독 모드를
자동 추측하지 않는다. 여러 binding 중 primary가 있으면 예산 계약은 primary로
해석한다.

### 8.4 저장 모델

원작 자료는 같은 MariaDB 안의 별도 소유 테이블과 별도 Chroma 컬렉션을 사용한다.
기존 세션 기억 행으로 복사하지 않는다.

| 저장 영역 | 역할 |
| --- | --- |
| `reference_works` | 작품 UUID, 대표명, 종류, 언어, 상태 |
| `reference_continuities` | 판본, 리메이크, 분기 등 섞으면 안 되는 연속성 |
| `reference_documents` | TXT/Markdown/JSON 원문, source, hash, provenance |
| `reference_timeline_nodes` | 사건 기준 시점, branch, parent, ordinal |
| `reference_entities`, aliases | 인물·장소·물품·세력과 별칭 |
| `reference_claims` | 짧은 사실, excerpt, 시점, 공개 범위, 지식 범위, 검수 상태 |
| `session_reference_bindings` | session과 작품·연속성·모드·시점·공개 상한 연결 |
| `session_reference_runtime` | 연결별 assisted 시점 후보와 최근 recall 진단 |

핵심 경계:

- 작품은 안정적인 UUID `work_id`로 식별하고 RisuAI의 가변 `char_*`에 종속하지 않는다.
- 동일 작품·연속성·문서 hash는 한 번만 입력한다.
- `valid_from`과 `reveal_from`을 분리해 발생 시점과 공개 시점을 혼동하지 않는다.
- `entity_scoped` claim은 허용된 knower가 있어야 인물 지식으로 사용한다.
- `narrator_only`는 서술 고증에는 사용할 수 있지만 등장인물의 지식으로 전파하지 않는다.
- 작품별 branch와 ordinal은 크로스오버에서도 서로 비교하지 않는다.
- 세션의 새 사실과 명시적 사용자 divergence가 원작 자료보다 우선한다.

### 8.5 입력, 추출과 검수

현재 직접 지원하는 입력은 TXT, Markdown, JSON이다.

1. 작품과 continuity 선택
2. 파일 원문과 SHA-256 저장
3. `reference_extract` 작업으로 chunk 분석
4. entity, claim, timeline 후보 생성
5. 원문 근거, 중복, 충돌과 시점 자동 검수
6. 승인 자료만 전용 Chroma 컬렉션에 색인

특정 작품에서 가져온 추출 prompt 예시는 제거했다. 일반적인 원문 근거와
entity/claim 판정만 사용한다. excerpt가 실제 원문 chunk에 있을 때만
`evidence_grounded=true`로 저장하며, 없는 의역은 원문 근거로 주입하지 않는다.

신규 자료의 명확한 작중 사실은 승인할 수 있고 광고, 메뉴, 편집 흔적, 제작진
정보, 중복은 제외한다. 추정·충돌·불명확 시점은 pending으로 남긴다. 사용자는
승인 자료를 수정, 복구 가능한 제외, 복원할 수 있다. 사용자 편집은 재추출이
덮어쓰지 않는다. 의미가 비슷하다는 이유만으로 합치지 않고 완전 동일한 사실만
안정 ID로 중복 억제한다.

연표는 숫자만 파싱하지 않는다. 승인 연표와 원문을 critic이 대조해 연대,
부모 사건과 선후 관계로 정렬하며 불명확한 항목은 unresolved로 보존한다.

### 8.6 검색과 coverage

검색 순서:

1. enabled binding과 현재 요청·직전 완성 대화·최신 장면 신호를 읽는다.
2. 작품·continuity·branch·시점·공개 상한으로 MariaDB 범위를 제한한다.
3. 별도 `archive_center_reference_vectors` 컬렉션에서 실제 후보를 찾는다.
4. MariaDB에서 review, temporal, reveal, knower 범위를 다시 hard filter한다.
5. 현재 session 사실과 충돌하거나 이미 로어북에 충분한 필드를 제외한다.
6. `partial/missing`을 중심으로 별도 원작 블록을 조립한다.

coverage 단계:

- `coverage_shadow.v1`: 후보별 needed, status, missing field와 근거 진단
- `coverage_shadow.v2`: 직전 완성 턴, 최신 장소, world rule 기반 scene signal
- `coverage_field_index.v1`: 작품 승인 자료와 활성 host context의 field inventory
- `coverage_apply.v1`: raw `selected`와 실제 `injection_items` 분리
- 명시적 `supplement/primary/unknown` 운용 적용

상태는 `covered/partial/missing/conflict/not_applicable/unknown`을 사용한다.
`covered`, `not_applicable`, `unknown`, `conflict`는 자동 주입하지 않는다.
Chroma 후보는 반환 순서, raw distance와 실제 cosine을 보존한다. field index가
후보 창 밖의 필요한 자료를 찾더라도 가짜 Chroma rank나 점수를 만들지 않는다.

후보 개수는 승인 source 수와 요청 limit에서 계산한다. 관계 확장의 별도 고정
4개 상한은 제거했다. 관계 claim 확장은 같은 주체 또는 명시적으로 언급된
entity에 한정하고 `primary_relation_expansion` provenance를 표시한다.

### 8.7 주입과 UI

우선순위:

1. 현재 사용자 입력과 최근 대화
2. 현재 세션의 확정 기억과 사실
3. 시점과 지식 범위를 통과한 원작 참고 자료

원작 자료는 `Original Work Reference` 블록으로 별도 조립한다. Effective Input
UI에서도 장기 기억용 `Assembled Auxiliary Context`와 분리해 표시한다. 원문
excerpt는 증거이지 지시문이 아니며 미래 canon 사건을 강제하지 않는다.

### 8.8 세션 lifecycle

- 연결은 사용자가 preview 후 명시적으로 적용한다.
- binding 자체가 사용 선언이며 별도의 중복 on/off 토글을 두지 않는다.
- 복사·이동은 binding/runtime만 복제하고 작품 본문과 vector는 공유한다.
- 턴 rollback은 binding을 유지하고 삭제 턴에서 나온 runtime 후보만 되돌린다.
- 세션 삭제는 binding/runtime만 제거한다.
- 작품 삭제는 활성 binding이 있으면 차단한다.
- 원작 collection과 일반 기억 collection 이름이 같으면 시작 설정을 거부한다.

### 8.9 원작 기능의 남은 검증

- 실제 RisuAI 장기 session에서 보완/단독 모드 품질
- 복사·이동·연결·rollback·session 삭제와 binding/runtime
- 시점·branch·공개 상한·knower hard filter
- 기존 승인 excerpt의 무LLM 원문 재검증
- 잘못된 entity/relationship 자료 정정과 재색인
- 웹 URL 직접 수집

자료 자체의 불균형은 ranking을 조정해 숨기지 않는다. 예를 들어 Huntr/X 구성원과
수지(서씨) 같은 잘못된 연결은 원문 보강, 승인 entity/relationship 정정, 재색인
후 같은 사례로 다시 확인해야 한다.

## 9. 오류와 부분 성공

목표 오류 envelope는 `code`, `severity`, `retryable`, `component`, `detail`이다.
일부 API와 원작 recall에는 적용됐지만 모든 runtime API의 공통 계약은 아니다.

| 실패 | 사용자 흐름 | 저장 판정 |
| --- | --- | --- |
| prepare timeout | 원래 입력 보존 | 아직 저장 없음 |
| complete timeout | 화면 출력 보존 | 처리 상태 확인 필요 |
| MariaDB 실패 | 화면 출력 보존, 저장 실패 경고 | 실패 |
| ChromaDB 실패 | 화면과 MariaDB 원문 보존 | vector 부분 실패 |
| critic 실패 | raw 저장 보존 | derived 부분 실패 |
| 원작 전용 실패 | 메인 기능 보존, 원작 경고 | 원작 recall 실패/저하 |

## 10. 설치, 자동 업데이트와 배포

### 10.1 지원 패키지 공통 안전 업데이트

3.0 계열 Windows, Linux, macOS, Termux 패키지는 다음 릴리스를 다음 시작에
적용할 수 있는 플랫폼별 updater를 포함한다. 일반 Android APK가 아니라 Termux
arm64 패키지가 현재 Android 지원 범위다.

1. GitHub latest release와 플랫폼 자산 확인
2. Windows는 경량 update ZIP, POSIX는 동일 OS·아키텍처 패키지 다운로드
3. release SHA-256과 managed-file manifest 검증
4. 외부 helper가 backend 종료 뒤 관리 파일만 journal과 함께 교체
5. 새 backend 시작과 메인 `/ready`, 정확한 `/version` 확인
6. 성공 시 commit, 실패 시 이전 관리 파일로 rollback 후 이전 버전 재시작

실행 중인 backend가 자신을 직접 덮어쓰지 않는다. 무상태 검증 실패나 helper
부재는 기존 시작을 막지 않지만, 부분 적용 여부를 증명할 수 없는 혼합 상태는
시작을 중단한다.

보호 대상:

- `.env.full.local`과 기타 실제 `.env*`
- `.runtime`, `.runtime-cache`, `.updates`
- MariaDB·ChromaDB 데이터와 collection
- vector, cache, secret/data, 사용자 prompt override

v1은 migration SQL이나 `mariadb-schema.exe`/`mariadb-schema` 추가·변경을 적용 전에 거부한다.
파일 rollback과 DB schema가 어긋나는 것을 막기 위한 제한이다. v1은 관리 파일
삭제도 지원하지 않으며 누락 파일이 있으면 mutation 전에 거부한다.

### 10.2 플랫폼 패키지

3.0.1 release 자산:

- Windows x64 전체 설치 패키지
- Windows x64 경량 자동 업데이트 패키지
- Linux x64와 Linux arm64 자동 설치 패키지
- macOS Intel과 macOS Apple Silicon 자동 설치 패키지
- Termux arm64 자동 설치 패키지
- 전체 자산용 `SHA256SUMS-3.0.1.txt`

POSIX 패키지는 Go 1.26.5로 목표 `GOOS/GOARCH`를 교차 빌드한다. 각 패키지에
플랫폼별 `archive-center-updater`, `PACKAGE_FILE_MANIFEST.json`, 내부
`SHA256SUMS.txt`를 넣는다. launcher는 다음 시작에 후보를 적용하고 새 backend의
`/ready`와 정확한 target version을 통과한 때만 commit한다. 실패하면 관리 파일을
rollback하고 이전 backend로 다시 시작한다. Termux는 공유 저장소의 noexec를
피하기 위해 updater를 보호된 실행 디렉터리에 복사해 실행한다.

초기 공개된 3.0.0 POSIX ZIP에는 updater가 없었으므로 그 패키지를 이미 설치한
사용자는 updater가 포함된 패치 패키지로 한 번 수동 교체해야 한다. 그 다음
릴리스부터는 UI에서 받은 업데이트가 다음 시작에 자동 적용된다. Windows 3.0.0
패키지는 기존부터 이 전환 경로를 포함한다.

### 10.3 3.0.1 POSIX 자동 업데이트 교정

3.0.1은 Windows에만 연결돼 있던 다음 시작 안전 적용 계약을 Linux, macOS,
Termux에도 연결한 패치 릴리스다.

- update download가 현재 OS에 맞는 필수 실행 파일명을 pending contract에 기록
- POSIX 빌더가 backend와 같은 `GOOS/GOARCH`의 updater를 함께 생성
- 모든 POSIX ZIP에 managed-file manifest와 내부 SHA-256 목록 포함
- ZIP의 실행 권한 정보가 부족해도 관리형 binary와 launcher를 `0755`로 복원
- 확정 전에 새 backend의 메인 `/ready`와 정확한 target version을 확인
- readiness 실패, version 불일치, commit 실패에서 이전 관리 파일로 rollback
- migration과 `mariadb-schema` 변경, 실제 env, DB/vector/runtime 데이터는 적용 범위에서 제외

로컬 배포 후보 검증에서는 Windows 전체/경량 ZIP과 POSIX 다섯 ZIP의 필수 파일,
manifest hash, `GOOS/GOARCH`, shell 구문, 비밀값·사용자 데이터 부재를 확인했다.
Linux형 3.0.0 복제본에 3.0.1 후보를 적용해 commit과 rollback을 각각 실행했고
두 경로 모두 `.env.full.local` 보존을 확인했다. 실제 Linux, macOS, Termux 기기
검증은 계속 별도 실기기 검증 항목으로 남는다.

## 11. 버전과 호환성 기준

### 11.1 Go

- `go.mod`와 CI 기준: Go 1.26.5
- Windows full release builder: 1.26.5 미만 명시 차단, toolchain manifest 기록
- POSIX package builder: 3.0.0 기본 버전과 Go 1.26.5 교차 빌드 확인
- 미완료: POSIX builder의 Windows와 동일한 명시적 하한 오류와 toolchain manifest
- `/version`의 `go_version`은 현재 `unknown`

### 11.2 ChromaDB

- 관리형 기준: ChromaDB 1.5.9
- API path: `/api/v2`
- tenant/database: `default_tenant`, `default_database`
- 실제 CI: create, upsert, query, get, count, delete와 서로 다른 질의 순위 변화
- 버전/API/차원 불일치에서 collection이나 MariaDB 데이터 자동 삭제 금지

현재 남은 호환성 작업:

- startup에서 `GET /api/v2/version` 지원 범위 확인
- `chroma_version_mismatch`, `chroma_api_path_mismatch`,
  `chroma_heartbeat_failed`, `chroma_collection_api_incompatible` 안정 code
- 임의 외부 Chroma와 관리형 Chroma 검증 계약 분리
- `cmd/managed-mariadb-e2e/report_summary.go`의 잔여 `/api/v1` 기본 표시 정리

### 11.3 credential

턴 요청은 아직 provider key 원문을 백엔드로 전달한다. 별도 설정 API,
credential handle/revision과 원격 HTTP 전송 경고 계약은 미완료다. credential을
응답, trace, dashboard 또는 로그에 원문으로 반환해서는 안 된다.

```text
git diff --check
```

3.0 계열 릴리스에서 확인한 항목:

- 전체 Go 테스트 통과
- JavaScript와 관련 PowerShell 구문 검사 통과
- production updater apply/status/commit, 명시 rollback, 변조 자산 사전 거부

- Linux형 3.0.0 복제 패키지에 3.0.1 후보를 적용한 뒤 commit과 rollback 각각 확인
- Linux x64/arm64, macOS Intel/Apple Silicon, Termux arm64 패키지에 updater와
  managed manifest 포함, 파일 해시·대상 `GOOS/GOARCH`·shell 구문 확인
- `.env.full.local`과 `.runtime` sentinel 보존
- 잘못된 DB 설정의 다음 시작에서 자동 rollback과 3.0.0 재시작
- 정상 설정의 `/ready`, `/version` 3.0.1과 commit
- GitHub `v3.0.0` latest metadata, 자산 크기와 digest 확인

자동 검증이 증명하지 않는 항목:

## 15. 명시적으로 보류한 항목

- Go가 만든 최종 주입 계획이 UI 상한보다 클 때 JavaScript에서 다시 자르는 작업
- 원작 자료와 장기 기억 entity를 node graph로 시각화하는 작업
- 여러 start message를 합친 turn 0을 원작 검색 질의로 직접 사용하는 작업

이 항목은 현재 치명 오류가 재현되지 않았거나 더 넓은 canonical contract가
필요하다. 테스트를 통과시키기 위한 새 우회·fallback·무조건 성공 처리를 추가하지 않는다.

## 16. 데이터 안전과 rollback 원칙

- 실제 `.env`, API key, token, 사용자 DB와 `.runtime`은 Git이나 release에 넣지 않는다.
- Chroma 실패가 MariaDB canonical row를 지우거나 전체 저장을 실패로 바꾸지 않는다.
- 세션 삭제와 턴 rollback은 원작 작품·문서·claim·vector를 삭제하지 않는다.
- 3.0 상태 행은 2.5가 읽지 않으면 사용되지 않을 뿐이며 되돌릴 때 DB 전체를 초기화하지 않는다.
- 의미 유사도만으로 기존 기억, 상태, entity 또는 claim을 자동 삭제·병합하지 않는다.
- 복사·이동·연결 baseline 이전의 imported prefix를 로컬 turn 0/1로 오인해 삭제하지 않는다.
- 새 경로가 입력, 저장, 삭제, 후처리 또는 기존 DB를 깨면 기본값으로 승격하지 않는다.

## 17. 라이브 테스트 출구 기준

- 원작 품질은 실제 사용자 반응이 필요한 제품 품질 항목으로 분류한다.
- 첫 실험은 `supplement`, `primary`는 별도 실험군으로 분리한다.
- 사용자가 확인하지 않은 대화 원문이나 원작 문서를 외부로 자동 수집하지 않는다.
- 피드백은 검색 관련성, 원작 사실 정확성, 기존 로어북 중복, 메인 기억 방해,
  원작과 무관한 이탈을 분리해 기록한다.
- 원작 오류가 메인 시작, `/ready`, 저장, 장기 기억을 막으면 즉시 중단한다.
- 자료 오류는 ranking으로 숨기지 않고 승인 자료를 정정·재색인해 재검증한다.
- 참가자 수나 기간만으로 완료 처리하지 않고 재현 사례와 회귀 여부를 함께 본다.
